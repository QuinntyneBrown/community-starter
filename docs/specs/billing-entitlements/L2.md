# BILL — Billing and Entitlements behavior

## L2-BILL-001: Keep Billing disabled cleanly by default

**Traces to:** L1-BILL-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The default profile has no active paid behavior, provider dependency, credential requirement,
financial Job, or claim that payment is supported; the declared free baseline remains usable.

### Acceptance criteria

1. Given default configuration, when the artifact starts, then paid routes, navigation, checkout controls, webhooks, schedulers, provider clients, credentials, seeded paid Plans, and marketing claims are absent or explicitly unavailable.
2. Given a client supplies a Plan, Price, Subscription, provider, or Entitlement identifier while Billing is disabled, when any protected action is attempted, then no paid state or access is created and no provider is called.
3. Given Billing is disabled after prior use, when the approved shutdown plan runs, then new checkout and renewal stop while existing financial records, customer access, refunds, Entitlements, and retention follow an explicit migration policy.
4. Given non-Billing capabilities run, when Billing is disabled, then they do not fail startup, enqueue financial work, or assume a paid Entitlement exists.

### States and accessibility

Do not render dead purchase controls; stale links show one accessible unavailable explanation and a
valid next action, while operators receive explicit disabled and migration status.

### Verification

- Domain/unit: disabled profile, free baseline, and no-implied-Entitlement rules.
- Integration/contract: absent routes/jobs/webhooks/credentials and no-provider-call probes.
- Component: navigation removal and stale-link unavailable state.
- Journey/live: deploy default and post-use-disabled profiles and verify every paid path is inert.
- Manual/rendered: truthful marketing/admin copy, keyboard navigation, and configuration review.
