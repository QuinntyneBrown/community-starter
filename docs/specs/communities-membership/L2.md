# COMM — Communities and membership behavior

## L2-COMM-001: Create a Community

**Traces to:** L1-COMM-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

An eligible Account can create a Community with a unique stable identifier and an initial active
Membership whose administrative Role prevents an ownerless Community.

### Acceptance criteria

1. Given an eligible Account and valid Community name, identifier, and visibility, when creation is requested, then one Community and one active administrative Membership are committed atomically.
2. Given the requested identifier is reserved or already assigned, when creation is requested, then the server returns a stable conflict, creates no Community, and creates no Membership.
3. Given the Account exceeds a configured creation or active-Community limit, when creation is requested, then the server returns the applicable limit and persists no partial state.
4. Given the same idempotent creation request is retried, when the first request already committed, then the same Community result is returned without a duplicate Community or Membership.

### States and accessibility

Show validating, creating, identifier-available, conflict, limit, success, and unavailable states;
associate validation, announce the committed result, and keep progress understandable without motion.

### Verification

- Domain/unit: initial Role, identifier, eligibility, and limit rules.
- Integration/contract: atomic creation, conflict, quota, and idempotent replay.
- Component: form, identifier feedback, conflict, and success states.
- Journey/live: create a Community and enter it as its initial administrator.
- Manual/rendered: keyboard, narrow-screen, zoom, and status-announcement review.

## L2-COMM-002: Configure Community identity

**Traces to:** L1-COMM-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Authorized Accounts can update Community presentation, access mode, and stable routing identity
under Permission, version, documented Unicode/case normalization, reserved/confusable-name,
uniqueness, and redirect or alias constraints.

### Acceptance criteria

1. Given an active Membership with the required Permission and valid values, when Community settings are updated, then the accepted values and a new version are returned.
2. Given the Account lacks the Permission or its Membership is inactive, when settings are updated, then the server rejects the request and preserves every setting.
3. Given a requested routing identifier is reserved, confusable, unavailable, or normalization-equivalent to an existing identifier, when it is changed, then the server returns a conflict and preserves the existing identifier and links.
4. Given a stale settings version, when an update is submitted, then the server returns a conflict with current safe state and does not overwrite the newer change.
5. Given an allowed routing identifier change commits, when old and new links are requested, then the new identifier is canonical, the old identifier follows the declared bounded redirect or alias policy, and another Community cannot claim that alias before its documented release.

### States and accessibility

Show loading, saved, unsaved, validation, conflict, forbidden, and retry states; group settings with
headings, announce saves, preserve input on conflict, and explain access modes in text.

### Verification

- Domain/unit: settings, visibility, normalization, reserved/confusable name, alias, and version rules.
- Integration/contract: update, forbidden, identifier conflict/redirect, and optimistic concurrency.
- Component: settings form, save, conflict reconciliation, and permission states.
- Journey/live: update identity and visibility, then confirm public and member views.
- Manual/rendered: keyboard, screen-reader grouping, reflow, and plain-language review.

## L2-COMM-005: Request gated Membership

**Traces to:** L1-COMM-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

A gated Community can receive one pending Membership request per Account and let an authorized
Membership approve or decline it from current policy and capacity. Requests and invitations share
one versioned Account-plus-Community entry-artifact set whose first valid activation is final.

### Acceptance criteria

1. Given an eligible Account and active gated Community, when Membership is requested with valid answers, then one pending request in the versioned Account-plus-Community entry-artifact set is created and the Account can see its status; an existing pending invitation is not accepted, revoked, or allowed to grant Role authority merely by this request.
2. Given a pending request already exists, when the Account retries or edits through the supported path, then no duplicate is created and the declared update policy is applied.
3. Given an authorized reviewer approves a current request while the Community is active, the Account remains eligible, capacity remains, and the configured join Role is active, same-Community, and within delegation authority, when the decision atomically rechecks Community lifecycle and entry-set version and commits, then the existing or new Membership identity becomes active with that winning request's current Role and one new interval, every competing pending artifact terminalizes as superseded without applying Role/Delivery, and an Audit Event and privacy-safe outcomes record the winner.
4. Given the Community is unavailable, reviewer lacks Permission, Account is ineligible, request/entry-set is stale, capacity is exhausted, or the configured Role is absent, cross-Community, inactive, or non-delegable, when a decision is attempted, then no Membership activation occurs and the current safe state is returned.
5. Given the requester withdraws or an authorized reviewer declines a current pending request, when the versioned transition commits or repeats, then the request becomes withdrawn or declined with a bounded safe outcome and Audit Event, no Membership/Role/capacity/Delivery effect occurs, and later approval requires a new current request rather than replaying that artifact.

### States and accessibility

Show request, pending, editable/withdrawable, approved, declined, full, stale, and forbidden states;
questions have labels, decisions are announced, and status is not conveyed by color alone.

### Verification

- Domain/unit: entry-set/request uniqueness, winning Role, review eligibility, withdrawal/decline, capacity, and decisions.
- Integration/contract: request/invitation coexistence, update/withdraw, approve/decline race, loser terminalization, and replay.
- Component: request form, status, review list, confirmation, and stale states.
- Journey/live: request Membership and approve it from a second authorized Account.
- Manual/rendered: keyboard forms, review table/list, reflow, and announcements.

## L2-COMM-006: Invite an Account

**Traces to:** L1-COMM-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Authorized Memberships can issue bounded, revocable Community invitations that create or activate
at most one Membership after the intended Account accepts; all invitations coordinate with gated
requests through the same versioned Account-plus-Community entry-artifact set.

### Acceptance criteria

1. Given an inviter with Permission, a valid target, active Community with capacity, and an active same-Community offered Role within the inviter's current delegation authority, when an invitation is issued, then at most one pending invitation exists for that Account/Community entry set, records the exact Role identity/version, inviter, expiry, artifact-set version, and stable identity, and requests generic Delivery; replacement requires an explicit current-version supersede that terminalizes the prior invitation rather than leaving role-different parallel offers.
2. Given the intended Account accepts a valid unused invitation while remaining eligible, when acceptance atomically rechecks active Community lifecycle, capacity, inviter authority, offered Role identity/version/ownership/delegation ceiling, invitation and entry-set versions, and absence of an active interval, then the existing or new Membership identity becomes active once with only that winning invitation's Role, every competing pending artifact terminalizes as superseded without Role/capacity/Delivery effects, and an Audit Event and privacy-safe outcomes record the winner.
3. Given the Community is unavailable, invitation is expired, revoked, lifecycle-cancelled, used, for another or ineligible Account, exceeds capacity, or its inviter/offered Role fails current checks, when acceptance is attempted, then no Membership changes and a safe invalid, changed, or unavailable result is returned.
4. Given the intended Account declines, an authorized inviter revokes, or the idempotent expiry scheduler reaches a pending invitation's deadline, when the current-version transition commits or repeats, then it becomes declined, revoked, or expired, later acceptance cannot activate Membership, no Role/capacity effect occurs, and a bounded safe outcome and Audit Event are retained.
5. Given issue/replacement, request approval, acceptance, decline, revoke, expiry, or retry races another entry-artifact transition, when processing commits, then one artifact-set version wins, stale decisions return the current safe outcome, no active Membership Role is overwritten, and any later Role change must use the separately authorized Role-assignment lifecycle.

### States and accessibility

Show inviting, generic sent, pending, accepted, declined, revoked, expired, and capacity states;
links have meaningful names, expiry is textual, and results are focused and announced.

### Verification

- Domain/unit: invitation/entry-set uniqueness, Role winner, delegation, target, accept/decline/revoke, and single use.
- Integration/contract: role-different replacement, request/accept race, loser terminalization, inviter/Role change, replay, and capacity.
- Component: invitation form/list, superseded, accept/decline/revoke, invalid, and active-already states.
- Journey/live: race a gated approval with role-different invitation acceptance and prove one Role/interval.
- Manual/rendered: copied-link, keyboard, screen-reader, and time-zone review.

## L2-COMM-007: Leave, remove, and suspend

**Traces to:** L1-COMM-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Voluntary exit and neutral administrative Membership corrections are distinct from punitive
enforcement, server-authorized, auditable, and cannot remove the last required administrator or
silently retain member access. Punitive suspension or removal is applied only by a Moderation Action.

### Acceptance criteria

1. Given an eligible active Membership, when its Account confirms leaving, then the Membership becomes inactive and member-only access ends without deleting authored content outside policy.
2. Given an authorized Membership uses the versioned non-punitive reason allow-list to correct duplicate, expired, ineligible, or Community-closing Membership state, when neutral removal commits, then access and effective Permissions end immediately with an auditable lifecycle reason that cannot allege or punish policy misconduct.
3. Given the target is the last Membership with a required administrative Role, when leave, removal, or suspension is attempted, then the server rejects it with remediation and preserves Membership state.
4. Given the actor lacks Permission, targets a protected higher-authority Role, or submits stale state, when a lifecycle change is attempted, then no Membership changes and a safe forbidden or conflict result is returned.
5. Given suspension, removal, or restriction is requested because of alleged conduct or policy violation, when the ordinary Membership endpoint is used, then it rejects the punitive change; only a committed Moderation Case and Moderation Action may transition it under the Action effect matrix, notice, expiry, and Appeal policy.

### States and accessibility

Show leave impact, removal/suspension reason, blocked continuity, confirmation, success, forbidden,
and conflict states; destructive intent is explicit and focus returns predictably.

### Verification

- Domain/unit: exit, suspension, authority hierarchy, and last-administrator rules.
- Integration/contract: leave, remove, suspend, revoked access, forbidden, and stale cases.
- Component: confirmations, reason form, remediation, and post-exit states.
- Journey/live: leave and suspend in separate fixtures, proving immediate member-only denial.
- Manual/rendered: keyboard, dialog focus, screen-reader, and destructive-copy review.

## L2-COMM-009: Assign a Role safely

**Traces to:** L1-COMM-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Role assignment changes effective Permissions only for an active Membership in the same Community
and only within the actor's current delegation authority.

### Acceptance criteria

1. Given an authorized actor, active target Membership, and delegable Role in the same Community, when assignment commits, then the target's effective Permissions change and the new Role is returned.
2. Given the Role belongs to another Community, the target is inactive, or the actor cannot delegate it, when assignment is requested, then the server rejects it and preserves the existing assignment.
3. Given two actors submit changes against the same Membership version, when one commits first, then the stale change receives a conflict and cannot overwrite the current Role.
4. Given a Role is changed, when the target makes its next protected request, then authorization uses the new effective Permissions without requiring a new client sign-in.

### States and accessibility

Show current/effective Role, selecting, saving, forbidden, stale, and success states; describe the
Permission impact in text, preserve focus on conflict, and announce committed changes.

### Verification

- Domain/unit: same-Community, delegation, active-target, and version rules.
- Integration/contract: assignment, cross-Community rejection, concurrency, and immediate enforcement.
- Component: Role selector, impact summary, forbidden, and conflict states.
- Journey/live: change a Role during another Account's active session and verify access changes.
- Manual/rendered: keyboard select, screen-reader summary, and narrow-screen review.

## L2-COMM-010: Preserve administrative continuity

**Traces to:** L1-COMM-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every active Community retains the configured minimum active Memberships with its required
administrative Role through transfers, exits, Membership suspension, Account deactivation or deletion,
and concurrent changes.

### Acceptance criteria

1. Given another eligible active Membership, when administrative authority is transferred, then the recipient gains the required Role before the prior holder can relinquish it.
2. Given no eligible replacement exists, when the last required administrator attempts demotion, leave, Membership suspension, Account deactivation, or deletion, then the server rejects the transition and preserves authority.
3. Given concurrent changes would each appear safe alone but together violate the minimum, when they commit, then at most the transition preserving the invariant succeeds and the other returns a conflict.
4. Given an exceptional recovery process is invoked, when authority changes, then stronger verification, declared evidence, and an audit record are required.

### States and accessibility

Show transfer selection, impact, stronger verification, blocked-with-remediation, conflict, and
success states; explain required continuity plainly and keep confirmation keyboard-contained.

### Verification

- Domain/unit: minimum-administrator, transfer ordering, and concurrency rules.
- Integration/contract: transfer, last-holder rejection, closure interaction, and recovery path.
- Component: transfer, remediation, challenge, and conflict states.
- Journey/live: transfer authority and attempt simultaneous last-administrator exits.
- Manual/rendered: keyboard, dialog focus, plain-language, and announcement review.

3. Given a retried command carries the same idempotency identity and payload, when its original result exists, then the same result is returned without consuming capacity twice.

## L2-COMM-013: Govern Space lifecycle

**Traces to:** L1-COMM-005

**Status:** proposed

**Source:** product brief + research

**Related:** —

An authorized Membership can create, update, archive, and restore a Space within exactly one
Community using a normalized, unique, stable identifier and a versioned server-owned lifecycle whose
impact contract covers every Space-scoped capability.

### Acceptance criteria

1. Given an active Membership with Permission and valid Space name and identifier, when creation is requested, then one active Space owned by the route Community is committed with normalized identity and the Community's default access policy.
2. Given an authorized actor, current version, and valid changes, when a Space is updated, then its accepted fields and version change while Community ownership and stable identity follow the declared immutability/redirect policy.
3. Given an eligible active Space is proposed for archive, when impact is evaluated, then a current dependency manifest previews Posts, Comments, Reactions, Bookmarks, open direct Conversations, nonterminal Events and RSVPs, Roles/rules, queued Jobs, Deliveries, realtime, Search, and Feed projections; the confirmed plan may grant a current Community administrator holding `ArchiveSpace` exact dependency authority only to end listed Conversations without Message read or participant management and to move listed Events to the preselected eligible active same-Community Space or cancel them without roster/private-location access, preserving fixed history, capacity and attendance effects, Audit Events, and privacy-safe participant/attendee notice, and archive cannot commit until those type-owned transitions complete.
4. Given an identifier conflict, stale version, cross-Community identifier, missing Permission, or incompatible lifecycle state, when a Space mutation is requested, then the server rejects it and preserves Space, content, and access state.
5. Given the dependency plan and current version are confirmed, when Space archive execution begins, then `archive-pending` atomically rejects new dependencies and rechecks its manifest, returning an updated conflict on mismatch, and permits only personal Reaction/Bookmark withdrawal, required safety/privacy/retention/export work, and exact current Space-plan Conversation end or Event move/cancel; if dependencies terminalize the Space becomes archived, while cancellation returns it to its prior stable lifecycle without reversing committed Conversation endings or Event cancellations, reviving terminal artifacts, or replaying suppressed work, and other scoped mutations remain rejected until a current explicit action is allowed.

### States and accessibility

Show create/edit, identifier validation, saving, conflict, active, archive impact/confirmation,
archive pending, dependency remediation, archived, restore, forbidden, and error states; status is
textual and destructive focus is contained.

### Verification

- Domain/unit: normalization, uniqueness, ownership, dependency plan, mutation matrix, version, and restore rules.
- Integration/contract: create/update, race Space archive with bounded Conversation end or Event move/cancel, every scoped path, suppression, restore, and cross-Community cases.
- Component: Space form/list, archive preview/pending/remediation/cancel, archived, restore, and conflict states.
- Journey/live: populate every scoped capability, resolve archive dependencies, probe every write, restore, and prove no terminal state revives.

**Traces to:** L1-COMM-002

**Related:** —

One Account has at most one stable Membership identity per Community across separate active

### Acceptance criteria

1. 2. Given the prior interval ended through an active suspension, Community ban, unresolved removal condition, eligibility restriction, or deletion state, when rejoin is attempted, then the current lifecycle and Moderation Action matrices deny or route remediation and no new or reactivated Membership appears.
3. Given open join, request approval, or invitation acceptance races for the same Account and Community, when it commits against Community, entry-set, and Membership versions, then exactly one eligible entry path creates one active interval with only its current default/configured/offered Role and terminalizes every loser, consumes capacity/activation Delivery once, and prevents stale artifacts from changing the active Role; later Role change uses only the Role-assignment lifecycle.
4. Given rejoin succeeds, when history, counts, Feeds, Search, Spaces, Events, Conversations, Notifications, and exports are built, then the prior and current intervals remain distinguishable, excluded-period access is not restored, and current visibility/rules determine new access.
5. Given activation fails after partial work is queued or a losing artifact already queued notice/Delivery work, when rollback/reconciliation runs, then Membership/capacity remain at the one authoritative outcome, losing work suppresses or reports its superseded status without claiming activation, and no invitation, request, Notification, Role, or projection contradicts the entry-set result.

### States and accessibility

conflict, active-new-interval, and failed states with history and access consequences in plain text.

### Verification

- Domain/unit: stable identity, entry-set winner, interval, Role precedence, restriction, uniqueness, and capacity.
- Integration/contract: open/gated/invited races, terminal losers, stale Role attempts, Action denial, rollback, and projections.
- Component: rejoin review, changed rules/Role, restricted, pending, conflict, and success states.

- Manual/rendered: consequence clarity, keyboard, screen reader, focus, and narrow-screen review.
