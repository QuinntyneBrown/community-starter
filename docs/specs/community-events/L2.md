# EVNT — Community events behavior

## L2-EVNT-001: Create a valid Event

**Traces to:** L1-EVNT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

An authorized organizer creates a draft Event in exactly one Community and optionally one active
same-Community Space, with bounded content, an explicit canonical time zone, valid schedule,
audience, location type, accepted applicable rules, and stable identity.

### Acceptance criteria

1. Given an Account with active Membership and Event-creation Permission, a valid one-time schedule, and optional active same-Community Space in which it may create Events, when creation is requested, then one draft Event is committed with that Account as the exactly one responsible organizer, zero co-organizers, immutable Community, explicit nullable Space identity, UTC instants, canonical time-zone identifier, local-time intent, and version.
2. Given the selected Space is archived, inaccessible, cross-Community, or missing required rules acceptance, the end precedes the start, the time zone is unsupported, a local time is nonexistent or ambiguous at a daylight-saving transition without an explicit supported offset choice, content exceeds bounds, or the location contract is incomplete, when submitted, then field-safe validation is returned and no Event is persisted.
3. Given a virtual or restricted venue contains private access details, when the Event is created, then those details are stored separately from ordinary Event summaries and inherit the declared audience.
4. Given the same idempotency key is retried, when creation is processed, then the original Event is returned without a duplicate.

### States and accessibility

Show editing, validating, saved draft, validation, conflict, and unavailable states; label date,
time-zone, duration, audience, and location controls and announce errors and save results.

### Verification

- Domain/unit: schedule, time-zone, audience, location, and idempotency rules.
- Integration/contract: create, invalid schedule, private location, and retry contracts.
- Component: Event form, field errors, time-zone choice, and saved state.
- Journey/live: create and reload a draft across a different viewer time zone.
- Manual/rendered: keyboard, screen-reader, zoom, localization, and narrow-screen review.

## L2-EVNT-003: Publish, revise, and cancel atomically

**Traces to:** L1-EVNT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Publication, material revision, cancellation, completion, and archival are explicit versioned Event
transitions whose downstream work begins only after persistence succeeds.

### Acceptance criteria

1. Given a complete authorized draft, when publish is requested against its current version, then the Event becomes published and one post-commit event is recorded.
2. Given stale version, missing required data, lost source or target Space Permission, archive-pending/archived target Space, cross-Community Space, missing applicable rules acceptance, or illegal lifecycle transition, when change is requested, then a conflict or forbidden result is returned and Event state remains unchanged; only an exact current-version move or cancellation authorized by the source Space archive plan remains available within that plan's fence.
3. Given a published change or same-Community Space move, when impact is previewed, then the accepted material-change matrix classifies at least time, time zone, venue or access link, audience, Space, capacity, cancellation, organizer, and participation-term changes; source/target access and RSVP consequences are explicit, a material change requires confirmation, preserves prior state for audit, and queues attendee work against the committed version.
4. Given persistence fails, when the transition terminates, then no Feed, Search, Notification, or realtime update claims the change succeeded.
5. Given a published Event's end has passed or a completed Event reaches its archive-eligibility date, when the idempotent completion scheduler or an authorized current-version archive action runs, then it becomes completed or archived once; new RSVP, reminder, and ordinary edit actions are rejected, retained history follows audience and retention policy, and reporting remains available for the disclosed evidence window.

### States and accessibility

Show draft, publishing, published, changed, conflict, cancellation confirmation, cancelled, completed,
and archived states; preserve edits on conflict and clearly announce destructive outcomes.

### Verification

- Domain/unit: lifecycle, material-change, version, and cancellation rules.
- Integration/contract: publish/update/cancel, Space-archive plan fences, conflict, audit, and outbox.
- Component: lifecycle controls, conflict recovery, impact preview, and confirmation.
- Journey/live: publish, revise, and cancel while verifying every projection follows committed state.
- Manual/rendered: keyboard, focus recovery, destructive copy, and reduced-motion review.

## L2-EVNT-004: Authorize Event organizers

**Traces to:** L1-EVNT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every nonterminal Event has exactly one accepted responsible organizer assignment and zero or more
accepted co-organizer assignments. Their bounded authority derives from current Account, Community,
Membership, Role, Permission, Space, Event state, and assignment rather than a client-supplied ID.

### Acceptance criteria

1. Given the responsible organizer or a Community administrator holding `ManageEventOrganizers` proposes an eligible active Member with current Event/Space access, when a bounded co-organizer or successor offer is issued, then at most one pending organizer offer and one active assignment may exist for that Event/Account pair, and exact proposed role, delegated operations, Event scope, history disclosure, inviter authority, expiry, version, and stable identity are recorded without access; repeated/concurrent issuance returns or explicitly supersedes the offer, and no actor may delegate operations they cannot grant.
2. Given the intended Member accepts or declines a current offer, when Account, Membership, Community/Space, rules, Role/Permission, Event, assignment/offer uniqueness, cardinality, inviter authority, and offer version/expiry are atomically rechecked, then acceptance creates or changes one assignment and any responsibility transfer leaves exactly one responsible organizer, while decline terminalizes the offer without access; an Audit Event records the outcome and no stale, duplicate, or replayed offer grants retrospective attendee/evidence access.
3. Given an accepted organizer performs an Event action or a current Community administrator holding `ArchiveSpace` executes a confirmed Space archive plan, when authorization runs and the exact assignment or plan plus current Community lifecycle version are rechecked, then the organizer's assignment permits only delegated operations and the archive executor may only move the exact listed Event to its preselected active same-Community Space or cancel it; the executor applies declared attendance/notice effects and gains no organizer assignment, roster, Message, or private-location access, while a guessed or cross-Community ID changes nothing.
4. Given planned removal, leave, deactivation, or Permission loss would remove the responsible organizer from a nonterminal Event, when requested, then an accepted eligible successor or Event cancellation must commit first; forced or simultaneous loss immediately ends ordinary management and new participation, after which the responsibility registry atomically transfers, grants bounded stewardship that may only transfer/cancel, or cancels the Event with attendee effects before claiming continuity resolved.
5. Given the responsible organizer or a Community administrator holding `ManageEventOrganizers` withdraws a pending offer or revokes a co-organizer, or an organizer loses eligibility, an offer expires, assignments race, or an operation is retried, when processing runs, then the actor's current delegation authority is required, the offer terminalizes or access/subscriptions end before the next management effect, one version/uniqueness/cardinality result wins, stale offers cannot restore authority, queued work reauthorizes, and terminal Events retain historical attribution without active organizer Permission.

### States and accessibility

Show responsible and co-organizer assignments, pending offer, accepted, transfer-required, frozen,
stewardship, cancelled, unavailable, and conflict states; controls explain restrictions but never
serve as enforcement.

### Verification

- Domain/unit: assignment/offer uniqueness, roles, delegation ceiling, accept/decline/withdraw, cardinality, and continuity.
- Integration/contract: offer/transfer races, forced loss, stewardship, Space-archive no-read actions, guessed ID, and revocation.
- Component: organizer list, offer scope/decision/withdrawal, successor remediation, frozen, cancelled, and changed states.
- Journey/live: race and withdraw offers, add a co-organizer, transfer responsibility, and force sole-organizer loss.
- Manual/rendered: restriction explanation, keyboard flow, and screen-reader names review.

## L2-EVNT-005: Enforce Event audience and discovery

**Traces to:** L1-EVNT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Event visibility is enforced consistently across direct URL, Community and Space lists, Feed,
Search, Notifications, realtime, and caches.

### Acceptance criteria

1. Given a viewer satisfies the current Community, Space, or attendee audience, when the Event is requested, then only fields allowed for that audience are returned.
2. Given a viewer is ineligible, when any read, count, autocomplete, or guessed direct URL is requested, then the Event's existence and private fields are not disclosed.
3. Given visibility narrows, Membership changes, or a Moderation Action applies, when the change commits, then ineligible Feed, Search, cache, and realtime projections disappear within the accepted Event visibility propagation objective, default five minutes, and queued Deliveries reauthorize before sending.
4. Given the viewer and an Event's responsible organizer have a Block in either direction, when discovery or participation is requested, then the Event is privacy-safely unavailable, a co-organizer assignment does not override the Block, and every blocked organizer identity and pairwise interaction remains suppressed under the shared-Community Block matrix.

### States and accessibility

Show discoverable, restricted, no-longer-available, and empty states without revealing private
audience rules; Event summaries use semantic date, location, and status markup.

### Verification

- Domain/unit: audience matrix and field-level disclosure rules.
- Integration/contract: direct, Feed, Search, cache, and post-change projection checks.
- Component: cards/detail, empty, restricted, and unavailable presentation.
- Journey/live: narrow a visible Event and prove removal from every viewer and projection path.
- Manual/rendered: semantic date/status, privacy copy, keyboard, and screen-reader review.

## L2-EVNT-006: Protect attendee and location privacy

**Traces to:** L1-EVNT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Attendee identity, RSVP response, contact data, virtual access links, and precise venue details follow
separate least-disclosure policies evaluated at request and Delivery time.

### Acceptance criteria

1. Given an Event audience may view a public attendee summary, when requested, then only permitted Profiles or aggregate counts are returned under the attendee's visibility choice.
2. Given an authorized organizer needs an attendee roster, when requested, then only operationally necessary fields are returned and the access is audited when sensitive.
3. Given an ineligible viewer or stale shared link requests attendee or restricted location data, when evaluated, then private values are withheld without revealing hidden RSVP state.
4. Given an Account withdraws, is removed, enters a blocked pair, or loses access, when future views and Deliveries occur, then attendee identity, aggregate contribution, and access-link disclosure follow the shared-Community Block matrix and current Event policy.

### States and accessibility

Explain attendee visibility and restricted-location timing before RSVP; counts, badges, maps, and
links have text alternatives and never depend on color or pointer interaction alone.

### Verification

- Domain/unit: field disclosure, attendee preference, organizer need, and revocation rules.
- Integration/contract: attendee list, aggregate, location, stale link, and audit behavior.
- Component: privacy explanation, roster, aggregate, and restricted-location states.
- Journey/live: RSVP privately, revoke access, and prove roster/location boundaries for each role.
- Manual/rendered: privacy copy, keyboard maps/links, and screen-reader review.

## L2-EVNT-007: Transition one current RSVP

**Traces to:** L1-EVNT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

One eligible Member with an active Community Membership has one current RSVP per Event,
transitioned idempotently under Event, audience, Space/rules, deadline, capacity, Block, and
Moderation state.

### Acceptance criteria

1. Given an eligible active Member with current Event/Space access, accepted applicable rules, and open attendance, when going, interested, declined, or withdrawal is requested, then one current RSVP transitions atomically and the authoritative result is returned.
2. Given the same operation is retried, when processed, then counts and state do not duplicate or regress.
3. Given the Event is cancelled, closed, inaccessible, in an archived Space, past its deadline, or the Account lacks active Membership or other eligibility, when RSVP is attempted, then a typed failure is returned and RSVP and capacity state remain unchanged.
4. Given a stale version conflicts with an organizer change, when RSVP is attempted, then the client receives current Event and allowed-action state for reconciliation.
5. Given an attendee loses Event access after holding a confirmed or waitlisted RSVP, when attendance reconciliation runs, then that RSVP becomes access-revoked once, any occupied capacity is released, the next eligible waitlisted RSVP is considered, and only a privacy-minimal status notice may be queued after commit.

### States and accessibility

Show available responses, saving, confirmed, waitlisted, conflict, closed, cancelled, and unavailable
states; use a correctly named control group, announce status, and avoid color-only selection.

### Verification

- Domain/unit: one-current-RSVP, transition, deadline, eligibility, and idempotency rules.
- Integration/contract: RSVP options, retry, closed/cancelled, and stale conflict behavior.
- Component: response group, pending, confirmed, waitlisted, and conflict states.
- Journey/live: transition and retry an RSVP while another session changes the Event.
- Manual/rendered: keyboard group, screen-reader status, touch targets, and narrow-screen review.

## L2-EVNT-008: Allocate capacity and waitlist fairly

**Traces to:** L1-EVNT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Finite capacity and waitlist promotion are transactional, deterministic, observable, and safe under
concurrent RSVP, withdrawal, capacity change, and retry.

### Acceptance criteria

1. Given remaining capacity, when concurrent eligible going requests commit, then confirmed attendance never exceeds capacity; excess eligible requests enter the waitlist only when it is enabled, otherwise they receive the same typed full result without creating an RSVP.
2. Given a confirmed attendee withdraws, when promotion runs, then the next eligible waitlisted RSVP is selected by the Event's versioned total ordering, default committed waitlist time followed by stable RSVP identity, promoted once, and notified after commit; later policy changes do not silently reorder existing entries.
3. Given capacity is reduced below confirmed attendance, when an organizer requests the change, then the exact affected count and ordering are previewed and the change is rejected unless the accepted policy explicitly permits over-capacity grandfathering or named-attendee removal with confirmation, reason, audit, capacity release, and notification.
4. Given promotion fails or is retried, when processing resumes, then capacity and RSVP state remain consistent and duplicate promotions or Deliveries do not occur.

### States and accessibility

Show spots available, full, waitlist position or policy-safe status, promoted, and organizer impact
states; announce changes without exposing other attendees' identities.

### Verification

- Domain/unit: capacity invariant, waitlist ordering, promotion, and reduction rules.
- Integration/contract: concurrent transactions, retries, promotion outbox, and conflict behavior.
- Component: full/waitlist/promoted and capacity-impact states.
- Journey/live: race RSVPs for the final spot, withdraw, and prove one deterministic promotion.
- Manual/rendered: status clarity, privacy, keyboard, and screen-reader review.

## L2-EVNT-010: Notify eligible attendees of changes

**Traces to:** L1-EVNT-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Material changes, cancellation, waitlist promotion, and configured reminders create Notifications
from committed state with current recipient, preference, and access checks.

### Acceptance criteria

1. Given a material Event change or cancellation commits, when post-commit processing runs, then each currently eligible affected attendee receives at most one correctly versioned Notification.
2. Given a reminder becomes due, when scheduler processing runs, then current Event, RSVP, locale, time zone, preference, and access state are reevaluated before Delivery.
3. Given the Event changes after a reminder is queued, when Delivery begins, then stale time or location is not sent and work is refreshed, suppressed, or replaced idempotently.
4. Given an Account withdrew or lost access, when attendee Delivery runs, then no restricted Event or location content is disclosed.

### States and accessibility

Event UI explains which changes notify attendees and displays reminder status without implying
guaranteed external Delivery; Notification content follows locale and accessibility requirements.

### Verification

- Domain/unit: material-change, recipient, reminder, stale-version, and deduplication rules.
- Integration/contract: scheduler, queued change, preference, revocation, and Delivery contracts.
- Component: impact summary and reminder/status presentation.
- Journey/live: queue a reminder, change the Event, and verify only current information is delivered.
- Manual/rendered: localized time, privacy-safe subject, plain text, and screen-reader review.

## L2-EVNT-012: Report and moderate Event harm

**Traces to:** L1-EVNT-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Eligible people can Report an Event, organizer, or Event conduct, and authorized Moderation Actions
reconcile Event visibility, attendance, location access, Deliveries, exports, and evidence.

### Acceptance criteria

1. Given an eligible reporter can currently view an Event or presents an unexpired Account-bound Report Receipt from a prior authorized view, when a valid Report is submitted, then one Report with the exact receipt/current-view Event-version evidence commits atomically and is safely acknowledged.
2. Given an authorized Moderator hides, limits, or cancels the Event, when the action commits, then direct access, Feed, Search, RSVP, location, Notification, and realtime paths reconcile to it.
3. Given the Event is edited or deleted after reporting, when the case is reviewed, then authorized reviewers retain the captured evidence and provenance under policy.
4. Given an unauthorized action, guessed Event, or ineligible Report without current access or a valid Report Receipt, when attempted, then no Event, RSVP, Report, or evidence state changes and private details or receipt validity are not disclosed.

### States and accessibility

Provide report, acknowledgement, action-applied, cancelled/removed, appeal-status, and unavailable
states with safe language, explicit labels, focus management, and no graphic-only warning.

### Verification

- Domain/unit: reporting eligibility, evidence, action, and projection-impact rules.
- Integration/contract: Report, action, post-edit evidence, and cross-projection reconciliation.
- Component: reporting flow and moderated/cancelled Event presentation.
- Journey/live: report and hide an Event, then prove every discovery and participation path closes.
- Manual/rendered: sensitive-content handling, keyboard, screen-reader, and reduced-motion review.

## L2-EVNT-013: Reconcile attendance after access loss

**Traces to:** L1-EVNT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Attendance is reconciled from current Event access so a removed, blocked, banned, or otherwise
ineligible Account cannot retain a place, receive restricted details, or block waitlist progress.

### Acceptance criteria

1. Given a confirmed or waitlisted attendee loses Event access through Membership, Block, audience, or Moderation Action state, when that change commits or reconciliation observes it, then the RSVP moves once to access-revoked with the cause class and effective time while restricted Event details remain undisclosed.
2. Given an access-revoked RSVP occupied finite capacity, when the transition commits, then capacity is released in the same transaction and one post-commit promotion command identifies the affected Event without duplicating promotion on retry.
3. Given capacity becomes available, when promotion runs, then candidates are rechecked against current Event access and the first eligible RSVP under the declared ordering is promoted; ineligible candidates are reconciled and skipped deterministically.
4. Given the former attendee can receive a status notice, when Notification work is created, then it states only that attendance is no longer active and provides an allowed remediation or appeal route without disclosing a hidden Event, private policy, blocker, or Moderation evidence.
5. Given access is later restored, when the Account views the Event, then no prior place is silently restored; the server returns only currently allowed RSVP or waitlist actions under current capacity and deadline state.

### States and accessibility

Show attendance no-longer-active, capacity-updating, remediation, and rejoin-available states without
revealing a hidden Event or another Account; status and next actions are programmatically associated.

### Verification

- Domain/unit: access-loss causes, one-way reconciliation, capacity release, and no-silent-restore rules.
- Integration/contract: Membership, Block, audience, and Moderation transitions with retries.
- Component: privacy-minimal inactive, remediation, and rejoin-available states.
- Journey/live: revoke a final attendee's access, promote once, restore access, and require a new RSVP.
- Manual/rendered: privacy wording, keyboard, screen reader, focus, and narrow-screen review.
