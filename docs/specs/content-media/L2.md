# CONT — Content and media behavior

## L2-CONT-001: Draft and publish a Post

**Traces to:** L1-CONT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

An eligible Membership with the required Permission can create a draft or published Post in exactly
one Community and optionally one active Space in that Community; publication requires current scope,
accepted rules, valid content, and approved Attachments.

### Acceptance criteria

1. Given an active Membership with Permission, valid content, and an optional active same-Community Space the Account may write, when a draft is created, then one private draft owned by the Account is committed with immutable Community and explicit nullable Space identity.
2. Given a valid draft whose required Attachments are approved, when publish is requested, then the Post becomes published once with stable identity and publication time.
3. Given the Membership is inactive, Permission or applicable rules acceptance is absent, content is invalid, the selected Space is archived, inaccessible, or cross-Community, or an Attachment is pending/rejected, when create or publish is requested, then the server returns a stable failure and leaves publication state unchanged.
4. Given the same idempotent create or publish request is retried, when its original result committed, then the same Post result is returned without a duplicate Post or publication event.

### States and accessibility

Show editing, autosave/manual-save policy, saving, saved draft, Attachment processing, publishing,
published, validation, conflict, forbidden, and error states; announce saves without excessive noise.

### Verification

- Domain/unit: draft/publication, Permission, Attachment readiness, and idempotency rules.
- Integration/contract: create draft, publish, invalid/pending, forbidden, and replay cases.
- Component: editor, save status, Attachment state, validation, and publish action.
- Journey/live: draft with an Attachment, process it, publish, and view from another Account.
- Manual/rendered: keyboard editor, screen-reader labels, reflow, and save announcements.

## L2-CONT-002: Edit a Post safely

**Traces to:** L1-CONT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Post edits require current author or delegated Permission, valid content, compatible Attachments, and
the current version; authorship and Community ownership cannot be reassigned, and Space movement is
an explicit same-Community transition rather than a forged field edit.

### Acceptance criteria

1. Given the author or authorized Membership, current version, and valid changes, when edit is submitted, then the Post is updated, its stable identity is preserved, and a new version is returned.
2. Given a non-author without Permission or an inactive Membership, when edit is submitted, then the server rejects it and preserves all Post fields and Attachments.
3. Given a stale version, when edit is submitted, then the server returns a conflict with current safe state and does not overwrite the newer edit.
4. Given a client changes author, Community, publication time, Space identity outside the move action, or another server-owned field, when edit is submitted, then the claim is ignored or rejected and ownership remains unchanged.
5. Given an authorized actor requests a Post move between the Community root and an active same-Community Space, when source and target access, Permission, rules acceptance, current version, visibility impact, Attachments, Comments, Feeds, Search, Bookmarks, Mentions, and Deliveries are validated and confirmed, then the nullable Space assignment changes once and downstream projections reconcile; a cross-Community or inaccessible move changes nothing.

### States and accessibility

Show loading, editing, unsaved, saving, saved, validation, stale conflict, forbidden, and recovery;
preserve Account-entered text on failure, focus the error summary, and provide a non-destructive compare path.

### Verification

- Domain/unit: edit authority, immutable ownership, validation, and version rules.
- Integration/contract: author/delegated edit, forbidden, stale, and forged-field cases.
- Component: editor, unsaved warning, validation, conflict comparison, and recovery states.
- Journey/live: race two edits and reconcile without silently losing either version.
- Manual/rendered: keyboard editor, long content, screen-reader errors, and reflow review.

## L2-CONT-003: Enforce Post visibility

**Traces to:** L1-CONT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every Post read is server-filtered by publication state, Community and optional Space visibility,
current Membership, Role, Permission, Block, moderation, and retention state; drafts remain
owner/authorized only.

### Acceptance criteria

1. Given a published Post and viewer authorized for both its Community audience and optional current Space, when detail, Feed, Search result, Tag list, export, or realtime delivery is requested, then the permitted Post representation may be returned.
2. Given a draft, removed, hidden, quarantined, or restricted Post, when an unauthorized viewer requests it directly or indirectly, then no content, Attachment, count, or existence detail is disclosed beyond policy.
3. Given a viewer loses Membership or a Post becomes restricted after being loaded, when the next request or event is evaluated, then current state denies access and stale client data grants no action.
4. Given a Post identifier belongs to another Community than the route or query scope, when it is requested, then the server rejects it and returns no cross-Community data.
5. Given the viewer and Post author have a Block in either direction, when any authenticated Post projection is requested, then the shared-Community Block matrix suppresses body, Attachments, author attribution, snippets, engagement, and links without treating the Block as a Community-wide Moderation Action.

### States and accessibility

Use distinct owner draft, member content, removed/hidden where disclosure is safe, and generic
unavailable states; provide accessible recovery navigation without naming private content.

### Verification

- Domain/unit: visibility matrix across lifecycle, Membership, Permission, Block, and moderation.
- Integration/contract: detail/list/Feed/Search/export/realtime and cross-Community probes.
- Component: draft label, restricted/removed/unavailable, and access-changed states.
- Journey/live: change Membership and moderation state while two viewers hold the Post open.
- Manual/rendered: privacy copy, keyboard recovery, screen-reader, and cached-view review.

## L2-CONT-004: Remove and restore a Post

**Traces to:** L1-CONT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Post removal is a server-authorized lifecycle transition with declared author/moderator behavior,
restoration window, Comment treatment, Attachment retention, and irreversible purge policy.

### Acceptance criteria

1. Given the author or Membership with removal Permission and an eligible Post, when removal is confirmed, then the Post leaves ordinary reads immediately and its removal actor/reason category is recorded according to policy.
2. Given the actor lacks authority or the Post is in an incompatible state, when removal is requested, then the server rejects it and preserves Post, Comment, and Attachment state.
3. Given a removed Post is within an authorized restoration window, when restore is confirmed, then it returns to the permitted prior state without duplicate Comments, Reactions, Bookmarks, or Attachments.
4. Given retention expires or irreversible purge is lawfully required, when purge completes, then Post content and Attachments are deleted/de-identified according to policy while required audit evidence remains minimized.

### States and accessibility

Show impact summary, confirmation, removed, restore available, restore forbidden/expired, and purged;
destructive controls are explicit, focus-contained, and never color-only.

### Verification

- Domain/unit: removal authority, lifecycle, restoration, dependent-data, and purge rules.
- Integration/contract: author/moderator removal, forbidden, restore, expiry, and retention processing.
- Component: confirmation, removed representation, restore, expiry, and error states.
- Journey/live: remove and restore with existing Comments, Reactions, Bookmarks, and Attachment.
- Manual/rendered: keyboard/dialog focus, screen-reader, reduced-motion, and copy review.

## L2-CONT-005: Stage an Attachment

**Traces to:** L1-CONT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

The server issues a bounded upload intent for an authorized Account, Community, purpose, size, and
declared media type; uploaded bytes remain untrusted and non-public until processing succeeds.

### Acceptance criteria

1. Given an active Membership with Permission and an allowed purpose, size, count, and declared type, when staging is requested, then one expiring upload intent tied to the Account and Community is issued.
2. Given limits, Permission, Community state, or declared type is invalid, when staging is requested, then no usable upload intent or public Attachment is created.
3. Given uploaded bytes exceed the bound or do not match the allowed content signature, when upload completes, then the Attachment is rejected and cannot be attached or retrieved publicly.
4. Given an upload intent is expired, replayed for another Account, or used for another Community/purpose, when upload is attempted, then it is rejected without altering an existing Attachment.

### States and accessibility

Show selecting, client precheck, uploading with textual progress, paused/retry if supported, rejected,
cancelled, and uploaded-for-processing; native input remains labeled and keyboard operable.

### Verification

- Domain/unit: purpose, Permission, limit, expiry, ownership, and binding rules.
- Integration/contract: stage, upload, size/signature mismatch, expiry, replay, and cross-scope cases.
- Component: picker, progress, cancel/retry, rejection, and processing handoff.
- Journey/live: upload valid and disguised/oversized files in two Community contexts.
- Manual/rendered: keyboard file input, progress announcement, reflow, and error association.

## L2-CONT-006: Process an Attachment

**Traces to:** L1-CONT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

An uploaded Attachment passes authoritative malware/safety validation and media processing before it
can become approved; processing is retry-safe and preserves a stable observable state.

### Acceptance criteria

1. Given staged bytes match a supported type and pass configured safety checks, when processing completes, then the Attachment becomes approved with authoritative metadata and required safe variants.
2. Given malware, unsafe content, corruption, decompression risk, or unsupported encoding is detected, when processing completes, then the Attachment becomes rejected/quarantined and no public variant is available.
3. Given a transient processing failure, when retry occurs, then one final Attachment state is produced without duplicate variants or references.
4. Given a Post publish is attempted while a required Attachment is pending or failed, when publication is evaluated, then the Post remains unpublished and returns the current Attachment reason.

### States and accessibility

Show queued, scanning/processing, approved preview, retrying, rejected, and service-error states;
status text is announced at meaningful boundaries, not as noisy percentage updates.

### Verification

- Domain/unit: processing transitions, approval gate, rejection, and retry idempotency.
- Integration/contract: valid, corrupt, disguised, unsafe, transient-failure, and publish-gate cases.
- Component: queued, progress, preview, rejected, retry, and failure states.
- Journey/live: process valid/invalid media and prove only approved content can publish.
- Manual/rendered: screen-reader status, keyboard retry/remove, reduced-motion, and preview review.

## L2-CONT-007: Protect Attachment access

**Traces to:** L1-CONT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Attachment retrieval and deletion derive authorization from current owning content, Account,
Community, Membership, approval, and retention state rather than possession of a storage path.

### Acceptance criteria

1. Given a viewer may access owning content and its approved Attachment, when retrieval is requested, then a bounded safe representation or expiring delivery authorization is returned with correct type and download policy.
2. Given the viewer loses access, the Attachment is pending/rejected, or its content is removed, when retrieval is attempted, then no bytes or storage details are disclosed even if an old application URL is known.
3. Given an Attachment is referenced by eligible content, when arbitrary deletion is requested, then the server rejects it or performs the declared coordinated content transition without leaving a broken reference.
4. Given an unreferenced staged Attachment exceeds its retention window, when cleanup runs, then it is deleted once without touching another Account's or Community's Attachment.

### States and accessibility

Show loading, preview/download, processing, unavailable, removed, and failed-media fallback states;
downloads have meaningful names, media controls are accessible, and decorative images are marked.

### Verification

- Domain/unit: owning-content access, approved state, reference, and cleanup rules.
- Integration/contract: authorized/unauthorized retrieval, stale authorization, deletion, and cleanup.
- Component: preview, download, fallback, processing, and unavailable states.
- Journey/live: retrieve, lose Membership, retry old URL, and run orphan cleanup across Communities.
- Manual/rendered: keyboard media/download, alternative text, captions where applicable, and reflow.

## L2-CONT-008: Add a Comment

**Traces to:** L1-CONT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

An eligible Membership can add one bounded Comment to a commentable Post or supported parent Comment
in the same Community and inherited optional Space, subject to current Space access/rules, safety,
depth, and rate policy; a Comment cannot select or move scope independently of its Post.

### Acceptance criteria

1. Given an accessible published Post, active Membership with Permission, an active Space when scoped and accepted applicable Space rules, and valid text, when a top-level Comment is submitted, then one Comment with immutable author, Community ownership, and inherited nullable Space identity is committed.
2. Given a valid parent Comment on the same Post and depth remains within policy, when a reply is submitted, then it is attached to that conversation with stable ordering identity.
3. Given the parent belongs to another Post/Community, the Post or parent is authored by the other Account in a blocked pair, conversation is locked, a Moderation Action forbids interaction, or depth/limits are exceeded, when submit is attempted, then no Comment is created.
4. Given the same idempotent submission is retried, when the first committed, then the same Comment is returned and counts increase only once.

### States and accessibility

Show composing, submitting, posted, validation, locked, throttled, forbidden, and retry states;
reply context is named, errors associate with the editor, and new Comment focus is deliberate.

### Verification

- Domain/unit: eligibility, parent scope, depth, lock, safety, and idempotency rules.
- Integration/contract: top-level/reply, cross-Post/Community, blocked, locked, limit, and replay cases.
- Component: Comment editor, reply context, validation, busy, and locked states.
- Journey/live: Comment and reply, then attempt cross-Community and locked submissions.
- Manual/rendered: keyboard editor, screen-reader context, reflow, and announcement review.

## L2-CONT-013: Commit content atomically

**Traces to:** L1-CONT-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Each content mutation commits its Post or Comment state, Tag/Attachment associations, counters, and
durable projection intent coherently before returning success or emitting realtime updates.

### Acceptance criteria

1. Given a valid mutation, when persistence commits, then the response represents the committed version and any externally visible event refers to that same Community and resource version.
2. Given persistence, association, or invariant enforcement fails, when the transaction rolls back, then no partial Post, Comment, Tag, Attachment reference, or counter is visible and no success event is emitted.
3. Given event delivery or projection update fails after commit, when retry occurs, then the committed content is not duplicated and the projection converges through an observable retry path.
4. Given concurrent mutations affect the same aggregate or count, when they commit, then version/invariant policy produces consistent state and rejected requests return conflict without fabricated success.

### States and accessibility

Show pending, committed, delayed refresh, conflict, retry/reconcile, and failed states; optimistic UI
is visibly provisional, rollback is announced once, and focus remains on the originating control.

### Verification

- Domain/unit: version, association, counter, and event-intent invariants.
- Integration/contract: transaction rollback, post-commit delivery failure, retry, and concurrency.
- Component: optimistic/pending, commit, rollback, delayed projection, and conflict states.
- Journey/live: inject persistence and delivery failures and verify response, Feed, counts, and events.
- Manual/rendered: status clarity, focus stability, screen-reader announcements, and offline retry.

## L2-CONT-015: Resolve Mentions and Quote References

**Traces to:** L1-CONT-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Mentions and Quote References are stable content relationships resolved under current visibility and
safety policy; typed display text alone never identifies a target or grants access.

### Acceptance criteria

1. Given an author enters a supported Mention token, when content is validated, then the server resolves it to one eligible Profile identity under Community, Membership, visibility, Block, and Moderation state; ambiguous or ineligible text remains non-linked or returns privacy-safe validation without confirming a hidden Account.
2. Given an author quotes an eligible Post or Comment, when content is committed, then a stable Quote Reference records target identity, target version, authoring context, and a permitted bounded excerpt cache that is explicitly nonauthoritative at later render time; a cross-Community, inaccessible, removed, or prohibited target creates no reference.
3. Given a Post or Comment with new Mentions commits, when post-commit processing runs, then each still-eligible target receives at most one Notification for that content version, self-Mentions and unchanged edit retries create none, and Delivery rechecks current access and Block state.
4. Given content is edited, when Mention and Quote Reference differences commit, then added eligible Mentions may notify once, removed or unchanged Mentions do not notify, handle changes do not retarget stored identities, and failure creates neither partial reference state nor premature Notification.
5. Given a referenced Profile or content target is edited or corrected or changes handle, visibility, Block, Moderation, deletion, or retention state, when the containing content is read, searched, exported, indexed, or delivered, then the current authorized target version regenerates the excerpt or renders a neutral edited/unavailable reference; privacy/safety corrections invalidate stale excerpt caches before the next disclosure, the reference never grants access, and a deleted identity cannot be reconstructed from stale display data.

### States and accessibility

Editors distinguish resolving, resolved, ambiguous, unavailable, quoted, validation, and stale-target
states; rendered links identify their purpose, unavailable references remain understandable as text.

### Verification

- Domain/unit: resolution, identity stability, quote eligibility, edit diff, and no-access-grant rules.
- Integration/contract: commit/outbox, Notification deduplication, handle change, Block, deletion, and export.
- Component: autocomplete, ambiguity, quote preview, validation, and unavailable-reference rendering.
- Journey/live: mention and quote, edit under retry, rename and block a target, then delete it.
- Manual/rendered: keyboard autocomplete, screen reader, privacy-safe errors, focus, and reflow.
