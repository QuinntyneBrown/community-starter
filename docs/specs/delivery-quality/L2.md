# QUAL — Delivery, quality, and operations detailed requirements

## L2-QUAL-001: Test selection follows risk and evidence scope

**Traces to:** L1-QUAL-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Verification shall use the layer closest to the risk and add cross-stack evidence for important community journeys. Domain/unit tests shall cover transitions, policies, and pure transforms; application/component tests shall cover coordination and UI states; integration tests shall cover HTTP, authentication, authorization, middleware, serialization, persistence, migrations, and providers; contract tests shall cover client/server compatibility; Playwright shall cover user outcomes; and published-artifact or staging smoke tests shall cover deployment behavior.

### Acceptance criteria

1. Given a changed product-defining rule, when the verification plan is reviewed, then focused domain/unit evidence covers allowed and rejected behavior and unchanged state.
2. Given persistence, authentication, authorization, serialization, middleware, or endpoint behavior changes, when evidence is reviewed, then integration tests exercise the relevant real boundary.
3. Given Angular logic or component behavior changes, when evidence is reviewed, then adjacent focused tests cover observable states and actions.
4. Given a user journey or cross-stack behavior changes, when it is claimed complete, then browser plus live-contract/full-stack evidence exists.
5. Given deployment or public routing changes, when claimed complete, then the published artifact is verified over HTTP rather than only through development servers.

### States and accessibility

Risk selection shall include material loading, empty, validation, error, conflict, authorization, destructive, reconnect, responsive, keyboard, and reduced-motion states.

### Verification

- Domain/unit: Audit changed domain risks against focused tests.
- Integration/contract: Audit provider and public-boundary risks against integration/contract suites.
- Component: Audit changed UI states against adjacent tests.
- Journey/live: Audit important outcomes against Playwright and live/full-stack coverage.
- Manual/rendered: Review the risk-to-evidence map and published behavior.

## L2-QUAL-002: Product rules and component behavior have focused tests

**Traces to:** L1-QUAL-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every product-defining invariant shall be unit-tested directly, including rejected transitions and proof that state remains unchanged. Application tests shall verify handler coordination, validation boundaries, persistence-before-notification, and typed failures. Backend tests shall prefer plain xUnit assertions or an explicitly approved permissively licensed assertion library. Angular service and component tests shall remain adjacent and focus on observable behavior, including applicable success and material failure states.

### Acceptance criteria

1. Given a community invariant, when its test suite runs, then at least one accepted and one rejected case execute the rule directly and rejected state is asserted unchanged.
2. Given a state-changing handler, when persistence fails in a focused test, then the test proves no notification or success result is emitted.
3. Given validator and policy responsibilities, when tests are inspected, then request-shape failures and state-dependent business failures are exercised at their respective layers.
4. Given an Angular service/component, when its adjacent tests run, then observable loading, empty, success, validation/error, and interaction states are covered where applicable.
5. Given a backend assertion dependency is proposed, when reviewed, then it is either unnecessary because plain xUnit suffices or has an approved permissive license and documented value.

### States and accessibility

Component tests shall query semantic roles/names, verify field-error association and focus behavior where applicable, and cover disabled explanations rather than only CSS classes.

### Verification

- Domain/unit: Run invariant, handler, validator, and pure-transform suites.
- Integration/contract: Confirm focused failures map correctly at public boundaries.
- Component: Run adjacent Angular service/component tests.
- Journey/live: Select a representative rule and prove it through the live journey.
- Manual/rendered: Review assertions for behavior rather than implementation detail.

## L2-QUAL-003: Integration tests use real provider behavior where it matters

**Traces to:** L1-QUAL-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Integration tests shall exercise authentication and authorization, Community scoping, persistence mappings, relational constraints, migrations, serialization, middleware, error mapping, and public endpoint contracts. When behavior can differ by database or another provider, tests shall use the production provider or a faithful production-compatible instance rather than an in-memory substitute. Test data shall be isolated by database or transaction and cleaned deterministically.

### Acceptance criteria

1. Given a provider-specific query, constraint, transaction, migration, or serialization behavior, when its integration test runs, then it uses the real production provider class and can expose provider differences.
2. Given two integration tests run in any order or concurrently as configured, when they complete, then isolated data prevents one test from affecting the other.
3. Given authentication, authorization, and Community-scope matrices, when integration tests run, then allowed and cross-Community/forbidden cases are covered through the HTTP boundary.
4. Given a schema migration, when the integration suite runs, then clean application and supported upgrade behavior are exercised before data-dependent endpoints.

### States and accessibility

Integration responses shall carry stable information needed for accessible validation, forbidden, unavailable, and conflict presentation without exposing provider diagnostics.

### Verification

- Domain/unit: Test provider-independent policy separately.
- Integration/contract: Run the complete real-provider suite with deterministic isolation/cleanup.
- Component: Consume representative real error contracts in UI tests.
- Journey/live: Run a critical journey against the same provider class.
- Manual/rendered: Inspect test infrastructure, cleanup, and provider parity.

## L2-QUAL-004: Contracts, journeys, and deployments receive cross-stack evidence

**Traces to:** L1-QUAL-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Typed DTO and error compatibility shall be checked through generated schema validation or live smoke tests. Playwright specs shall be journey-oriented and prove user outcomes across feature composition. Critical paths shall run against a live API whenever the default browser suite fakes the backend. Deployment tests shall verify route ownership, assets, headers, migrations, health, and critical API behavior from the exact published artifact or staging deployment.

### Acceptance criteria

1. Given a client or server DTO/error contract changes, when CI runs, then schema/generated compatibility or a live contract smoke detects incompatible shape or semantics.
2. Given a critical community journey, when browser evidence is reviewed, then a live/full-stack run proves server authorization, persistence, and returned state even if the fast default suite uses a fake.
3. Given a release artifact, when deployment checks run, then migrations, health, marketing root, app deep links, assets/headers, and critical API behavior are exercised over HTTP.
4. Given only a mock, unit test, or faked browser result exists, when a cross-stack requirement status is evaluated, then it is not marked implemented.

### States and accessibility

Journey and deployment evidence shall include representative error, conflict, authorization, responsive, keyboard, and reduced-motion behavior where affected.

### Verification

- Domain/unit: Validate DTO/schema generation inputs where deterministic.
- Integration/contract: Run compatibility checks and live API smoke tests.
- Component: Verify client rendering of representative contract variants.
- Journey/live: Run fast deterministic and critical live Playwright suites.
- Manual/rendered: Inspect published routing, headers, and cross-stack evidence scope.

## L2-QUAL-005: Tests are behavioral, isolated, and deterministic

**Traces to:** L1-QUAL-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Tests shall be named for observable behavior and expected outcome, use Arrange/Act/Assert or Given/When/Then consistently, and have one behavioral reason to fail. Valid domain state shall use small builders or fixtures rather than mocks that erase product meaning. Clocks, identifiers, external services, and startup readiness shall be deterministic at defined seams. Tests shall not depend on order, a developer's seeded state, arbitrary sleeps, or arbitrary timeouts.

### Acceptance criteria

1. Given a test name and failure, when reviewed, then the observable behavior and expected outcome are clear and one product reason explains the failure.
2. Given domain state is needed, when the test arranges it, then a small valid builder/fixture preserves real invariants rather than mocking the domain into arbitrary shape.
3. Given time, identity, or external behavior affects the outcome, when the test runs repeatedly, then deterministic seams produce the same result.
4. Given asynchronous readiness or UI change, when the test waits, then it observes user-visible state, a network event, or an explicit readiness signal rather than sleeping for a fixed duration.
5. Given the suite is shuffled or run from clean state, when it completes, then no test depends on another test or a developer seed.

### States and accessibility

Behavioral assertions shall prefer semantic user outcomes, including accessible name/role/state, over private component structure or incidental CSS.

### Verification

- Domain/unit: Repeat and shuffle deterministic focused suites.
- Integration/contract: Run isolated tests in configured parallel/order modes.
- Component: Check semantic assertions and eliminate arbitrary timers.
- Journey/live: Run journeys with deterministic service startup/readiness.
- Manual/rendered: Review naming, fixtures, waits, and behavioral failure scope.

## L2-QUAL-006: Browser journeys use stable page objects and faithful dependencies

**Traces to:** L1-QUAL-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Playwright specifications shall be organized around member outcomes and use capability-oriented page objects. Locators shall prefer roles, labels, and visible names; test IDs shall be reserved for cases without stable semantics. A fake backend may improve speed and determinism, but it shall implement the documented API and failure contracts, be maintained as a test dependency, and never serve as the sole evidence for critical server behavior.

### Acceptance criteria

1. Given a Playwright spec, when its structure is reviewed, then it describes a community journey and delegates stable interaction details to capability-oriented page objects.
2. Given an element with a meaningful role, label, or visible name, when located, then the test uses that semantic locator rather than CSS structure or a test ID.
3. Given a fake backend response, when compared with live OpenAPI/contract behavior, then DTOs, status codes, Problem Details, latency/state branches, and relevant events remain faithful.
4. Given a critical journey passes only against the fake, when readiness is evaluated, then live/full-stack evidence is still required.

### States and accessibility

Role/label/name-first locators shall make accessible semantics part of browser stability, and journeys shall cover focus/keyboard behavior for changed flows.

### Verification

- Domain/unit: Validate fake response builders against shared schemas where possible.
- Integration/contract: Run fake/live contract parity checks.
- Component: Verify stable roles, labels, and visible names.
- Journey/live: Execute deterministic fake and critical live page-object suites.
- Manual/rendered: Review selector semantics and page-object capability boundaries.

## L2-QUAL-007: Requirement status is backed by appropriately scoped evidence

**Traces to:** L1-QUAL-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Acceptance and integration tests that prove a named behavior shall carry a `Traces to: L2-...` reference. `docs/specs/README.md` shall maintain a lightweight coverage table linking each requirement to current evidence and status. Evidence shall not be overstated: prose, mocks, acceptance criteria, focused tests, live journeys, and production telemetry prove progressively different scopes.

### Acceptance criteria

1. Given an acceptance or integration test claims a named requirement, when its source is inspected, then exactly one or more explicit `Traces to: L2-...` references identify what it proves.
2. Given a requirement coverage row, when its links are followed, then evidence exists at the layers demanded by the requirement and the stated status matches that evidence.
3. Given a mock or unit test is the strongest evidence, when a cross-stack behavior is assessed, then its coverage row does not claim live implementation.
4. Given a requirement is superseded, when the coverage index is updated, then the replacement and historical evidence remain linked rather than silently removed.

### States and accessibility

Coverage shall explicitly identify accessibility, responsive, failure, and manual-rendered evidence where those states are part of the requirement.

### Verification

- Domain/unit: Validate requirement and trace identifier syntax.
- Integration/contract: Resolve test-to-requirement links and evidence files.
- Component: Confirm component evidence names relevant state coverage.
- Journey/live: Confirm critical L2 entries link live journey results.
- Manual/rendered: Audit coverage status against actual evidence scope.

## L2-QUAL-008: Canonical restore, build, test, and e2e commands work cleanly

**Traces to:** L1-QUAL-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

The root README shall expose canonical commands that restore, build, and test the backend solution and install, build, unit-test, and end-to-end-test the frontend. Automation shall use pinned SDK/package-manager inputs and `npm ci`, not `npm install`. Optional `eng/` wrappers shall preserve clear exit codes, run the same underlying commands locally and in CI, and require no undocumented developer state.

### Acceptance criteria

1. Given a clean checkout with documented prerequisites, when backend restore, no-restore build, and no-build test commands run, then they succeed or fail with truthful non-zero status.
2. Given a clean frontend workspace, when `npm ci`, production build, unit tests, and e2e commands run, then the committed lockfile is honored and not modified.
3. Given an `eng/` wrapper, when an underlying command fails, then the wrapper returns a non-zero exit code and does not conceal the failing command/result.
4. Given local and CI command definitions, when compared, then they use the same toolchain inputs and quality behavior rather than divergent hidden flags.

### States and accessibility

Canonical frontend and journey commands shall include accessibility checks required by changed surfaces, not only visual rendering.

### Verification

- Domain/unit: Test wrapper argument and exit-code behavior where scripted.
- Integration/contract: Run the documented backend/frontend command sequence from clean state.
- Component: Execute the canonical component-test command.
- Journey/live: Execute the canonical e2e and live-contract commands.
- Manual/rendered: Follow README setup exactly and inspect output clarity.

## L2-QUAL-009: Repository formatting and tracked artifacts are governed

**Traces to:** L1-QUAL-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Tracked text shall use UTF-8, final newline, no trailing whitespace, and LF except Windows-only scripts when required. JSON, YAML, HTML, SCSS, TypeScript, and Markdown structure shall use two spaces; C# shall use four. Root and stack-specific `.editorconfig`, frontend Prettier, and build-enforced .NET analyzers shall match CI. Lockfiles, pinned SDK files, migrations, required deterministic contracts, and intentionally distributed documents shall be tracked; build output, local databases, coverage, test-runner screenshots/results, IDE state, secrets, credentials, and real personal data shall not. Changes shall not reformat unrelated files.

### Acceptance criteria

1. Given tracked source, when formatting and whitespace checks run, then encoding, line ending, indentation, final-newline, and trailing-whitespace policies pass for each file class.
2. Given local formatter/analyzer commands and CI, when their configuration is compared, then the same `.editorconfig`, Prettier, and analyzer rules are enforced.
3. Given source-control status after build and tests, when inspected, then generated output, local databases, coverage, results/screenshots, IDE state, credentials, and secrets are not included by default.
4. Given a focused change, when its diff is inspected, then required lockfiles/migrations/contracts are present and unrelated files are not reformatted.

### States and accessibility

Generated accessibility or visual evidence may be retained only through an intentional reviewed artifact process; transient runner output remains untracked.

### Verification

- Domain/unit: Run format, whitespace, analyzer, and tracked-artifact policy checks.
- Integration/contract: Build/test and confirm worktree artifact hygiene.
- Component: Run Prettier and component lint/type checks.
- Journey/live: Run browser tests and verify transient artifacts stay untracked.
- Manual/rendered: Review focused diff and required intentional documents.

## L2-QUAL-010: Iteration starts narrow and ends with every required gate

**Traces to:** L1-QUAL-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

During implementation, engineers shall run the narrowest relevant test for fast feedback, then complete every gate required by the affected scope before handoff. Backend-only changes require focused unit/integration evidence then backend build/test; frontend-only changes require focused component tests then frontend build/test; cross-stack journeys require both stacks plus Playwright and live contract; visual changes require screenshots and accessibility checks.

### Acceptance criteria

1. Given a backend-only behavior change, when handed off, then focused relevant tests and the canonical backend build/test have passed.
2. Given a frontend-only behavior change, when handed off, then focused adjacent tests and the canonical frontend build/test have passed.
3. Given a cross-stack change, when handed off, then both stack gates, browser journey, and live-contract/full-stack evidence have passed.
4. Given a substantial visual change, when handed off, then representative viewport screenshots, automated checks, and manual accessibility/visual inspection accompany component/build evidence.

### States and accessibility

Final gates shall cover all materially affected interaction, error, responsive, keyboard, contrast, zoom/reflow, and reduced-motion states.

### Verification

- Domain/unit: Inspect focused and full backend results.
- Integration/contract: Inspect boundary and live-contract results.
- Component: Inspect focused and full frontend results.
- Journey/live: Inspect deterministic and live journey results.
- Manual/rendered: Review screenshots and manual accessibility notes.

## L2-QUAL-011: Database, marketing, deployment, and documentation changes have specialized checks

**Traces to:** L1-QUAL-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Database changes shall include migration generation/review, clean-database apply, supported upgrade rehearsal, and integration tests. Marketing/deployment changes shall include static validation/build and published routing smoke. Documentation-only changes shall validate affected links, formatting, commands, identifiers, and claims. No specialized change may rely solely on the generic unit suite.

3. Given a documentation-only change, when handed off, then affected links, requirement/ADR identifiers, formatting, commands, and evidence/status claims validate.
4. Given a specialized check fails, when readiness is evaluated, then generic green tests do not override the failure.

### States and accessibility

Marketing/deployment checks shall include keyboard, zoom/reflow, reduced motion, and representative viewport behavior when the changed surface is user-facing.

### Verification

- Domain/unit: Validate migration/docs/static policy tooling.
- Integration/contract: Apply databases and smoke published routes/contracts.
- Component: Run affected UI/static component checks.
- Journey/live: Exercise upgraded data and deployed public/member journeys.
- Manual/rendered: Review migrations, commands, links, headers, and rendered pages.

## L2-QUAL-012: CI restores, formats, analyzes, and builds every stack

**Traces to:** L1-QUAL-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every pull request pipeline shall restore from committed lockfiles and pinned SDKs, run format/lint/analyzer checks, and build backend and frontend with the documented warnings policy enforced. CI shall execute the same canonical commands as local development and shall fail on formatting drift, analyzer violations, type/template errors, warnings covered by policy, lockfile drift, or build failure.

### Acceptance criteria

1. Given a pull request, when CI restores, then pinned SDK/package-manager inputs and lockfiles determine dependencies and no install step mutates them.
2. Given formatting, lint, analyzers, strict TypeScript/templates, or warnings-policy violations, when CI runs, then the responsible job fails with actionable output.
3. Given backend and frontend source, when CI builds them, then production-equivalent build configurations succeed before test or artifact promotion gates can be considered complete.
4. Given local canonical commands, when compared with CI steps, then substantive format/analyze/build behavior is the same.

### States and accessibility

Frontend static gates shall include available template accessibility linting, while acknowledging that automated checks do not prove accessible behavior.

### Verification

- Domain/unit: Test workflow scripts and policy configuration where practical.
- Integration/contract: Run clean restore and both production builds in CI-shaped environment.
- Component: Run frontend lint/type/template build gates.
- Journey/live: Confirm no journey starts from an artifact that failed static/build gates.
- Manual/rendered: Review job output, parity, and warnings policy.

## L2-QUAL-013: CI runs layered tests and validates source artifacts

**Traces to:** L1-QUAL-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

CI shall run unit, application, component, integration, contract, and deterministic browser suites required by scope; validate documentation links and required generated artifacts; and scan dependencies and source for vulnerabilities and secrets. Jobs shall preserve trustworthy exit codes and useful diagnostics. Failures shall block merge or artifact publication except through an explicit authorized exception process.

### Acceptance criteria

1. Given a pull request, when CI runs, then required unit, component, integration, contract, and deterministic journey suites execute after successful prerequisite gates.
2. Given a broken documentation link, stale required generated contract, secret finding, or policy-blocking dependency issue, when validation runs, then CI fails.
3. Given a test or scanner process exits non-zero, when wrappers/jobs complete, then the pipeline remains failed and does not publish a releasable artifact.
4. Given an exception is authorized, when applied, then owner, scope, rationale, compensating control, and review/expiry trigger are recorded.

### States and accessibility

CI shall run automated accessibility checks appropriate to changed surfaces, while manual keyboard/screen-reader/reflow evidence remains separately required by risk.

### Verification

- Domain/unit: Run all focused/unit suites and policy validators.
- Integration/contract: Run integration, schema/contract, docs, dependency, and secret checks.
- Component: Run component and automated accessibility suites.
- Journey/live: Run deterministic browser journeys.
- Manual/rendered: Review diagnostics, coverage, exceptions, and missing manual evidence.

## L2-QUAL-014: One verified artifact is promoted with protected credentials

**Traces to:** L1-QUAL-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

The pipeline shall publish deployable artifacts once after required gates and promote the same immutable artifact through environments. Deployment shall run only from protected branches/environments using OIDC or equivalent short-lived scoped credentials. Untrusted pull requests shall have no production credentials. Environments intentionally lacking configuration shall skip or stop deployment explicitly rather than weaken build and test behavior.

### Acceptance criteria

1. Given an artifact passes required gates, when published, then its immutable identifier or digest is recorded with source revision and evidence.
2. Given promotion from one environment to another, when deployment occurs, then the artifact identifier/digest is unchanged and no privileged rebuild is performed.
3. Given an untrusted pull request, when pipeline permissions are inspected, then it cannot obtain protected secrets, cloud credentials, or production deployment authority.
4. Given an environment is intentionally absent, when deployment conditions evaluate, then the result is explicit and earlier gates still execute normally.

### States and accessibility

Any maintenance or unavailable page used during deployment shall be sanitized, keyboard operable, focus-visible, and provide understandable retry/support information.

### Verification

- Domain/unit: Validate artifact metadata and workflow condition scripts.
- Integration/contract: Exercise build, publish, skip, and promotion paths.
- Component: Test maintenance/unavailable presentation if supplied.
- Journey/live: Compare deployed digest and run post-promotion smoke.
- Manual/rendered: Review environment protections, permissions, and credential lifetime.

**Source:** product brief + research

**Related:** —

After deployment and before promotion completion, automated smoke tests shall verify process health and dependency readiness where distinguished, root marketing ownership, refreshed application deep links, critical API behavior, authentication entry, realtime endpoint routing where used, static assets and metadata, cache/security headers, and migration readiness. 4. Given any release-critical smoke fails, when the pipeline responds, then promotion stops and the documented rollback or forward-fix path is initiated with ownership.

### States and accessibility

### Verification

- Domain/unit: Validate smoke route/expectation configuration.
- Integration/contract: Run HTTP, header, migration, health, and API smoke against deployment.
- Component: Confirm deployed application/public components hydrate/render correctly.
- Journey/live: Execute the release-critical public and authenticated journeys.

## L2-QUAL-020: Definition of done and production-readiness claims match evidence

**Traces to:** L1-QUAL-006

**Source:** product brief + research

**Related:** —

A change is done only when its requested user outcome and material failures work, the server owns affected invariants, accepted design-system behavior is implemented, appropriately traced tests pass, builds/static checks pass, accessibility/security/data/deployment effects are addressed, documentation/specs/decisions/changelog are current, the diff is focused, and handoff reports verification and genuine remaining risk. Production-readiness language shall separately state the status of independent security, privacy, accessibility, recovery, load, and operational review; demos and vertical slices shall be labeled honestly.

### Acceptance criteria

1. Given a change is marked done, when audited, then the user outcome, material failures, server authority, design-system conformance, tests, builds, documentation, and focused-diff evidence all pass.
2. Given accessibility, security, data migration, or deployment effects exist, when handoff occurs, then their evidence, decision, or remaining risk is explicit rather than assumed from green unit tests.
3. Given a product is described as production-ready, when evidence is reviewed, then independent security, privacy, accessibility, recovery, load, and operational review status is stated individually.
4. Given the artifact is a seeded demo, initial vertical slice, or has unresolved review gaps, when documentation or marketing describes it, then those limitations remain visible.
5. Given any required gate or evidence is missing, when completion is assessed, then the change remains incomplete rather than redefining done around available evidence.

### States and accessibility

Completion evidence shall include applicable loading, empty, validation, conflict, authorization, destructive, responsive, keyboard, zoom/reflow, contrast, and reduced-motion states.

### Verification

- Domain/unit: Audit invariant and static-gate evidence.
- Integration/contract: Audit provider, migration, security, and contract evidence.
- Component: Audit component, design-system, and accessibility evidence.
- Journey/live: Audit end-to-end user outcome and material failures.
- Manual/rendered: Compare readiness claims, reviews, screenshots, docs, and remaining risk.

## L2-QUAL-021: Provision environments through reviewed infrastructure code

**Traces to:** L1-QUAL-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Development, staging, and production infrastructure shall be declared under `infra/` through a
reviewable, repeatable infrastructure-as-code workflow covering network boundaries, compute,
relational data, object storage, DNS and TLS, workload identities, secrets references, telemetry,
backups, and configured optional services. Manual emergency changes require capture and drift repair.

### Acceptance criteria

1. Given a supported empty environment account or subscription, when the reviewed infrastructure workflow runs, then it converges the declared resources, identities, policies, observability, backup, and outputs without undocumented console prerequisites.
2. Given the same desired state is applied again, when no declaration changed, then the plan contains no destructive or unexplained material difference and execution remains idempotent.
3. Given a proposed plan deletes, replaces, exposes, or weakens a stateful or security boundary, when delivery is reviewed, then explicit approval, backup or migration evidence, impact, and recovery path are required before apply.
4. Given deployed infrastructure drifts or an emergency manual change occurs, when drift detection runs, then an owned difference is recorded and reconciled into code or safely reverted.

### States and accessibility

Plan and apply output identifies create, change, replace, delete, no-op, blocked, and drift states in
plain text; destructive scope is explicit and diagnostics do not print secret values.

### Verification

- Domain/unit: Test infrastructure policy, naming, environment, and destructive-plan guards.
- Integration/contract: Validate and plan every supported environment with provider-backed checks.
- Component: Not applicable; validate any rendered infrastructure report for accessible structure.
- Journey/live: Provision a disposable environment, deploy the artifact, smoke it, and destroy only its verified resources.
- Manual/rendered: Security, operations, cost, recovery, and least-privilege review of production plans.

## L2-QUAL-023: Isolate preview and non-production environments

**Traces to:** L1-QUAL-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Preview, development, test, and staging environments shall use distinct least-privilege identities,
data, secrets, domains, callbacks, delivery destinations, analytics, and access policy. They shall not
receive production credentials or unapproved production personal data, and publicly reachable
ephemeral environments shall have authentication, expiry, ownership, cleanup, and indexing controls.

### Acceptance criteria

1. Given an untrusted pull request creates or tests an environment, when credentials are issued, then no production secret, identity, network grant, callback, data store, or delivery destination is available to its code.
2. Given a non-production environment starts, when its data and providers are inspected, then only synthetic or explicitly approved minimized data and environment-specific destinations are present.
3. Given a preview is publicly reachable, when provisioned, then accountable owner, access control, `noindex` behavior, expiry, cost bound, and automated cleanup are configured before its URL is shared.
4. Given a preview expires or its change closes, when cleanup runs, then its compute, data, objects, credentials, DNS, callbacks, and retained artifacts are removed and the verified result is auditable.

### States and accessibility

Environment banners and access pages name non-production status, owner, expiry, and data restrictions
using text and accessible semantics rather than color alone.

### Verification

- Domain/unit: Test environment classification, expiry, routing, and credential policy.
- Integration/contract: Inspect identities, network/data/provider separation, `noindex`, and cleanup.
- Component: Test non-production banner and access-denied/expired states.
- Journey/live: Provision, use, expire, and fully remove a representative preview environment.
- Manual/rendered: Security, privacy, accessibility, and cost review of each environment class.
