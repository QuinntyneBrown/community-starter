# UXDS — Experience and design system detailed requirements

## L2-UXDS-001: Canonical foundations and assets have one source

**Traces to:** L1-UXDS-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The repository shall provide `design-system/tokens.css`, `base.css`, licensed shared assets, a rendered `index.html` reference, and usage/change documentation as the canonical platform-neutral web source. Angular and static marketing shall consume these files through deterministic build or copy steps rather than maintain hand-copied token sets. A referenced missing design-system asset shall fail the build.

### Acceptance criteria

1. Given an Angular or marketing build, when shared styles and assets are resolved, then they originate from the canonical design-system source or its deterministic generated output.
2. Given a canonical token changes, when both surfaces are rebuilt, then the new value appears in the application and marketing without manually editing duplicate token definitions.
3. Given a referenced shared asset is absent, when a consuming surface builds, then the build fails with an actionable non-secret error.
4. Given a shared font, icon, logo, or image, when it is included, then its license and distribution intent are documented.

### States and accessibility

Canonical foundations include focus, disabled, selected, destructive, success, warning, error, and reduced-motion behavior shared across surfaces.

### Verification

- Domain/unit: Validate token and asset references with deterministic tooling.
- Integration/contract: Build both consumers and compare resolved canonical assets.
- Component: Render representative components using only shared foundations.
- Journey/live: Inspect the same core visual semantics on public and member journeys.
- Manual/rendered: Review the design-system reference and asset-license record.

## L2-UXDS-002: Semantic tokens cover all product roles and states

**Traces to:** L1-UXDS-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The design system shall define a limited primitive palette and scale, then expose semantic CSS custom properties by purpose rather than screen or literal color. Semantic coverage shall include surfaces, text, borders, actions, statuses, category/data colors, typography roles and responsive scale, spacing, content widths, control sizes, radius, elevation, z-index, focus, selection, disabled/destructive/success/warning/error states, motion duration/easing, and reduced motion. Themes shall be added only when the product requires them.

### Acceptance criteria

1. Given feature styling, when a shared visual decision is needed, then it uses an existing semantic token or adds a deliberately named semantic role instead of embedding a visually close primitive or literal.
2. Given a token name, when reviewed, then it describes purpose such as surface, text, action, or status rather than a transient page or literal hue.
3. Given component state variants, when rendered, then focus, disabled, selected, destructive, success, warning, error, and motion behavior resolve from documented semantics.
4. Given a proposed light, dark, or high-contrast theme, when no product requirement justifies it, then the starter does not create speculative duplicate token sets.

### States and accessibility

Semantic state tokens shall maintain WCAG 2.2 AA contrast and shall never make color the sole carrier of meaning.

### Verification

- Domain/unit: Lint feature styles for forbidden literals or primitive misuse where practical.
- Integration/contract: Verify token output is stable across app and marketing builds.
- Component: Render every applicable semantic state.
- Journey/live: Exercise status and action states in the primary community journey.
- Manual/rendered: Inspect contrast, token naming, responsive typography, and reduced motion.

## L2-UXDS-003: Components progress from foundations to product signatures

**Traces to:** L1-UXDS-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Shared UI shall progress from foundations to primitives, feedback, composition, product-signature components, and page patterns using the primary journey as its proving ground. A component shall enter the shared library only when two contexts need the same semantics or it is an intentional platform primitive. The starter shall not create a broad abstract library before product behavior and visual states are understood.

### Acceptance criteria

1. Given the first community journey, when UI work begins, then the required reset, type, color, spacing, grid, focus, form, action, and feedback foundations exist before broad feature styling.
2. Given a component is proposed for the shared library, when reviewed, then it is either a deliberate primitive or has at least two contexts requiring the same semantic API.
3. Given a domain-defining interaction such as Membership, contribution, moderation, or trust status, when designed, then an accepted mock or recorded design review shows the pattern identifies the acting role, target and scope, current state, permitted next actions, material consequence, and relevant trust or safety signal rather than presenting an unlabeled generic CRUD form.
4. Given shared component documentation, when a component is applicable, then default, hover, focus-visible, active, disabled, loading, selected, invalid, empty, long-content, narrow-width, and high-contrast states are represented.

### States and accessibility

Every promoted component shall document applicable interaction states, accessible name/role/value behavior, keyboard operation, and responsive constraints.

### Verification

- Domain/unit: Test pure formatting or state transforms extracted from components.
- Integration/contract: Verify shared APIs remain compatible in both consuming contexts.
- Component: Exercise all documented variants and accessibility assertions.
- Journey/live: Use shared primitives and a product-signature component in the core journey.
- Manual/rendered: Review the progression and state gallery for premature or missing abstractions.

## L2-UXDS-004: Angular source is strict, focused, and adjacent

**Traces to:** L1-UXDS-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Angular shall enable strict TypeScript and template checking, use standalone components with `ChangeDetectionStrategy.OnPush`, and add no NgModule without a framework or integration reason. Kebab-case filenames shall use explicit `.component`, `.service`, `.guard`, `.interceptor`, `.model`, and `.spec` roles, with component TypeScript, template, SCSS, and tests adjacent. Types and classes shall use `PascalCase`, values and functions `camelCase`, and boolean names shall read as questions such as `isOpen`, `canModerate`, or `hasError`. Focused files shall extract non-trivial pure transforms into named helpers with focused adjacent tests. Prettier shall enforce single quotes and a 100-character width using the Angular HTML parser.

### Acceptance criteria

1. Given Angular source, when type checking and production build run, then TypeScript and template errors fail the build under strict settings.
2. Given a new component, when inspected, then it is standalone, uses `OnPush`, follows role-bearing kebab-case filenames, and keeps implementation, template, style, and spec adjacent.
3. Given a proposed NgModule, when reviewed, then a documented framework or integration requirement justifies it.
4. Given non-trivial deterministic transformation logic, when extracted, then it has a descriptive named function and focused adjacent tests rather than being buried in a template or oversized component.
5. Given formatting runs, when source is checked, then automated Prettier output is authoritative and hand-alignment does not create drift.

### States and accessibility

Strict templates and adjacent tests shall cover accessible bindings, absent values, and representative component states without runtime null failures.

### Verification

- Domain/unit: Test extracted pure transforms.
- Integration/contract: Run strict type/template checking and the production build.
- Component: Execute adjacent component tests under `OnPush` behavior.
- Journey/live: Run the affected route in the production build.
- Manual/rendered: Inspect file focus, naming, adjacency, and formatter configuration.

## L2-UXDS-005: Presentational components receive data and emit intent

**Traces to:** L1-UXDS-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

New Angular code shall use `inject()` and prefer signal `input()`, `output()`, `signal()`, and `computed()` for local synchronous state. Presentational components shall receive typed product data and emit user intent; pages and services shall own routing, remote calls, and feature coordination. Component APIs shall use canonical product nouns, and only deliberate public components shall be exported from a library's `public-api.ts`.

### Acceptance criteria

1. Given a reusable presentational component, when its source is inspected, then it performs no HTTP call or route navigation and communicates action through a typed output.
2. Given local derived state, when dependencies change, then a computed value derives it rather than an effect or subscription manually synchronizing duplicate state.
3. Given a component API, when names are reviewed, then inputs and outputs use canonical community nouns and describe meaning rather than incidental color or position.
4. Given a component is not a supported library contract, when the public API is inspected, then it is not exported.

### States and accessibility

Presentational inputs shall include the semantic state needed to render names, labels, disabled reasons, validation, selection, and status without reaching into remote services.

### Verification

- Domain/unit: Test computed state and typed intent contracts.
- Integration/contract: Compile consuming pages against deliberate public exports.
- Component: Assert inputs render and outputs emit without transport side effects.
- Journey/live: Complete an action through a page coordinator and presentational component.
- Manual/rendered: Review public APIs and canonical component vocabulary.

## L2-UXDS-006: Remote and derived state remain explicit

**Traces to:** L1-UXDS-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Typed remote contracts, authentication, interceptors, HTTP services, and realtime clients shall remain in the API library. API origins shall be injected and same-origin production shall use relative URLs. Services shall expose narrow methods named for community actions and return the established typed asynchronous primitive. Subscriptions shall not merely copy values between state containers; necessary subscriptions shall handle lifecycle, error, and completion explicitly.

### Acceptance criteria

1. Given a member page needs remote state, when it obtains that state, then it calls a typed API-library method named for a product action rather than assembling ad hoc URLs.
2. Given a production same-origin build, when a request is issued, then it uses relative API or hub paths and no library contains a hard-coded deployment hostname.
3. Given a value is derived solely from synchronous state, when implemented, then it uses a computed value rather than a subscription that copies state.
4. Given a subscription is necessary, when its source errors, completes, or the consumer is destroyed, then lifecycle behavior is explicit and leak-free.

### States and accessibility

Remote state APIs shall distinguish loading, empty, success, validation, conflict, authorization, offline/reconnecting, and retryable failure for accessible page composition.

### Verification

- Domain/unit: Test service mapping and derived-state helpers.
- Integration/contract: Verify typed client contracts and injected relative origins.
- Component: Test every exposed remote-state branch.
- Journey/live: Exercise success, conflict, and reconnect behavior against live contracts.
- Manual/rendered: Inspect API ownership, subscriptions, and error communication.

## L2-UXDS-007: Templates and navigation use semantic web behavior

**Traces to:** L1-UXDS-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Templates shall prefer native buttons, links, labels, headings, lists, forms, and dialogs before ARIA substitutes. Real router links or anchors shall replace click handlers on generic containers. Template expressions shall remain simple, non-trivial derivation shall move to computed values or pure helpers, current Angular control-flow syntax shall be consistent, and repeated rows shall track stable identity.

### Acceptance criteria

1. Given an action, navigation target, form control, heading, list, or dialog, when rendered, then the corresponding native semantic element is used unless a documented limitation requires an accessible substitute.
2. Given navigation, when a user opens it by keyboard, assistive technology, or standard browser link behavior, then it behaves as a real link rather than a generic element with a click handler.
3. Given a template expression becomes non-trivial, when refactored, then derivation moves to a typed computed value or pure helper.
4. Given a reordered or updated list, when Angular renders it, then rows retain identity through stable tracking rather than array position.

### States and accessibility

Semantic roles, names, current/expanded/selected state, heading order, and focus behavior shall remain correct through all control-flow branches.

### Verification

- Domain/unit: Test pure template helpers.
- Integration/contract: Run strict template and accessibility checks.
- Component: Query by role, label, and visible name across branches.
- Journey/live: Navigate and operate the feature by keyboard and assistive-technology semantics.
- Manual/rendered: Inspect the accessibility tree and browser link behavior.

## L2-UXDS-008: Styling, layout, motion, icons, and media follow system rules

**Traces to:** L1-UXDS-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Feature styles shall remain component-scoped and consume semantic CSS custom properties, with SCSS used for component composition where useful; global CSS shall contain only tokens, reset, typography, and truly global layout or utilities. Component selectors shall use a short product prefix. CSS layout and responsive primitives shall be preferred over viewport-reading TypeScript. Feature code shall avoid `!important`, deep selectors, global tag overrides, and one-off z-index escalation. Class names shall be readable and semantic rather than encode a literal color. Icons shall be accessible SVG or an approved set rather than emoji. Media and other assets shall be optimized, license-checked, never hotlinked as undeclared production brand dependencies, and shall reserve dimensions. Purposeful bounded animation shall provide a reduced-motion alternative.

### Acceptance criteria

1. Given a feature stylesheet, when reviewed, then it contains no unjustified global override, deep selector, `!important`, literal shared design value, or arbitrary z-index escalation.
2. Given a responsive layout, when viewport size changes, then CSS primitives adapt it without JavaScript viewport polling unless a documented behavioral need exists.
3. Given a product icon, component selector, or semantic class, when reviewed, then it uses accessible licensed SVG or an approved icon set, the short product prefix, and meaning-based names rather than emoji or literal colors.
4. Given an image or embedded media item, when the page loads, then it is optimized, license-approved, locally controlled or explicitly decided, and width/height or aspect ratio reserves space to prevent avoidable layout shift.
5. Given animation, when reduced motion is requested, then a non-motion or substantially reduced alternative preserves meaning and operation.

### States and accessibility

Layout, icons, media, and motion shall preserve readable content, keyboard targets, accessible names, contrast, zoom/reflow, and reduced-motion preferences.

### Verification

- Domain/unit: Lint forbidden style patterns and validate asset metadata where automated.
- Integration/contract: Build styles/assets and measure representative layout stability.
- Component: Render narrow, wide, long-content, icon, media, and motion variants.
- Journey/live: Exercise the journey at representative viewports and reduced motion.
- Manual/rendered: Inspect computed styles, layout shifts, icon names, and licenses.

## L2-UXDS-009: Feature states are complete and server truth is reconciled

**Traces to:** L1-UXDS-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every feature shall intentionally handle applicable loading, empty, success, validation, error, conflict, unauthorized, forbidden, destructive-confirmation, retry, reconnect, narrow-screen, keyboard, long-content, and reduced-motion states. Server-returned permissions and allowed actions shall be treated as authoritative hints, and the client shall reload or reconcile after stale-state conflicts. Route guards are navigation experience only and shall never be treated as authorization.

### Acceptance criteria

1. Given a feature's supported state matrix, when each state is triggered, then it renders an intentional message, action, or recovery path rather than a blank, broken, or indefinitely loading surface.
2. Given the server reports an action unavailable, when the UI renders it, then the control and explanatory text reflect that state without claiming client-side enforcement.
3. Given state or authorization changes after rendering, when submission returns a conflict or forbidden response, then the page reconciles authoritative state and preserves useful user context.
4. Given a client route guard allows navigation, when the API rejects access, then the UI handles the server response and does not expose protected data.

### States and accessibility

All listed states shall preserve focus, accessible names, associated validation, deliberate announcements, keyboard recovery, and non-color-only meaning.

### Verification

- Domain/unit: Test state reducers/mappers and reconciliation logic.
- Integration/contract: Verify server allowed-state and failure mappings.
- Component: Render and assert the complete applicable state matrix.
- Journey/live: Exercise empty, stale-conflict, forbidden, destructive, and reconnect paths.
- Manual/rendered: Inspect representative viewport screenshots and state transitions.

## L2-UXDS-010: Shared primitives are semantic and operable

**Traces to:** L1-UXDS-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Product and marketing surfaces shall meet WCAG 2.2 AA. Shared primitives shall provide visible `:focus-visible`, logical headings and landmarks, a skip link where navigation repeats, real labels and accessible names, usable control/touch sizes, keyboard-equivalent operation, and text, iconography, shape, or position in addition to color for state. Native semantics shall be used before ARIA substitutes.

### Acceptance criteria

1. Given keyboard-only use, when focus moves through a page, then order is logical, the indicator is visible, repeated navigation can be skipped where present, and every operation has a keyboard path.
2. Given any control or input, when inspected through the accessibility tree, then it has the correct role, accessible name, state, and a real associated label where applicable.
3. Given a success, warning, error, selection, or destructive state, when color is unavailable, then text or another non-color cue still conveys its meaning.
4. Given contrast is measured for text, controls, focus, and meaningful graphics, when evaluated, then applicable WCAG 2.2 AA thresholds are met.

### States and accessibility

This requirement defines the baseline for default, hover, focus-visible, active, disabled, selected, invalid, destructive, and high-contrast states.

### Verification

- Domain/unit: Validate primitive state logic where pure.
- Integration/contract: Run automated accessibility checks as a minimum gate.
- Component: Test roles, names, labels, keyboard behavior, and state attributes deeply.
- Journey/live: Complete the primary journey using keyboard-only navigation.
- Manual/rendered: Inspect focus, contrast, accessibility tree, and touch/control sizing.

## L2-UXDS-011: Dynamic status and responsive composition remain accessible

**Traces to:** L1-UXDS-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Asynchronous errors and status changes shall be announced deliberately without making routine updates noisy. Page composition shall remain understandable under zoom, text reflow, forced colors, reduced motion, narrow and wide viewports, long localized content, and screen-reader navigation. Shared primitives shall receive deep accessibility testing, while pages shall be tested for accessible names, reading order, focus flow, and composed behavior.

### Acceptance criteria

1. Given an asynchronous validation error, save result, or material connection-status change, when it occurs, then an appropriately scoped live announcement communicates it without announcing every background update.
2. Given 400% zoom or a narrow viewport, when the core journey is used, then content reflows without two-dimensional scrolling except for content that inherently requires it and no action becomes unreachable.
3. Given forced-colors or reduced-motion preferences, when the journey renders, then meaning, focus, selection, and operation remain perceivable.
4. Given long content or translated-length text, when it appears, then controls and layouts adapt without clipping essential information or obscuring focus.

### States and accessibility

Dynamic and responsive variants are first-class accessibility states and shall not be deferred to post-feature cleanup.

### Verification

- Domain/unit: Test announcement-selection and responsive state helpers where pure.
- Integration/contract: Run automated checks across representative routes and preferences.
- Component: Verify live regions, reflow, forced-colors, long-content, and reduced-motion variants.
- Journey/live: Complete the journey at zoom, narrow width, and reduced motion.
- Manual/rendered: Perform screen-reader, reflow, forced-colors, and focus-flow inspection.

## L2-UXDS-014: Localize every user-facing contract

**Traces to:** L1-UXDS-006

**Status:** proposed

**Source:** product brief + research

**Related:** —

All user-facing application, marketing, email, administration, moderation, validation, and system
text shall be extractable and keyed by stable meaning. Locale selection shall use an explicit Account
preference where present, then an approved negotiation and fallback chain. Server failures shall carry
stable machine codes and parameters so clients never parse English prose to decide behavior.

### Acceptance criteria

1. Given a supported locale and complete message catalog, when a journey renders, then navigation, controls, validation, errors, status, Notifications, and generated system text use that locale with correct plural and select rules.
2. Given a translation key is missing or malformed, when rendering occurs, then the declared fallback is used, the person sees no raw key or broken template, and an owned diagnostic is emitted without private parameters.
3. Given the API returns a validation, authorization, conflict, or workflow failure, when the client presents it, then a stable code and typed parameters select localized text without inspecting server prose.
4. Given pseudo-localization is enabled, when every representative surface builds and renders, then untranslated literals, unsafe interpolation, clipping, and assumptions about English word order are detectable by the quality gate.

### States and accessibility

The locale control has an accessible name, identifies the current and fallback locale, preserves focus
when text changes, and does not announce the entire page after a preference update.

### Verification

- Domain/unit: Test locale negotiation, fallback, message parameters, and plural/select rules.
- Integration/contract: Verify stable API codes and localized Notification/email rendering.
- Component: Run literal detection and pseudo-localized component states.
- Journey/live: Change locale and complete public, member, moderation, and Account journeys.
- Manual/rendered: Linguistic and accessibility review of supported locales and fallback behavior.

## L2-UXDS-015: Render locale and time-zone values consistently

**Traces to:** L1-UXDS-006

**Status:** proposed

**Source:** product brief + research

**Related:** —

Numbers, measurements, plural forms, lists, dates, and times shall use the viewer's
supported locale and IANA time-zone preference with a documented fallback. The server shall persist
instants in UTC and preserve a local-time zone or offset only when product meaning, such as an Event
schedule, requires it.

### Acceptance criteria

1. Given an Account locale and time zone, when a shared instant, count, measurement, or duration is rendered, then formatting follows those preferences while the underlying server value and ordering remain unchanged.
2. Given an Event crosses a daylight-saving gap, overlap, or offset change, when created and viewed, then the selected local-time semantics are deterministic and the displayed zone is unambiguous.
3. Given an Account has no valid preference, when locale-sensitive content renders, then the documented negotiation and fallback produce a valid value rather than server-local time or an ambiguous date.
4. Given a form accepts a locale-sensitive value, when submitted, then display parsing cannot change canonical date, time, measurement, or identity semantics and validation explains the accepted format.

### States and accessibility

Dates and countdowns expose complete text alternatives, exact values where decisions depend on them,
and no rapid or noisy announcements; time-zone selection supports search, keyboard, and zoom.

### Verification

- Domain/unit: Test locale formats, UTC conversion, zone fallback, and daylight-saving boundaries.
- Integration/contract: Verify timestamp, measurement, and Event contracts across supported locales/zones.
- Component: Test ambiguous dates, long zone names, validation, and exact-time alternatives.
- Journey/live: Complete Event, Notification, and moderation-deadline journeys across zone changes.
- Manual/rendered: Linguistic, accessibility, and daylight-saving review.

## L2-UXDS-017: Require accessible media alternatives

**Traces to:** L1-UXDS-006

**Status:** proposed

**Source:** product brief + research

**Related:** —

Meaningful images require author-supplied alternative text or an explicit decorative decision;
audio and video require captions, transcripts, and controls appropriate to the accepted media scope.
Automated suggestions may assist but cannot silently replace author responsibility, and Moderation
Actions may restrict an unsafe alternative without exposing hidden media.

### Acceptance criteria

1. Given an author attaches a meaningful image, when publication is requested, then valid alternative text is required under the configured content policy and is stored and authorized with the Attachment.
2. Given an image is deliberately decorative, when rendered, then it is ignored by assistive technology and no meaningful information depends on it.
3. Given supported audio or video is published, when consumed, then synchronized captions where applicable, a transcript, keyboard-operable controls, and non-autoplay defaults meet the declared accessibility policy.
4. Given media visibility, Block, Moderation Action, edit, or deletion state changes, when alternatives are read through direct, Feed, Search, export, or Notification paths, then the same current authorization and lifecycle policy applies.

### States and accessibility

Upload and edit flows expose missing, generating-suggestion, present, invalid, moderated, and unavailable
alternative states with associated errors and accessible media controls.

### Verification

- Domain/unit: Test alternative requirement, decorative state, ownership, and lifecycle rules.
- Integration/contract: Verify media/alternative authorization, edit, moderation, export, and deletion.
- Component: Test alt-text forms, caption/transcript controls, validation, and keyboard playback.
- Journey/live: Publish, consume, moderate, and delete representative accessible media.
- Manual/rendered: Screen-reader, keyboard, caption quality, transcript, zoom, and reduced-motion review.
