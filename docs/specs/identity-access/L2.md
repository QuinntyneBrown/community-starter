# AUTH — Identity and access behavior

## L2-AUTH-001: Register an Account safely

**Traces to:** L1-AUTH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Registration creates at most one pending Account for a normalized sign-in identifier and does not
grant authenticated or Community access before required verification succeeds.

### Acceptance criteria

1. Given a valid unregistered identifier and policy-compliant credentials, when registration is submitted, then one pending Account is created and a time-bounded verification action is issued.
2. Given an identifier already belongs to an Account, when registration is submitted, then the response is indistinguishable from the accepted recovery-safe response and no second Account is created.
3. Given malformed input or credentials that do not meet policy, when registration is submitted, then the server returns field-safe validation details and persists no Account or credential.

### States and accessibility

Show submitting, accepted, validation, throttled, and unavailable states; associate errors with
fields, preserve valid input, move focus to the summary, and announce the generic accepted result.

### Verification

- Domain/unit: normalized-identifier uniqueness and pending-state rules.
- Integration/contract: registration, duplicate, validation, and verification-dispatch contracts.
- Component: form, generic accepted result, error summary, and busy state.
- Journey/live: register through delivery of the verification action.
- Manual/rendered: keyboard, narrow-screen, zoom, and announcement review.

## L2-AUTH-002: Verify Account ownership

**Traces to:** L1-AUTH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

A single-use, time-bounded verification action proves control of an Account identifier before that
identifier is treated as verified; replacement is a versioned security transition, not an in-place
string edit.

### Acceptance criteria

1. Given a pending Account and its valid unused verification action, when verification is confirmed, then the Account identifier becomes verified and the action becomes unusable.
2. Given an expired, invalid, or previously used action, when verification is attempted, then the server returns the same safe failure, reveals no Account state, and leaves verification unchanged.
3. Given a signed-in Account with recent stronger verification requests a replacement sign-in identifier, when a purpose-bound change is created, then the current verified identifier remains authoritative, the normalized replacement is reserved only as a pending candidate under the Account version and uniqueness policy, and cancellation or expiry releases it without changing sign-in.
4. Given the replacement is independently verified and remains unique, when commit rechecks Account lifecycle, current identifier, candidate, security assurance, and version atomically, then the candidate becomes the sole current sign-in identifier, the prior identifier loses sign-in/recovery authority and enters the accepted privacy-preserving reuse quarantine, the Account authentication epoch advances and invalidates every other outstanding recovery, identifier-change, and factor-reset action, other Sessions are revoked, the continuing Session is rotated if allowed, and privacy-safe notices go to both controlled destinations without notification failure rolling back the change.
5. Given verification, cancellation, retry, concurrent registration/change, Account lifecycle, or uniqueness state races replacement, when the transition commits, then one current identifier wins, stale or losing operations change nothing and return a safe conflict/prior result, the old identifier remains authoritative unless full commit succeeded, and post-commit rollback requires a fresh verified replacement or Account recovery rather than reactivating stale proof.

### States and accessibility

Show current identifier, replacement pending, verifying, verified/changed, cancelled, expired/invalid,
uniqueness conflict, resend accepted, and service-error states without exposing Account existence;
focus and announce results and provide a keyboard-operable recovery path.

### Verification

- Domain/unit: expiry, single-use, candidate uniqueness, replacement commit, reuse quarantine, and rollback rules.
- Integration/contract: confirmation, replay, resend, replacement races, Session rotation/revocation, and dual notice.
- Component: current/pending replacement, cancel, result, conflict, resend, and accessible status states.
- Journey/live: register, verify, sign in, and verify an identifier change.
- Manual/rendered: copied-link, narrow-screen, keyboard, and screen-reader review.

## L2-AUTH-003: Recover Account access

**Traces to:** L1-AUTH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Recovery uses a private, single-use, time-bounded proof bound to the Account's current identifier and
authentication epoch, and invalidates access that could remain under a displaced credential holder.

### Acceptance criteria

1. Given any syntactically valid identifier, when recovery is requested, then the server returns the same accepted response and only issues an action for an eligible Account, binding its non-recoverable verifier to purpose, Account, current verified identifier version, authentication epoch, environment, and expiry.
2. Given a valid unused recovery action and acceptable replacement credentials, when recovery atomically rechecks the bound Account lifecycle, current identifier version, authentication epoch, action, and credential policy, then credentials change, the epoch advances, all Sessions and every other outstanding recovery/identifier-change/factor-reset action are revoked, and the used action becomes terminal.
3. Given an action is invalid, expired, replayed, or stale because identifier replacement, prior recovery, factor reset, Account lifecycle, or authentication epoch changed, when recovery is attempted, then no credential, identifier, factor, or Session state changes and the response discloses no Account details.
4. Given identifier replacement, recovery, factor reset, or repeated completion races against the same authentication epoch, when conditional commit runs, then exactly one transition advances the epoch and every loser reloads a generic stale result without reactivating an old identifier, proof, credential, factor, or Session.

### States and accessibility

Show requesting, generic accepted, replacing, success, invalid/expired, throttled, and unavailable
states; errors are programmatically associated and successful revocation is announced.

### Verification

- Domain/unit: identifier/epoch binding, expiry, single use, conditional winner, and revocation requirements.
- Integration/contract: enumeration resistance, recovery versus identifier/factor races, replay, and failure contracts.
- Component: generic request result and accessible replacement form states.
- Journey/live: recover access and prove prior sessions can no longer call a protected endpoint.
- Manual/rendered: password-manager, keyboard, zoom, and announcement review.

## L2-AUTH-004: Enter Account deactivation safely

**Traces to:** L1-AUTH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

`active`, `deactivated`, `deletion-pending`, and `deleted` are the canonical Account lifecycle states.
Deactivation is the one reversible `active → deactivated` transition; deletion is the separate
privacy workflow. Identity enforces access effects, while the shared lifecycle policy resolves every
required ownership and operational responsibility before either transition.

### Acceptance criteria

1. Given a signed-in Account lacks recent stronger verification, when deactivation is requested, then the server requires fresh verification and leaves the Account lifecycle and responsibilities unchanged.
2. Given deactivation would orphan a required Community or Space owner/manager, billing contact, Event organizer, Conversation controller, pending invitation/request, Webhook Endpoint, or another registered responsibility, when confirmed, then it is rejected with exact transfer/cancel/close remediation and no state changes.
3. Given all responsibilities are resolved, when deactivation commits, then lifecycle becomes `deactivated`, ordinary Sessions are revoked, participation and optional Deliveries stop, and retained Profile/content/evidence follows privacy policy without beginning irreversible deletion.
4. Given a deactivated Account proves control within the restoration policy and no independent Action forbids restoration, when reactivation commits, then lifecycle returns to `active` without duplicating the Account or silently restoring banned Memberships; `deletion-pending` and `deleted` cannot use this path.

### States and accessibility

Show impact summary, stronger-verification, blocked-with-remediation, confirmation, grace-period, and
completed states; destructive intent requires explicit text, focus containment, and no color-only cue.

### Verification

- Domain/unit: canonical lifecycle, responsibility preconditions, deactivation, restoration, and Action precedence.
- Integration/contract: every responsibility blocker, revocation, deactivation, restore, and deletion-path separation.
- Component: destructive confirmation, remediation, and completion states.
- Journey/live: deactivate an eligible Account and prove access revocation and permitted restoration.
- Manual/rendered: keyboard, screen-reader, reduced-motion, and destructive-copy review.

## L2-AUTH-005: Sign in without enumeration

**Traces to:** L1-AUTH-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Sign-in establishes an ordinary Session only after valid proof of an active eligible Account.
Deactivated, deletion-pending, eligibility-restricted, or Moderation-Action state may instead use the
separate restricted-remediation issuance matrix; observable failures do not distinguish Accounts.

### Acceptance criteria

1. Given a verified eligible Account and valid proof, when sign-in succeeds, then a new bounded session is established and only an approved local destination may be used for continuation.
2. Given the identifier is unknown, the proof is wrong, or the Account is ineligible for an ordinary Session, when ordinary sign-in is attempted, then the same generic failure is returned and no ordinary Session is created.
3. Given a continuation target is external or malformed, when sign-in succeeds, then the server ignores it and uses the safe default destination.
4. Given valid proof belongs to an Account whose current lifecycle, eligibility, or Moderation state permits restricted remediation, when the separate remediation flow succeeds, then only a typed restricted-remediation Session is issued under its current issuance matrix and no ordinary participation route becomes available.

### States and accessibility

Show idle, submitting, generic failure, stronger-verification required, throttled, and unavailable
states; preserve the identifier when safe, never expose the secret, and announce failures once.

### Verification

- Domain/unit: eligibility and continuation-target rules.
- Integration/contract: success, generic failure, cookie/token properties, and redirect validation.
- Component: sign-in, busy, error, and stronger-verification states.
- Journey/live: sign in and navigate to both valid and rejected continuation targets.
- Manual/rendered: password-manager, keyboard, narrow-screen, and error-announcement review.

## L2-AUTH-006: Require stronger verification

**Traces to:** L1-AUTH-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

The server requires an enrolled stronger proof for configured high-risk sign-ins and sensitive
Account or Community actions, with bounded elevation and recoverable enrollment.

### Acceptance criteria

1. Given an Account is required to enroll a stronger factor, when enrollment begins, then activation requires proof of that factor under the current Account authentication epoch and recovery material is shown only through the protected enrollment flow.
2. Given a challenge bound to the current Account, authentication epoch, active factor identity/version, Session, purpose, and expiry, when verification atomically rechecks and consumes it, then the Session receives only the bounded assurance needed for the pending action.
3. Given a factor/recovery proof is invalid, replayed, expired, or stale because the factor, identifier, authentication epoch, Session, or Account lifecycle changed, when verification is attempted, then assurance is not elevated, protected state is unchanged except for abuse controls, and a safe failure is returned.
4. Given a sensitive action is attempted after assurance expires, when authorization is evaluated, then the server requires verification again before changing state.

### States and accessibility

Show enrollment, challenge, alternate recovery, invalid, locked/throttled, success, and expiry states;
codes are selectable and labeled, time limits are not color-only, and focus returns to the action.

### Verification

- Domain/unit: enrollment, factor/epoch challenge binding, expiry, replay, and bounded-assurance rules.
- Integration/contract: challenge versus factor remove/replace/reset and identifier/recovery races, enrollment, and step-up.
- Component: enrollment, code entry, alternate method, and expiry states.
- Journey/live: enroll, complete a protected action, expire assurance, and challenge again.
- Manual/rendered: authenticator/password-manager, keyboard, zoom, and screen-reader review.

## L2-AUTH-007: Rotate and revoke sessions

**Traces to:** L1-AUTH-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Sessions have bounded lifetimes, rotate renewal proof, support Account-visible revocation, and treat
reuse as possible compromise.

### Acceptance criteria

1. Given an eligible active session with valid renewal proof, when renewal occurs, then a bounded replacement is issued and the prior renewal proof can no longer renew.
2. Given previously rotated renewal proof is reused, when renewal is attempted, then the affected session family is revoked and no replacement session is issued.
3. Given an Account signs out one session or revokes another listed session, when revocation commits, then subsequent protected requests from that session are unauthorized.
4. Given the Account is deactivated, deletion-pending, deleted, has credentials recovered, or has a Moderation Action whose effect matrix revokes or restricts that Session class, when a protected request is made, then affected ordinary Sessions are rejected even before local expiry and only a separately issued restricted-remediation Session may invoke explicitly allowed actions.
5. Given a sign-in identifier replacement commits, when Session consequences apply, then every other Session family is revoked, any permitted continuing recently verified Session rotates to a new family, and cached claims or credentials containing the prior identifier cannot authenticate, renew, or recover the Account.

### States and accessibility

Show current and other sessions, renewal-in-progress, expired, revoked, confirmation, and reconnect
states; device descriptions are textual, current-session status is announced, and actions are labeled.

### Verification

- Domain/unit: lifetime, rotation, identifier replacement, reuse, and revocation-family rules.
- Integration/contract: renew, replay, single/all-session revoke, identifier replacement, and protected-request contracts.
- Component: session list, current marker, confirmation, and expired-session handling.
- Journey/live: renew, revoke another session, and prove replay and closure denial.
- Manual/rendered: keyboard, narrow list, time-zone text, and announcement review.

## L2-AUTH-008: Evaluate current Permissions

**Traces to:** L1-AUTH-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Each protected Community action is authorized on the server from the current Account lifecycle,
Moderation Actions, active Membership, assigned Role, effective Permissions, and target state at
request time.

### Acceptance criteria

1. Given an active Account without a prohibiting Action and an active Membership whose Role grants the required Permission, when an eligible action is requested in that Community, then the server may perform it and records the acting Account.
2. Given no active Membership or no effective Permission, when the action is requested, then the server returns the safe forbidden or unavailable response and changes no target state.
3. Given a Role, Permission, Membership, or target state changes after a page is rendered, when the action reaches the server, then current state determines the result and stale client claims confer no access.

### States and accessibility

Show available, unavailable-with-reason when safe, pending, forbidden, and conflict states; disabled
controls remain understandable, and server failures are announced without leaking restricted details.

### Verification

- Domain/unit: effective-Permission and target-eligibility rules.
- Integration/contract: allowed, forbidden, absent-Membership, and stale-state cases.
- Component: allowed-action rendering and forbidden/conflict reconciliation.
- Journey/live: change a Role during an active session and verify immediate enforcement.
- Manual/rendered: keyboard access, disabled explanations, and privacy-safe copy review.

## L2-AUTH-009: Enforce Community boundaries

**Traces to:** L1-AUTH-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every Community-owned read and mutation is scoped by the server to the target Community and current
Membership; a supplied Community identifier is never proof of access.

### Acceptance criteria

1. Given an Account belongs to Community A but not Community B, when it requests a Community B resource identifier through any endpoint, then no resource data is returned and no state changes.
2. Given a route identifier and body Community identifier disagree, when the request is evaluated, then the authoritative route/resource scope is used or the request is rejected, and no cross-Community write occurs.
3. Given a batch, Search result, Feed, export, or realtime subscription spans stored records, when results are produced, then every item is independently constrained to Communities the Account may access.
4. Given an opaque identifier from another Community is guessed, when it is requested, then response shape and timing policy do not confirm its existence beyond the authorized disclosure policy.

### States and accessibility

Use a consistent unavailable experience for hidden resources, an explicit forbidden experience only
when safe, and accessible recovery navigation without naming another Community's private content.

### Verification

- Domain/unit: scope-composition and route/body mismatch rules.
- Integration/contract: cross-Community reads, writes, batches, exports, and subscriptions.
- Component: unavailable, forbidden, and safe-navigation states.
- Journey/live: use two Accounts and two Communities to attempt identifier and realtime crossover.
- Manual/rendered: privacy wording, keyboard recovery, and announcement review.

## L2-AUTH-010: Expose allowed actions safely

**Traces to:** L1-AUTH-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Responses may expose current allowed actions and safe denial reasons so clients can explain state,
but those fields are advisory snapshots and never replace server authorization.

### Acceptance criteria

1. Given an authorized resource response, when allowed actions are included, then they reflect the same current Membership, Role, Permission, and state policy used by its mutation endpoints.
2. Given the client submits an allowed-action field or enables a hidden action locally, when a mutation is requested, then the server ignores the claim and independently authorizes the action.
3. Given authorization changes between read and mutation, when the mutation is rejected, then no state changes and the response provides a stable safe code that allows the client to reconcile.

### States and accessibility

Actions have semantic labels and safe explanations for unavailable states; conflict reconciliation
preserves focus where possible and announces changed access without exposing hidden policy details.

### Verification

- Domain/unit: common policy result for action projection and mutation enforcement.
- Integration/contract: allowed-action parity, forged client claim, and stale authorization cases.
- Component: enabled, explained-unavailable, and reconciliation states.
- Journey/live: render an action, revoke Permission, attempt it, and refresh allowed state.
- Manual/rendered: keyboard, screen-reader description, and non-color-state review.

The platform limits authentication, verification, recovery, invitation, and other security-sensitive

2. Given attempts target known and unknown identifiers, when limits or responses apply, then externally observable behavior does not reveal which identifier belongs to an Account.
3. Given a legitimate Account reaches a temporary limit, when the retry window expires or an approved recovery path succeeds, then access can resume without administrative data repair.
4. Given distributed attempts target one Account or many Accounts, when abuse controls evaluate them, then Account, origin, and broader risk signals may be combined without trusting client-supplied identity.

limit once, avoid countdown-only access, and never require pointer precision to recover.

- Component: throttled, retry, and recovery states.

## L2-AUTH-013: Link an external identity

**Traces to:** L1-AUTH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

When external identity is enabled, an Account may link or unlink a provider identity identified only
by its verified issuer and subject through a deliberate, recoverable, server-controlled flow.

### Acceptance criteria

1. Given a configured provider returns valid state/nonce and a cryptographically verified issuer, subject, audience, and sign-in proof, when deliberate sign-up is confirmed, then at most one new Account is associated with that issuer-subject pair; a matching email alone never links an existing Account.
2. Given a signed-in Account completes recent authentication and proves an unlinked provider identity, when linking commits, then the issuer-subject pair is associated atomically; an email claim, client-supplied subject, or unverified provider response is rejected and existing Account links are preserved.
3. Given the same issuer-subject pair is linked or concurrently being linked to another Account, when another link or sign-in is attempted, then uniqueness allows at most one success and all external failures remain enumeration-safe without disclosing the associated Account.
4. Given an Account requests unlinking, when no other verified sign-in or recovery method would remain, then the server rejects unlinking and preserves the link; otherwise recent authentication is required, affected session assurance is renewed or revoked according to policy, and the unlink commits.
5. Given link, unlink, or provider sign-in succeeds or materially fails, when processing completes, then privacy-safe security audit and configured Account notification occur; only enabled provider adapters behind the Infrastructure boundary are used, and invalid production issuer, client, secret, redirect, or endpoint configuration prevents readiness rather than enabling an unsafe fallback.

### States and accessibility

Show provider available/unavailable, redirecting, verifying, deliberate sign-up/link confirmation,
already linked, generic conflict, unlink impact, recovery-required, success, and safe failure states;
provider buttons have accessible names and focus returns predictably from redirects.

### Verification

- Domain/unit: issuer-subject identity, no-email-link, uniqueness, recovery-method, and unlink rules.
- Integration/contract: provider proof/state/nonce validation, deliberate sign-up, linking races, enumeration safety, session consequences, audit/notification, and unlink.
- Component: provider list, redirect return, link/unlink confirmations, conflict, and recovery states.
- Journey/live: sign up, link, sign in, race duplicate linking, and unlink with/without recovery.
- Manual/rendered: keyboard, screen-reader provider names, redirect focus, privacy copy, and reflow.

## L2-AUTH-014: Protect local credentials and action secrets

**Traces to:** L1-AUTH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

When local password credentials are enabled, each password shall be stored only as a uniquely salted,
adaptive one-way hash using a reviewed memory-hard or identity-platform algorithm with versioned cost
parameters. Verification, recovery, identifier-change, invitation, export, and other bearer action
secrets shall be generated with cryptographic entropy and stored only as purpose-bound non-recoverable
verifiers; database disclosure alone must not reveal a usable password or bearer action.

### Acceptance criteria

1. Given two Accounts choose the same password, when credentials are persisted, then their stored verifiers differ through unique salts and neither the password nor a reversibly encrypted equivalent is stored or logged.
2. Given a valid password is checked against an older accepted cost or algorithm, when authentication succeeds, then the verifier is upgraded under the current policy without exposing or retaining the plaintext.
3. Given a verification, recovery, identifier-change, factor-reset, invitation, or export bearer secret is issued, when persistence is inspected, then only a purpose-, subject-, environment-, expiry-, and applicable current identifier/authentication-epoch-bound hash or keyed verifier exists and single use is enforced atomically.
4. Given the credential/token database is copied without separately protected server secrets, when an attacker attempts direct replay, then stored action verifiers cannot be presented as bearer values and credential cracking remains bounded by the accepted adaptive cost policy.
5. Given hashing, entropy, pepper/key, or cost configuration is missing or below policy, when production starts or credentials are written, then readiness or the operation fails safely without printing secret material.

### States and accessibility

Credential UI supports password managers and paste, explains requirements before failure, provides
generic invalid/expired states, and never reveals stored-verifier details or blocks assistive input.

### Verification

- Domain/unit: Test algorithm/version selection, salts, rehash, token purpose/expiry, and single-use rules.
- Integration/contract: Inspect persisted forms, race token use, rotate protected keys, and verify redaction.
- Component: Test password-manager, validation, expired action, and recovery states.
- Journey/live: Register, rehash on sign-in, recover, replay, and inspect all telemetry/storage artifacts.
- Manual/rendered: Independent cryptography/identity, accessibility, and operational-cost review.

A restricted-remediation Session is a short-lived, non-participatory authentication class issued

1. Given a deactivated, deletion-pending, eligibility-restricted, or Action-restricted Account presents valid enumeration-safe proof and required stronger verification, when remediation authentication succeeds, then a separately typed Session records exact allowed actions, state/matrix versions, audience, issue/expiry, and assurance without gaining ordinary route, API, hub, Feed, Search, Community, content, Message, Event, or integration access.
2. 3. 4. Given eligibility/guardian or Moderation state restricts participation, when the matrix is evaluated, then non-delegable privacy/export/deletion, eligible Appeal, guardian remediation, security, billing cancellation/history, and approved support actions remain available only where the corresponding current policy permits them.
5. Given lifecycle, eligibility, credential, assurance, or Action state changes after issuance, when any request, renewal, or queued effect uses the Session, then exact scope is re-evaluated, disallowed work is rejected, the Session cannot expand or convert to ordinary, and issuance/use/revocation is audited and rate bounded.

## L2-AUTH-016: Govern stronger-factor lifecycle

**Traces to:** L1-AUTH-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Stronger factors and recovery material have explicit enroll, list, add, replace, remove, reset,
consume, rotate, compromise, and audit behavior without exposing reusable secrets.

### Acceptance criteria

1. Given a recently stronger-verified Account lists or adds a factor, when the request succeeds, then only safe factor type/label/created/last-used metadata is listed and a new factor becomes active only after independent proof, unique binding, an authentication-epoch advance that invalidates outstanding factor challenges/actions, required Session rotation/revocation, and an Audit Event.
2. Given an Account removes or replaces a factor, when current proof, factor/Account version, and risk policy pass, then the change conditionally advances the authentication epoch, invalidates outstanding challenges and recovery/identifier-change/factor-reset actions, and rotates/revokes affected Sessions only if the accepted minimum viable factor/recovery invariant remains; removal of a last required method is blocked with safe reset or support remediation.
3. Given recovery codes or equivalent material are generated or rotated, when presented, then plaintext is shown once, stored only as purpose-bound one-way verifiers, each code is consumed atomically once, and regeneration immediately invalidates every unused prior code.
4. Given a device/factor is lost or compromise is suspected, when the reviewed reset flow re-establishes Account control against its bound current identifier and authentication epoch, then conditional commit advances that epoch, affected factors, recovery material, every outstanding recovery/identifier-change/factor-reset action, assurance, and Session families are revoked or rotated, independent unaffected identity proofs are required, and privacy-safe security notice and Audit Events are created.
5. Given primary credential recovery occurs for an MFA-enrolled Account, when access is restored, then MFA is not silently disabled; current factors remain required or a distinct high-assurance reset with delay/risk controls completes before sensitive actions become available.

### States and accessibility

Show active factor list, adding, verifying, recovery-codes-once, replacing, last-method blocked, lost,
reset pending, compromised, revoked, and completed states with non-secret labels and alternatives.

### Verification

- Domain/unit: Factor uniqueness/version, authentication epoch, minimum method, recovery rotation, reset, and precedence.
- Integration/contract: Add/remove/replace versus live challenges/actions, race last factor, replay codes, recover, and revoke Sessions.
- Component: Factor list/enrollment, code save acknowledgement, remove, lost/reset, and blocked states.
- Journey/live: Enroll two factors, rotate codes, lose one, recover credentials, and verify no bypass.
- Manual/rendered: Authenticator/password-manager, keyboard, screen reader, copy safety, and recovery review.
