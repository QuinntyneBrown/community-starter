# INTG — APIs and integrations

## L2-INTG-001: Publish accurate API and error contracts

**Traces to:** L1-INTG-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

All supported `/api` endpoints use explicit request and response DTOs, plural resource nouns, stable
identifiers, and dedicated action or subresource routes for unambiguous transitions. OpenAPI describes
authentication, validation, success, pagination, concurrency, and sanitized Problem Details shapes.

### Acceptance criteria

1. Given a supported endpoint, when its runtime behavior is compared with OpenAPI, then request, response, status, authentication, and representative error contracts agree.
2. Given a route identifier conflicts with a body identifier, when the request is handled, then the route is authoritative or the request is rejected without mutating state.
3. Given an API failure occurs, when the client receives Problem Details, then it has a stable machine code and correlation identifier without persistence entities, stack traces, secrets, or private details.
4. Given a persistence model changes, when contract checks run, then no navigation graph or undeclared field appears at the HTTP boundary.

### States and accessibility

Human-readable API documentation exposes authenticated, forbidden, validation, conflict, unavailable,
and rate-limited examples with navigable headings, tables, and code labels.

### Verification

- Domain/unit: Test stable problem-code mapping and route/body identity policy.
- Integration/contract: Compare generated OpenAPI with live success and failure responses.
- Component: Verify the Angular client renders typed failures without parsing prose.
- Journey/live: Exercise one create, query, transition, conflict, and forbidden response end to end.
- Manual/rendered: Review API documentation completeness and sensitive-data absence.

## L2-INTG-002: Preserve compatible API evolution

**Traces to:** L1-INTG-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The platform records which contracts are internal and which are compatibility commitments. Additive
change is preferred; a breaking external change requires an accepted decision, migration window,
consumer notice, parallel support or explicit version boundary, telemetry, and removal criteria.

### Acceptance criteria

1. Given a contract change is additive and optional, when old and new clients call it, then both remain valid under the declared compatibility policy.
2. Given a change removes, renames, or changes meaning of a committed field or status, when contract validation runs without an approved migration record, then the release fails.
3. Given a deprecated contract reaches removal criteria, when it is removed, then usage evidence, consumer communication, replacement, and rollback or forward-fix path are recorded.

### States and accessibility

Deprecation documentation names availability, replacement, dates, and errors in text rather than
color, and generated documentation remains keyboard navigable.

### Verification

- Domain/unit: Test compatibility classification rules where automated.
- Integration/contract: Run prior-consumer fixtures against the new artifact.
- Component: Compile and test the generated first-party client against the current schema.
- Journey/live: Exercise old and replacement paths during an example compatibility window.
- Manual/rendered: Review notices, dates, and migration guidance.

## L2-INTG-003: Bound and paginate collection contracts

**Traces to:** L1-INTG-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every collection endpoint defines a maximum page size, stable sort with deterministic tie-breaker,
opaque cursor semantics, filter allow-list, cancellation, and current-viewer authorization. Counts
and cursors cannot reveal inaccessible resources.

### Acceptance criteria

1. Given more permitted records exist than one page, when a consumer follows cursors, then each eligible item appears once in deterministic order without duplicates or omissions caused by equal sort values.
2. Given a requested page size exceeds the bound or a filter is invalid, when the server validates it, then it applies the documented bound or returns a stable validation problem without unbounded work.
3. Given records outside the caller's Community or visibility scope exist, when the collection is requested, then items, counts, facets, cursors, and timing do not disclose them.
4. Given the client cancels a request, when cancellation propagates, then database and external work stop where supported.

### States and accessibility

First-party collection UI offers loading, empty, partial, retry, and end-of-results states and an
accessible alternative to infinite scrolling with preserved reading position.

### Verification

- Domain/unit: Test cursor encoding, stable comparison, bounds, and filters.
- Integration/contract: Test concurrent inserts, equal sort keys, scope filtering, and cancellation.
- Component: Test accessible load-more and retry behavior.
- Journey/live: Traverse multiple pages while records are added and permissions change.
- Manual/rendered: Inspect focus, announcements, and API examples.

## L2-INTG-004: Make retryable mutations idempotent

**Traces to:** L1-INTG-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Mutations that clients or providers may safely retry accept a bounded idempotency key scoped to the
authenticated actor, operation, and environment. The server returns the committed result for an
equivalent duplicate and rejects key reuse with a different request.

### Acceptance criteria

1. Given two equivalent requests use the same valid key, when they arrive sequentially or concurrently, then one product transition and one set of post-commit effects occur and both receive the committed outcome.
2. Given the same key is reused with a materially different request, when validation runs, then the server returns a stable conflict and no second transition occurs.
3. Given an unauthenticated actor or different Community repeats another actor's key, when the request is handled, then the stored outcome is not disclosed or reused.
4. Given an idempotency record expires, when cleanup occurs, then retention and replay guarantees match the documented client retry window.

### States and accessibility

First-party clients distinguish processing, committed, retrying, and conflict outcomes without
duplicating controls or announcements.

### Verification

- Domain/unit: Test equivalence fingerprint, scope, conflict, and expiry rules.
- Integration/contract: Race duplicate requests and inspect state, events, and response replay.
- Component: Test double-submit prevention and conflict reconciliation.
- Journey/live: Simulate a lost response followed by retry through the full stack.
- Manual/rendered: Review retry guidance and user-visible status clarity.

## L2-INTG-005: Issue scoped integration credentials

**Traces to:** L1-INTG-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Only authorized Community roles can create a Service Identity or credential. Creation requires a
name, owner, Community, explicit Permissions, environment, expiry or review date, and recent
authentication; secret material is shown once and stored only in protected form.

### Acceptance criteria

1. Given an authorized administrator requests a credential within their grantable Permissions, when creation succeeds, then the secret is displayed once and its protected fingerprint, owner, scope, and expiry are audited.
2. Given an actor requests a Permission they do not hold or cannot delegate, when authorization runs, then creation is rejected and no credential is stored.
3. Given a credential is used outside its Community, environment, action, or lifetime, when the request is authenticated, then access fails without revealing the protected resource.
4. Given a credential has no accountable owner or passes its review date, when policy runs, then it is disabled or escalated according to the accepted lifecycle rule.

### States and accessibility

Creation separates one-time secret display from metadata, provides copy alternatives, warns before
leaving, and supports keyboard, screen readers, zoom, and safe error recovery.

### Verification

- Domain/unit: Test grantable scope, expiry, review, and secret-display policy.
- Integration/contract: Verify protected storage, authentication, Community scope, and Audit Events.
- Component: Test create, one-time reveal, copy, expiry, and forbidden states.
- Journey/live: Create, use, expire, and cross-scope probe a credential.
- Manual/rendered: Security review secret handling and permission comprehension.

## L2-INTG-006: Enforce quotas and attribution

**Traces to:** L1-INTG-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Integration requests have actor-, Community-, credential-, operation-, and cost-aware bounds. Every
accepted mutation and security-relevant read is attributable to its Service Identity and human owner;
limits cannot be bypassed by changing untrusted forwarding headers or credentials within one quota.

### Acceptance criteria

1. Given usage remains within an allowed quota, when an integration calls an endpoint, then the response exposes applicable remaining or reset information without leaking other Communities' use.
2. Given a limit is exceeded, when another call arrives, then bounded work is rejected with a stable rate-limit response and no mutation occurs.
3. Given trusted proxy configuration differs from a supplied client header, when limits calculate identity, then only the trusted network chain influences network-based controls.
4. Given an operator traces an integration mutation, when its Audit Event is inspected, then credential, Service Identity, owning Account, Community, operation, and correlation are attributable.

### States and accessibility

Usage views expose normal, approaching-limit, limited, and unavailable states in text and support
accessible tables and time representations.

### Verification

- Domain/unit: Test quota keys, costs, windows, and trusted-identity policy.
- Integration/contract: Exercise distributed/concurrent limits and audit attribution.
- Component: Test usage and rate-limit recovery states.
- Journey/live: Exhaust a representative quota and verify reset behavior.
- Manual/rendered: Review documentation, operator visibility, and privacy of dimensions.

## L2-INTG-007: Revoke and rotate integration access

**Traces to:** L1-INTG-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Credential owners can list last-used metadata, rotate with an explicit overlap window, and revoke
immediately. Ownership loss, Community retirement, an Account suspension Moderation Action, or security response can revoke
all affected access without editing production data manually.

### Acceptance criteria

1. Given an owner revokes a credential, when the transition commits, then subsequent authentication fails and active Jobs or subscriptions reauthorize before continuing.
2. Given rotation starts with an approved overlap, when the old key reaches its deadline, then it stops authenticating while the new key remains valid.
3. Given an Account or Community loses eligibility or a Community enters retirement, when lifecycle handling runs, then owned integration credentials and Webhook Endpoints are disabled idempotently and in-flight or queued work must reauthorize before any external effect.
4. Given a revoke request is repeated, when handled, then the terminal result is stable and no unrelated credential changes.

### States and accessibility

The UI exposes active, expiring, rotating, expired, revoked, never-used, and unknown-last-use states;
revocation uses accessible destructive confirmation.

### Verification

- Domain/unit: Test rotation overlap, revocation, and owner-lifecycle rules.
- Integration/contract: Verify immediate authentication and queued-work effects.
- Component: Test list, rotate, reveal-new, revoke, and failure states.
- Journey/live: Rotate under traffic and revoke across active and queued requests.
- Manual/rendered: Review timestamps, warnings, and keyboard behavior.

## L2-INTG-008: Register safe Webhook Endpoints

**Traces to:** L1-INTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

An authorized Community administrator registers an HTTPS Webhook Endpoint with a verified event
allow-list and destination policy. Network validation blocks local, private, metadata, credentialed,
redirected, and otherwise prohibited destinations before verification or delivery.

### Acceptance criteria

1. Given a public allowed HTTPS destination, when registration and challenge succeed, then the Endpoint becomes active with a one-time signing secret and audited owner.
2. Given a destination resolves or redirects to a prohibited address, embeds credentials, exceeds redirect bounds, or fails TLS policy, when validation runs, then registration remains inactive and no protected request is made.
3. Given an administrator selects an event outside their Community or Entitlements, when saving, then the server rejects the selection and preserves the prior subscription.
4. Given DNS changes after registration, when a Delivery begins, then destination policy is re-evaluated before connection.
5. Given the accepted possession-proof lifetime expires or a DNS, certificate, redirect, domain-registration, ownership, or risk signal changes, when the next Delivery or scheduled revalidation occurs, then the Endpoint pauses before protected data is sent and must pass a fresh bounded challenge; proof with no accepted expiry or failed revalidation cannot remain active.

### States and accessibility

Registration exposes unverified, active, paused, failing, and disabled states with one-time secret
handling, safe diagnostics, and accessible validation and confirmation.

### Verification

- Domain/unit: Test event allow-list, proof expiry, revalidation, and endpoint lifecycle.
- Integration/contract: Test possession challenge, DNS/certificate changes, redirect, TLS, IP-range, timeout, and response-size defenses.
- Component: Test registration, challenge, secret reveal, pause, and validation states.
- Journey/live: Register an allowed test receiver and attempt representative prohibited targets.
- Manual/rendered: Security review egress policy and administrator guidance.

## L2-INTG-009: Deliver signed webhook events reliably

**Traces to:** L1-INTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Only committed product events create webhook Deliveries. Each bounded payload has a stable event ID,
event type and version, Community scope, occurrence time, and signature covering timestamp and raw
body. Delivery uses timeouts, backoff, attempt history, and a terminal failure state.

### Acceptance criteria

1. Given a product transaction rolls back, when post-commit processing runs, then no webhook Delivery exists for the rejected change.
2. Given a committed subscribed event, when delivery occurs, then the body and timestamp verify against the Endpoint secret and exclude fields outside the event contract.
3. Given the receiver times out or returns a retryable response, when retry policy runs, then attempts use bounded backoff with the same event ID and do not block the originating request.
4. Given retry bounds are exhausted or a permanent response occurs, when processing completes, then the Delivery enters a visible terminal state and owned alerting applies.
5. Given a Delivery attempt begins after event creation, when authorized fields are assembled, then current Endpoint ownership, Community lifecycle, Membership, Block, resource visibility, Moderation Action, deletion, and Entitlement state are rechecked; an ineligible event is suppressed or reduced to its explicitly subscribed privacy-safe lifecycle contract before any connection.

### States and accessibility

Delivery history exposes queued, delivering, succeeded, retrying, failed, and redacted-payload states
using accessible tables, timestamps, and status text.

### Verification

- Domain/unit: Test event selection, signature, retry classification, and terminal state.
- Integration/contract: Verify outbox/commit ordering, HTTP timeouts, signatures, and payload schemas.
- Component: Test history, filtering, and error-detail states.
- Journey/live: Trigger success, timeout, retry, and permanent failure against a controlled receiver.
- Manual/rendered: Review payload minimization and troubleshooting documentation.

## L2-INTG-010: Replay failed webhook Deliveries safely

**Traces to:** L1-INTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

An authorized administrator may replay an eligible Delivery without changing its original event
identity or product state. Replay uses the current Endpoint status and secret, creates a distinct
attempt record, is rate bounded, and never reveals a payload the administrator may no longer view.

### Acceptance criteria

1. Given an eligible failed Delivery and active Endpoint, when replay is confirmed, then a new attempt uses the same event ID and current signing configuration without re-running the product mutation.
2. Given the Endpoint or administrator is no longer authorized, or current Community, Membership, Block, visibility, Moderation Action, deletion, or Entitlement state no longer permits the event contract, when replay is requested, then it is rejected or limited to an allowed lifecycle tombstone and no forbidden payload is reconstructed or sent.
3. Given replay is requested repeatedly or concurrently, when bounds apply, then attempts remain observable and cannot create an unbounded queue.
4. Given the event payload is no longer retainable or viewable, when history is inspected, then metadata remains but protected content is not reconstructed or disclosed.

### States and accessibility

Replay requires accessible confirmation and exposes queued, limited, forbidden, expired-payload, and
completed outcomes with non-color status.

### Verification

- Domain/unit: Test replay eligibility, bounds, and event-identity preservation.
- Integration/contract: Verify replay does not mutate domain state and reauthorizes endpoint access.
- Component: Test confirmation and lifecycle feedback.
- Journey/live: Replay a failed Delivery after key rotation and after authorization removal.
- Manual/rendered: Review operator audit trail and payload disclosure.

## L2-INTG-011: Authenticate and deduplicate provider callbacks

**Traces to:** L1-INTG-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Inbound provider callbacks use raw-body signature verification, timestamp and replay bounds, provider
account/environment matching, size and content-type limits, and a durable provider-event identity.
Accepted input dispatches ordinary Application use cases rather than writing persistence directly.

### Acceptance criteria

1. Given a correctly signed current callback for the configured provider account and environment, when handled, then it is acknowledged only after durable deduplication and processing state are established.
2. Given a signature, timestamp, account, environment, content type, or size is invalid, when the callback arrives, then no product state changes and the response does not disclose verification secrets.
3. Given the same provider-event ID arrives concurrently or out of order, when processing runs, then idempotent lifecycle rules produce one valid result and no duplicate financial or user-visible effect.
4. Given processing fails after acknowledgement, when the Job retries, then the tracked event resumes or reaches a diagnosable terminal state.

### States and accessibility

Operator views expose received, duplicate, processing, succeeded, retrying, invalid, and terminal states
without displaying secrets or unrestricted raw bodies.

### Verification

- Domain/unit: Test ordering, deduplication, transition, and replay-window policies.
- Integration/contract: Exercise provider fixtures over raw-body middleware and real persistence.
- Component: Test safe event history and failure diagnostics.
- Journey/live: Send valid, duplicate, concurrent, stale, and tampered callbacks.
- Manual/rendered: Security review signature handling, redaction, and provider runbook.

## L2-INTG-012: Preview and validate bulk imports

**Traces to:** L1-INTG-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Bulk import accepts only documented, bounded formats into a quarantine area, scans and parses without
active content, shows an authorization-aware preview, and requires confirmation before idempotent
Application use cases commit valid rows. Partial-import semantics are explicit.

### Acceptance criteria

1. Given an unsupported, oversized, malicious, or structurally invalid file, when uploaded, then it remains quarantined or is rejected and no product records are created.
2. Given a valid file includes rows outside the actor's grantable Community scope, when preview runs, then those rows are reported safely and cannot be confirmed.
3. Given an authorized actor confirms a valid preview, when import processes, then row identities make retry idempotent and results distinguish committed, skipped, and failed rows.
4. Given the source changes between preview and confirmation, when its integrity token differs, then confirmation is rejected and no stale preview is committed.

### States and accessibility

The flow supports upload progress, scanning, invalid, preview, confirming, partial, completed, and
failed states; row errors are associated, downloadable accessibly, and not conveyed by color alone.

### Verification

- Domain/unit: Test format, row identity, authorization, and partial-result policy.
- Integration/contract: Verify quarantine, scanning, parsing, Application dispatch, and cleanup.
- Component: Test large error sets, progress, confirmation, and retry states.
- Journey/live: Import mixed valid/invalid data, repeat it, and attempt cross-Community rows.
- Manual/rendered: Review documentation, screen-reader navigation, and destructive consequences.

## L2-INTG-013: Isolate external-provider failure

**Traces to:** L1-INTG-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

External clients use typed configuration, bounded concurrency, connection and overall timeouts,
cancellation, retry only for safe transient operations, redacted diagnostics, and circuit or
degradation policy appropriate to the capability. Provider failure cannot corrupt committed state.

### Acceptance criteria

1. Given a provider is slow or unavailable, when a synchronous request reaches its bound, then work is cancelled or detached according to contract and the caller receives a stable degraded outcome.
2. Given an operation is not safe to retry, when a transient failure occurs, then it is not repeated without an idempotency guarantee.
3. Given a provider response is oversized, malformed, or contains unsafe content, when parsed, then it is rejected without unbounded resource use or private diagnostic leakage.
4. Given a provider recovers, when health and circuit policy permit traffic, then pending work resumes within bounded concurrency and reconciles authoritative state.

### States and accessibility

User surfaces distinguish unavailable, delayed, retryable, and permanently failed outcomes, preserve
input where safe, and expose keyboard-accessible retry without repeated noisy announcements.

### Verification

- Domain/unit: Test failure classification, retry safety, and degradation policy.
- Integration/contract: Inject timeout, cancellation, malformed, oversized, and recovery behavior.
- Component: Test degraded and recovery states.
- Journey/live: Interrupt a representative provider across request and background paths.
- Manual/rendered: Review timeout budgets, operator ownership, and user messaging.

## L2-INTG-014: Propagate lifecycle changes to integrations

**Traces to:** L1-INTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Deletion, visibility loss, and Community retirement have explicit outbound integration contracts;
the platform communicates what it controls without claiming it erased a subscriber's remote copy.

### Acceptance criteria

1. Given an externally projected resource is deleted, anonymized, hidden, or made ineligible after a prior successful Delivery, when lifecycle processing commits, then a durable outbox records at most one versioned tombstone for each immutable receiving-target version that accepted the earlier event, binding its historical Endpoint, destination, subscription contract, possession proof, and signing-key contract; it contains stable external identity, action, effective time, and no fields now forbidden by policy and is never redirected to a newer target that did not receive the resource.
2. Given an Endpoint accepts such events, when registration or event-version adoption occurs, then its subscriber contract states required idempotent removal or minimization behavior, retention expectations, retry window, and the fact that remote compliance and erasure remain the subscriber's responsibility.
3. Given a historical receiving-target version is no longer possession-verified or safe, or its tombstone cannot be delivered before retry retention ends, when the Delivery becomes terminal, then an owned privacy-safe exception records affected event, Endpoint, and target-version identities, last attempt, remediation status, and policy outcome without sending to another destination or representing remote deletion as complete.
4. Given a Community enters retirement or an Endpoint owner loses authority, when the transition commits, then control access, inbound credentials, new ordinary events, and queued ordinary payloads stop immediately; precomputed minimized tombstones for immutable prior receiving-target versions are authorized under platform lifecycle policy and those versions enter retiring status solely for bounded attempts while possession and destination safety remain current, before their outbound signing material and destinations disable, unless an accepted security revocation requires immediate cutoff and records an owned privacy exception instead.
5. Given a data-subject or Community lifecycle report is produced, when integration propagation is summarized, then it distinguishes queued, delivered, acknowledged if supported, terminal, subscriber-responsible, and locally completed states and never treats HTTP acceptance as proof of remote erasure.

### States and accessibility

Show lifecycle queued, delivered, terminal, subscriber-responsible, Endpoint disabled, and locally
complete states with timestamps, contract version, safe remediation, and no misleading success label.

### Verification

- Domain/unit: lifecycle-event eligibility, minimization, deduplication, and completion semantics.
- Integration/contract: tombstone schema, immutable receiving-target version, URL/subscription changes, retries, retirement, and reports.
- Component: propagation status, terminal exception, retirement, and responsibility explanations.
- Journey/live: deliver content, delete it, fail its tombstone, retire the Community, and inspect outcomes.
- Manual/rendered: privacy wording, keyboard, screen reader, status distinctions, and operator usability.

## L2-INTG-015: Manage Webhook Endpoint lifecycle

**Traces to:** L1-INTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Webhook Endpoint subscriptions, destinations, status, signing secrets, ownership, and retirement change
through current-version transitions whose queued-work and authorization effects are explicit.

### Acceptance criteria

1. Given a current authorized Community administrator changes an active Endpoint's subscribed event allow-list, when the versioned update commits, then every requested type is revalidated against current Permission and Entitlements, the exact prior and resulting set is audited, and queued Deliveries are reauthorized or suppressed against the resulting set before attempt.
2. Given the destination URL changes, when the update commits, then the old target version stops receiving ordinary events immediately and a distinct new target version remains unverified until the complete DNS, redirect, TLS, network, and possession challenge succeeds; the old version retains only bounded, minimized tombstone eligibility for payloads it previously accepted while its proof and destination remain safe, and neither queued nor lifecycle work is rerouted across versions.
3. Given an authorized administrator pauses, resumes, or deletes an Endpoint, when the current-version transition commits, then pause admits no new ordinary Delivery and retains only the declared bounded queue, resume repeats destination and ownership validation before work continues, and delete revokes control and secrets, terminalizes queued work, and retains only policy-required redacted history.
4. Given a signing secret rotates, when activation commits, then the new secret is revealed once, signatures identify their key version, an accepted bounded overlap permits in-flight verification, and the old secret expires automatically; an emergency revoke has no overlap, and each later attempt signs with the one active version selected at attempt time.
5. Given concurrent changes, a repeated operation, Community retirement, owner authority loss, possession-proof expiry, failed scheduled revalidation, or a destination that becomes prohibited occurs, when lifecycle processing runs, then one versioned transition prevails, control and ordinary delivery stop before the next protected payload, fresh challenge is required where recovery is allowed, lifecycle-tombstone handling follows immutable receiving-target and retiring boundaries, and no stale worker, credential, or client state can reactivate the Endpoint.

### States and accessibility

Show unverified, active, paused, failing, rotating, retiring, disabled, deleted, conflict, and
prohibited states with destination-safe diagnostics, queue consequences, confirmation, and no secret
redisplay.

### Verification

- Domain/unit: subscription, target version, proof expiry, status, rotation overlap, retirement, and queue-effect rules.
- Integration/contract: destination change, scheduled challenge, immutable tombstone route, pause/resume/delete, rotate/revoke, and stale worker.
- Component: edit, validation, one-time secret, impact confirmation, conflict, and terminal states.
- Journey/live: update subscriptions, rotate, change URL, pause/resume, lose ownership, and retire.
- Manual/rendered: security copy, redaction, keyboard, screen reader, focus, and destructive-action review.
