# MKTG — Marketing and public web detailed requirements

## L2-MKTG-001: Marketing remains static and runtime-independent

**Traces to:** L1-MKTG-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The public marketing site shall use meaningful static HTML and CSS with only minimal progressive JavaScript. It shall reside outside the Angular workspace, own `/`, consume canonical design-system assets through a deterministic build step, render useful navigation and content with JavaScript unavailable, and neither load the Angular application bundle nor call private authenticated APIs.

### Acceptance criteria

1. Given JavaScript is disabled, when a visitor requests `/`, then the primary proposition, navigation, proof, and next action remain readable and operable.
2. Given the root page loads, when its network requests are inspected, then no Angular application bundle or private authenticated API call is present.
3. Given shared visual foundations change, when marketing is rebuilt, then it consumes the canonical design-system output without a hand-maintained duplicate token source.
4. Given marketing-specific composition, when source ownership is inspected, then it remains under `marketing/` unless the authenticated application genuinely needs the same component semantics.

### States and accessibility

The static surface shall provide keyboard navigation, visible focus, semantic landmarks/headings, zoom/reflow, reduced-motion behavior, and usable content under asset or script failure.

### Verification

- Domain/unit: Validate static link, metadata, and asset-reference rules.
- Integration/contract: Build marketing and inspect generated HTML/network dependencies.
- Component: Test any progressive enhancement independently from core content.
- Journey/live: Browse `/` and follow primary navigation with JavaScript disabled.
- Manual/rendered: Inspect semantic structure, focus, narrow view, and canonical styling.

## L2-MKTG-002: Content follows a clear outcome-to-action narrative

**Traces to:** L1-MKTG-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The initial public content shall communicate, in order, the people served and outcome, the problem in their vocabulary, the community's differentiating mechanism, a concrete workflow or artifact, credible proof or honest current constraints, the next action, and applicable privacy, terms, contact/support, and accessibility links. Real interface imagery or lightweight HTML demonstrations shall be preferred to generic decoration.

### Acceptance criteria

1. Given a first-time visitor scans the page, when content order is inspected, then audience/outcome, problem, mechanism, concrete experience, proof/status, and next action are discoverable in that sequence.
2. Given product vocabulary, when marketing copy is compared with the glossary, then visitor-friendly synonyms map to one canonical community concept and do not invent a competing domain model.
3. Given proof or product status is presented, when verified against current capability, then it is specific and honest rather than an unsupported production, security, integration, or customer-outcome claim.
4. Given legal, privacy, support, or accessibility links apply to launch, when the footer or equivalent navigation is used, then each destination resolves to meaningful content.

### States and accessibility

Content shall retain logical heading order, meaningful alternative text, readable line lengths, sufficient contrast, and understandable layout on narrow screens and at zoom.

### Verification

- Domain/unit: Validate required content regions and link destinations in built HTML.
- Integration/contract: Verify public copy and product-status data against accepted product documentation.
- Component: Test any HTML demonstration in default and failure states.
- Journey/live: Ask representative visitors to identify audience, mechanism, proof, and next action.
- Manual/rendered: Review content hierarchy, realism, and accessibility links.

## L2-MKTG-003: Calls to action and product claims are honest and stable

**Traces to:** L1-MKTG-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every primary call to action shall have one stable destination and measurable intent. Sign-in shall target `/sign-in`; account creation shall target `/sign-up` or the canonical onboarding route. Marketing shall not link primary actions to an authenticated dashboard or a mock file, and shall use ordinary anchors that work before JavaScript. Application sign-out shall return to its sign-in route rather than ambiguous `/`.

### Acceptance criteria

1. Given a sign-in or account-creation link in built marketing HTML, when followed without JavaScript, then it reaches the configured public application route rather than a mock or authenticated-only destination.
2. Given repeated primary calls to action, when their intent is the same, then they resolve to one stable configured destination and share a documented measurement meaning.
3. Given a member signs out, when sign-out completes, then navigation returns to the application sign-in route and does not imply that marketing owns session recovery.
4. Given a claimed integration, security property, customer result, or readiness level, when the claim lacks current evidence, then it is omitted or labeled as planned/limited rather than presented as complete.

### States and accessibility

Calls to action shall have descriptive accessible names, visible focus, keyboard operation, and clear disabled/unavailable explanation if onboarding is not open.

### Verification

- Domain/unit: Parse built HTML and assert canonical CTA destinations.
- Integration/contract: Verify configured authentication routes and analytics intent identifiers.
- Component: Test CTA enhancement without breaking anchor behavior.
- Journey/live: Follow sign-in, sign-up/onboarding, and sign-out paths in a deployed build.
- Manual/rendered: Review link text, focus, claim accuracy, and destination clarity.

## L2-MKTG-004: One-origin publishing assembles deterministic assets

**Traces to:** L1-MKTG-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

The first-phase publish process shall build Angular with content-hashed bundles, emit its entry under a non-marketing name such as `app.html`, copy Angular output into API static files, copy canonical design-system assets, copy marketing while leaving its `index.html` as the root default, and publish one API artifact. Every step shall be deterministic and fail on missing expected output.

### Acceptance criteria

1. Given a clean publish, when the artifact is inspected, then marketing `index.html`, application `app.html`, content-hashed Angular bundles, canonical design assets, and API binaries are all present once at their documented locations.
2. Given the artifact root, when its default document is resolved, then marketing remains the owner rather than the Angular entry document.
3. Given an expected application, marketing, or design-system output is missing, when publishing runs, then the process fails rather than shipping a partial artifact.
4. Given the same locked source and toolchain inputs, when publishing repeats, then route ownership and manifest relationships remain deterministic.

### States and accessibility

Published assets shall include every stylesheet, font, icon, image, social image, and reduced-motion behavior required by accessible public and application rendering.

### Verification

- Domain/unit: Test publish-manifest and expected-file validation.
- Integration/contract: Produce and inspect the complete artifact from a clean checkout.
- Component: Run application and marketing component/static checks against published assets.
- Journey/live: Serve the artifact over HTTP and traverse public-to-app navigation.
- Manual/rendered: Inspect artifact contents and render both entry documents.

## L2-MKTG-005: Request routing protects every runtime owner

**Traces to:** L1-MKTG-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

The deployed middleware pipeline shall allow static defaults to claim `/` before any SPA fallback. Known marketing and asset paths shall serve static files; known Public Projection routes shall reach their anonymous HTML renderer; `/api/*`, `/hubs/*`, and `/health` shall reach ASP.NET Core; known member routes shall serve `app.html` and then Angular; unknown non-file routes shall follow a documented app-fallback or 404 policy; and unknown files shall never receive an accidental SPA shell with `200`.

### Acceptance criteria

1. Given `/`, a known marketing file, a canonical asset, and an eligible Public Projection route, when each is requested, then the documented static or anonymous HTML owner responds, the root is marketing HTML, and none falls through to `app.html`.
2. Given `/api/*`, `/hubs/*`, or `/health`, when requested, then the endpoint is not intercepted by static default or SPA fallback middleware.
3. Given a refreshed known member deep link, when requested, then `app.html` is returned and Angular renders the intended route.
4. Given an unknown file path, when requested, then it returns the deliberate not-found behavior rather than `app.html` with a successful status.
5. Given an unknown non-file path, when requested, then behavior matches the documented fallback/404 policy consistently.

### States and accessibility

Not-found, service-unavailable, and member-route fallback responses shall provide appropriate semantics and shall not trap users in an unannounced blank shell.

### Verification

- Domain/unit: Test route-classification policy where isolated.
- Integration/contract: Exercise every routing class over the published HTTP artifact.
- Component: Verify Angular not-found/deep-link components where used.
- Journey/live: Refresh a member deep link and navigate public, API, health, and missing paths.
- Manual/rendered: Inspect status codes, response bodies, middleware order, and error pages.

## L2-MKTG-006: Public-to-application navigation is deployment-configurable

**Traces to:** L1-MKTG-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

During same-origin deployment, Angular shall use relative API and hub paths and development shall use a proxy or injected base URL. Library code shall receive origins through injection tokens. Marketing CTA destinations shall be ordinary relative anchors in same-origin deployment and shall become deployment-configured absolute `app` URLs when origins split, without source edits to reusable libraries.

### Acceptance criteria

1. Given a same-origin production build, when application network calls and marketing anchors are inspected, then they use relative paths and contain no development hostname.
2. Given local development, when Angular calls the API, then a documented proxy or injected development URL supplies the origin.
3. Given a split-origin deployment, when marketing builds, then configured absolute application origins populate sign-in and onboarding links without hand-editing HTML per environment.
4. Given an Angular library, when its source is inspected, then it neither hard-codes a hostname nor reads a deployment-specific global.

### States and accessibility

Navigation remains standard anchor/router behavior with descriptive names, focus visibility, and functional keyboard operation in every topology.

### Verification

- Domain/unit: Test origin/CTA configuration mapping.
- Integration/contract: Build same-origin and split-origin configurations and inspect output.
- Component: Test injected API and hub base configuration.
- Journey/live: Traverse marketing-to-auth and application-to-API paths in each supported topology.
- Manual/rendered: Review generated URLs and standard link behavior.

## L2-MKTG-007: Public metadata and indexing are deliberate

**Traces to:** L1-MKTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every public page shall provide a unique title and description, semantic headings, Open Graph and social-card metadata, a social image, favicon, and canonical URL when the production domain is known. The public site shall provide `robots.txt`, `sitemap.xml`, and appropriate structured data. Authenticated application and API routes shall be excluded from indexing, and robots/sitemap content shall describe only intended public URLs.

### Acceptance criteria

1. Given each public page, when its built head is inspected, then title, description, canonical URL when known, social metadata/image, and favicon are present and page-specific.
2. Given `robots.txt` and `sitemap.xml`, when fetched in production, then they resolve successfully and enumerate or permit only intended public content.
3. Given an application or API route, when crawler directives and response metadata are inspected, then it is not presented as indexable public content.
4. Given structured data is emitted, when validated, then its type and claims match visible, currently true page content.

### States and accessibility

Metadata complements rather than replaces semantic visible headings and meaningful image alternatives; social imagery does not carry essential content alone.

### Verification

- Domain/unit: Validate metadata uniqueness, canonical formats, robots, sitemap, and structured data.
- Integration/contract: Fetch built public and protected routes and inspect crawler-facing responses.
- Component: Test metadata generation for parameterized public pages if any.
- Journey/live: Run public discovery smoke checks on staging or the published artifact.
- Manual/rendered: Review search/social previews and visible heading correspondence.

## L2-MKTG-008: Public performance has an explicit budget

**Traces to:** L1-MKTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

The public site shall define and enforce a performance budget for useful-content rendering, transferred JavaScript, CSS, fonts, images, and layout stability. Useful content shall not require client JavaScript or the Angular bundle. Images shall use suitable dimensions and formats with reserved layout space; fonts should be self-hosted when licensing permits; and non-critical scripts shall be deferred.

### Acceptance criteria

1. Given a production public-page build, when measured under the documented profile, then each declared performance-budget threshold passes or the release is blocked with an approved exception.
2. Given JavaScript fails or is disabled, when the page loads, then useful content and primary navigation still render.
3. Given images and fonts load slowly, when the page renders, then reserved dimensions prevent avoidable layout shift and fallback typography keeps content usable.
4. Given a non-critical script or embed, when page loading begins, then it is deferred and does not block primary content rendering.

### States and accessibility

Performance optimization shall preserve readable zoomed text, alternative text, focus order, motion preferences, and content rather than hide essential information behind lazy scripts.

### Verification

- Domain/unit: Validate media dimensions and budget configuration.
- Integration/contract: Run repeatable production performance checks and asset analysis.
- Component: Test progressive enhancement under script and asset failure.
- Journey/live: Load the public journey under constrained network and disabled JavaScript.
- Manual/rendered: Inspect layout stability, font fallback, and meaningful first render.

## L2-MKTG-009: Analytics, embeds, and authenticated data remain private by design

**Traces to:** L1-MKTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Public analytics and embeds shall have a documented product question, data purpose, minimization rule, retention/ownership, and consent behavior before activation. The static marketing site shall not call authenticated/private APIs or expose member data. Privacy, support, accessibility, and applicable legal destinations shall be reachable without requiring tracking consent.

### Acceptance criteria

1. Given an analytics event or third-party embed, when proposed, then its declared question, collected fields, purpose, consent requirement, retention, processor/owner, and disable behavior are documented.
2. Given consent is required and not granted, when a visitor loads the page, then non-essential analytics or embeds do not execute and primary content remains usable.
3. Given marketing network traffic, when inspected, then no authenticated API, member identifier, private community content, token, or session detail is transmitted to the public surface.
4. Given a visitor declines tracking, when they follow privacy, support, accessibility, sign-in, or onboarding links, then those destinations remain operable.

### States and accessibility

Consent controls shall be keyboard operable, clearly labeled, non-coercive, and understandable without color or motion; embedded fallbacks shall be accessible.

### Verification

- Domain/unit: Validate analytics schemas against declared fields and consent categories.
- Integration/contract: Inspect network behavior before and after consent.
- Component: Test consent and embed fallback states.
- Journey/live: Complete public navigation with consent declined and storage cleared.
- Manual/rendered: Review privacy language, control accessibility, and collected payloads.

## L2-MKTG-010: Cache policy distinguishes immutable and revalidated content

**Traces to:** L1-MKTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Content-hashed Angular bundles and versioned immutable assets shall receive long-lived immutable caching, normally one year. HTML, unhashed CSS/JavaScript, robots, sitemap, and unversioned downloads shall require revalidation unless their filenames are versioned. Release retention or invalidation shall ensure an older cached HTML shell cannot reference removed bundles.

### Acceptance criteria

1. Given a content-hashed bundle or versioned immutable asset, when fetched, then its cache headers permit the documented long immutable lifetime.
2. Given HTML, robots, sitemap, or an unhashed asset, when fetched, then cache headers require revalidation rather than treating it as immutable.
3. Given a client holds the immediately previous HTML shell during deployment, when it requests referenced bundles, then those bundles remain available or an atomic release strategy prevents a broken shell.
4. Given a downloadable document is intentionally cached immutably, when reviewed, then its filename is versioned and replacement behavior is documented.

### States and accessibility

Cache transitions shall not leave users with an inaccessible blank shell; recovery behavior shall retain a readable error and retry path.

### Verification

- Domain/unit: Test cache-policy mapping by asset class.
- Integration/contract: Assert headers from the published HTTP artifact.
- Component: Verify application update/retry presentation if a bundle cannot load.
- Journey/live: Simulate a release while a prior shell is cached.
- Manual/rendered: Inspect CDN/server configuration and stale-client behavior.

## L2-MKTG-011: Origin splitting requires a documented trigger and security plan

**Traces to:** L1-MKTG-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Marketing shall move to an independent `www` host and the app/API to `app` only when an explicit trigger exists: independent ownership or release cadence, multiple content types/localization/CMS, edge/CDN/custom-domain needs, materially different anonymous traffic, or distinct availability, security, or deployment ownership. Before splitting, the decision shall address DNS, TLS, redirects, absolute CTA configuration, per-host robots/sitemaps, analytics boundaries, CSP, rollback, and cookie-domain/CSRF implications. CORS shall not be added merely for marketing because marketing does not call the API.

### Acceptance criteria

1. Given a split-origin proposal, when its ADR is reviewed, then at least one observable trigger is current and consequences, costs, reversibility, and owner are recorded.
2. Given the split is approved, when rollout planning completes, then DNS, TLS, redirects, CTA origins, crawler files, analytics boundaries, CSP, monitoring, and rollback are covered.
3. Given header-based app authentication remains isolated to `app`, when marketing traffic is inspected, then no cross-origin API dependency or unnecessary CORS policy is introduced.
4. Given cookie authentication is used across relevant hosts, when the design is reviewed, then cookie domain, SameSite, secure transport, and CSRF controls are explicit and tested.

### States and accessibility

Redirects and cross-host navigation shall preserve meaningful titles, focus on destination, keyboard operation, authentication error recovery, and accessible failure pages.

### Verification

- Domain/unit: Test host/CTA/redirect configuration mapping.
- Integration/contract: Verify TLS, redirects, CSP, cookies/CSRF where applicable, and absence of unnecessary CORS.
- Component: Test authentication navigation and failure presentation after host changes.
- Journey/live: Traverse public-to-app, sign-in, sign-out, and rollback paths across hosts.
- Manual/rendered: Review the ADR, DNS/host plan, and per-host crawler/analytics configuration.

## L2-MKTG-012: Published public behavior is release-blocking evidence

**Traces to:** L1-MKTG-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every release shall verify the published artifact or staging host over HTTP rather than relying on a development server. Verification shall cover marketing root ownership without Angular, refreshed application deep links, API/health/realtime exclusion from fallback, all styles/fonts/media/downloads/social metadata, sign-in/onboarding destinations, cache headers, unknown-file behavior, keyboard flow, zoom/reflow, reduced motion, representative viewports, robots, and sitemap. A failure in those contracts shall block release.

### Acceptance criteria

1. Given the release candidate, when `/` and a refreshed member route are requested, then marketing owns the root without Angular and the member route returns `app.html` and renders correctly.
2. Given API, health, realtime, asset, metadata, CTA, and missing-file requests, when smoke tests run, then each reaches its documented owner, headers, destination, or not-found behavior.
3. Given keyboard-only use, zoom/reflow, reduced motion, and representative narrow/wide viewports, when the public-to-auth journey is reviewed, then it remains operable and understandable.
4. Given robots and sitemap are fetched, when compared with the deployed public route set, then they describe only intended indexable content.
5. Given any release smoke assertion fails, when deployment promotion is evaluated, then promotion stops rather than accepting development-server evidence.

### States and accessibility

Published verification shall include public loading/media failure, not-found, unavailable, consent, focus, reflow, and reduced-motion behavior.

### Verification

- Domain/unit: Run static validation for links, metadata, and route manifests.
- Integration/contract: Execute HTTP smoke tests against the exact release artifact.
- Component: Run progressive enhancement and public-state tests.
- Journey/live: Complete public-to-sign-in/onboarding journeys at representative preferences.
- Manual/rendered: Inspect published pages, headers, metadata previews, and accessibility behavior.

## L2-MKTG-013: Publish indexable Community content safely

**Traces to:** L1-MKTG-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Explicitly public Communities, Profiles, Posts, and Events may have crawlable Public Projections whose
route, fields, metadata, cache, and retirement remain separate from the authenticated application.

### Acceptance criteria

1. Given public indexing is enabled for a resource class by the current Community Owner or a Community Administrator holding `ManagePublicIndexing`, when the current Community Owner opts in the Community, the owning Account holder opts in their Profile or Post, or a current Event Organizer holding `ManageEventPublicProjection` opts in their Event, then one stable canonical Public Projection exposes only the allow-listed anonymous fields if current Community, author/organizer, content, privacy, and Moderation state all permit it; every resource is non-indexable by default, and support impersonation, break-glass, or platform operation cannot supply owner consent.
2. Given a Visitor or crawler requests a Public Projection, when it is rendered, then useful semantic HTML, title, description, canonical URL, social metadata, and applicable structured data are available without Angular, a session, personalized state, or a private API call, under an explicit route owner that cannot fall through to the application shell.
3. Given robots and sitemap artifacts are generated, when eligible Public Projections change, then only current canonical URLs appear, aliases redirect without competing canonicals, and private, draft, restricted, duplicate, parameterized, application, API, and search-result URLs remain excluded.
4. Given public consent is withdrawn or visibility, publication, Account lifecycle, deletion, Community retirement, or Moderation Action makes a projection ineligible, when the change commits, then the canonical public route returns `404` when existence is unsafe to disclose or `410` when the previously public retirement may be disclosed, never a `200` tombstone or application shell; either response and any safe HTML carry `noindex, nofollow`, expose no private fields, and complete cache/metadata invalidation and sitemap removal within the accepted public-projection objective, default five minutes, without claiming a crawler's later removal time.
5. Given content is publicly indexable, when the owner enables or reviews it, then the UI explains anonymous access, search/cache and sharing consequences, Block limitations for logged-out viewers, opt-out effect, and any retained moderation or legal record without using public status to grant mutation or private-field access.

### States and accessibility

Show private-by-default, public-preview, confirmation, indexable, updating, retired, moderated, and
unavailable states; public HTML remains semantic, localized, keyboard usable, and reflow safe.

### Verification

- Domain/unit: eligibility matrix, opt-in authority, allow-list, canonical identity, and retirement rules.
- Integration/contract: public route, role/Permission authority, 404/410, metadata, robots, sitemap, cache invalidation, and auth/API isolation.
- Component: public preview, consent, consequence explanation, opt-out, and retired states.
- Journey/live: publish each resource type, crawl it, narrow visibility, and prove projection withdrawal.
- Manual/rendered: no-JavaScript, metadata preview, keyboard, screen reader, localization, and reflow.
