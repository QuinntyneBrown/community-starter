# MESS — Messaging and realtime behavior

## L2-MESS-001: Start an authorized Conversation

**Traces to:** L1-MESS-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every Conversation has one immutable Community and optional immutable same-Community Space scope.
The server creates it only when every initial participant is eligible under current Account,
Membership, Space, rules, Moderation Action, Block, and recipient-contact policy.

### Acceptance criteria

1. Given two eligible Accounts with active Membership and accepted applicable rules request initial direct contact in the same selected Community and optional Space, when creation is requested in either participant order, then the one stable Conversation for that unordered pair and exact immutable scope is returned or created and the same transaction applies the recipient's direct-admission policy to create either one pending request or one active consent interval; a shell alone grants no participation, and the same pair in another Community is distinct.
2. Given a participant is unknown, inactive, outside the permitted scope, or not contactable, when creation is requested, then the server returns a scoped safe failure and persists no Conversation.
3. Given the same or different idempotency keys concurrently request the same direct Account pair/scope, when creation commits, then a uniqueness constraint and retry policy produce one Conversation and no parallel direct thread.
4. Given direct creation lacks Permission or rules acceptance, uses an inaccessible or cross-Community Space, attempts to migrate an existing Conversation's Community/Space, violates content bounds, or retries a prior operation, when processed, then it creates or changes no Conversation or returns the original result without a duplicate.

### States and accessibility

Show selecting, validating, created, ineligible, throttled, and unavailable states; identify invalid
participants without leaking private Accounts, retain valid choices, and announce the result.

### Verification

- Domain/unit: participant eligibility, Conversation/admission uniqueness, and idempotency rules.
- Integration/contract: create with pending/preauthorized admission, reciprocal race, scoped failure, retry, and unauthorized contracts.
- Component: participant picker, validation, busy, and result states.
- Journey/live: start a Conversation and open its committed detail.
- Manual/rendered: keyboard, screen-reader, narrow-screen, and error-summary review.

## L2-MESS-002: Enforce Message request and Block policy

**Traces to:** L1-MESS-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Recipient preferences, Message-request state, and Blocks are evaluated on the server for discovery,
Conversation creation, sending, notifications, and realtime delivery.

### Acceptance criteria

1. Given otherwise eligible Accounts with no prior terminal direct interval start contact, when Account/Membership, Community/Space, rules, recipient contact preference, Block, Moderation Action, capacity, pair/scope request, and interval state are atomically evaluated, then approval-required policy creates exactly one pending request whose recorded recipient alone may decide, while explicit no-approval policy creates exactly one active interval for both participants at the disclosed current `visibleFrom` boundary and terminalizes a stale pending request without Message or Delivery; concurrent, reciprocal, creation, request, or retry paths produce only that one safe pending-or-active result.
2. Given the intended recipient accepts or declines a pending request, when the transition commits, then the server atomically rechecks request target/version/expiry, sender and recipient Account/Membership, Community/Space access and rules, current contact preference, Block, Moderation Action, capacity, request state, and absence of another active interval; only one eligible acceptance opens one disclosed direct consent interval and Deliveries, while stale, ineligible, expired, duplicate, or replayed decisions grant nothing and preserve a privacy-safe terminal/prior result.
3. Given either Account has an applicable Block, when discovery, request, send, notification, or subscription is attempted, then the action reveals no private status and no Message or Delivery is created.
4. Given a Block or Account/Membership eligibility, Community/Space access, applicable rules, contact-policy, or Moderation state ends eligibility for an active direct Conversation, when access is reevaluated, then the current consent interval receives a fixed terminal boundary, further interaction and queued effects stop, prior history follows current visibility/retention policy, authorized evidence remains available, and later eligibility restoration cannot reactivate that interval.
5. Given the original sender withdraws a current request, the intended recipient declines, the idempotent expiry scheduler reaches its deadline, or those transitions race acceptance, no-approval admission, or retry, when the request version commits, then exactly one ordered outcome wins; a terminal withdrawal/decline/expiry suppresses queued request notices, cannot later open an interval, exposes only privacy-safe status, and prevents a new request or alternate Conversation until the configured cooldown, while a previously committed acceptance remains authoritative.

### States and accessibility

Show request pending with sender withdrawal, accepted, declined/unavailable, withdrawn, expired,
cooldown, blocked-safe, and error states without exposing the recipient's private choice; all
decisions and recovery actions are keyboard operable and expiry is announced without a countdown.

### Verification

- Domain/unit: pair/scope request and active-interval uniqueness, version/expiry, acceptance eligibility, and termination matrix.
- Integration/contract: reciprocal admission, accept/withdraw/decline/expiry races, duplicate acceptance, scope loss, Block, notices, and send enforcement.
- Component: pending/withdraw controls, accept/decline, expired/cooldown, and safe unavailable states.
- Journey/live: race withdrawal/expiry with acceptance, then lose/restore scope and require fresh consent.
- Manual/rendered: privacy copy, focus order, announcements, and narrow-screen review.

## L2-MESS-004: Commit one validated Message

**Traces to:** L1-MESS-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

A send use case reauthorizes the sender, validates bounded content, persists one Message atomically,
and schedules downstream work only after the commit succeeds.

### Acceptance criteria

1. Given an active participant and valid bounded content, when a Message is sent, then one Message with stable identity and server timestamp is committed.
2. Given invalid content, no active consent interval, lost owning-Community Membership, archive-pending/archived optional Space, lost Space access/rules acceptance, Block, or rate limit, when sending is attempted, then a typed safe failure is returned and no Message, Notification, or realtime event is created.
3. Given the same client operation is retried, when the server processes it, then the committed Message is returned without duplicate content or downstream effects.
4. Given persistence fails, when the send use case terminates, then no success response or publishable event is produced.

### States and accessibility

Show composing, sending, sent, validation, throttled, permission-changed, retryable, and failed states;
preserve recoverable text, announce failure, and never rely on color for delivery status.

### Verification

- Domain/unit: content bounds, send eligibility, idempotency, and ended-interval rules.
- Integration/contract: persistence, failure mapping, duplicate retry, and downstream outbox behavior.
- Component: composer, pending, retry, validation, and permission-change states.
- Journey/live: send once through the live API and verify one persisted Message and one visible result.
- Manual/rendered: keyboard composer, zoom, status announcement, and long-content review.

## L2-MESS-006: Page Messages and track reading privately

**Traces to:** L1-MESS-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Authorized participants retrieve a stable cursor-paginated Message history, while read state follows
the configured privacy contract and never leaks activity to an ineligible viewer.

### Acceptance criteria

1. Given an authorized participant, when Message pages are requested, then results use stable ordering and cursors without duplicates, skips, or inaccessible items.
2. Given a cursor is invalid or outside the retained history, when used, then a typed failure is returned without exposing Message or participant details.
3. Given read receipts are enabled and a participant marks a committed boundary read, when saved, then read state advances monotonically and duplicate updates have no extra effect.
4. Given read receipts are disabled or the requester lost access, when presence or read state is requested, then private activity is not returned or updated.

### States and accessibility

Show initial loading, load-older, empty, unread boundary, privacy-disabled, and retry states; preserve
reading position, offer a keyboard load-more action, and avoid noisy per-Message announcements.

### Verification

- Domain/unit: cursor order, monotonic read boundary, and receipt-privacy rules.
- Integration/contract: pagination, invalid cursor, authorization, and read-update contracts.
- Component: history loading, unread marker, load-more, and retry behavior.
- Journey/live: page a live history and synchronize read state across two authorized sessions.
- Manual/rendered: focus preservation, screen-reader structure, narrow-screen, and long-thread review.

## L2-MESS-007: Apply edit, deletion, and retention policy

**Traces to:** L1-MESS-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Message edits, author removal, participant deletion requests, and retention are distinct,
server-authorized transitions with disclosed effects on history and evidence.

### Acceptance criteria

1. Given an author is within the allowed edit policy, when valid replacement content is submitted against the current version, then the Message changes once and is visibly marked edited.
2. Given a stale version, unauthorized actor, or expired edit window, when edit or deletion is attempted, then a conflict or forbidden response is returned and content remains unchanged.
3. Given author removal is permitted, when committed, then normal participants see the declared tombstone while required safety evidence remains restricted and intact.
4. Given retention expiry or an eligible data request, when lifecycle processing runs, then Message data is erased, anonymized, or retained exactly under the documented policy with an auditable result.

### States and accessibility

Show editing, conflict, edited, removal confirmation, tombstone, retention-limited, and error states;
preserve drafts on conflict and label history changes without color alone.

### Verification

- Domain/unit: edit window, concurrency, tombstone, and retention transition rules.
- Integration/contract: edit, stale conflict, delete, evidence, and retention-job contracts.
- Component: edit form, conflict recovery, confirmation, and tombstone states.
- Journey/live: edit then remove a Message and verify participant versus moderator views.
- Manual/rendered: keyboard, focus recovery, destructive copy, and announcement review.

## L2-MESS-008: Report a Message without losing evidence

**Traces to:** L1-MESS-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

An eligible participant can Report a Message with a restricted evidence snapshot that survives later
edits, author removal, participant departure, and ordinary retention processing.

### Acceptance criteria

1. Given an eligible participant can currently view a Message or presents an unexpired Account-bound Report Receipt from a prior authorized view, when a valid Report is submitted, then one Report and the exact receipt/current-view Message-version evidence commit atomically without notifying the reported Account directly.
2. Given the Message changes or is removed after reporting, when the Moderation Case is reviewed, then authorized reviewers can access the captured evidence and its provenance.
3. Given an ineligible actor, invalid target, guessed identifier, or absent/expired/forged Report Receipt without current access, when reporting is attempted, then no Report or evidence is created and private Message existence or receipt validity is not disclosed.
4. Given the reporter later loses Conversation access, when status is checked, then only the reporter-safe acknowledgement remains available, not restricted case details.

### States and accessibility

Show reason selection, optional context, submitting, acknowledged, duplicate-safe, and unavailable
states; use explicit labels, accessible error association, and safe confirmation copy.

### Verification

- Domain/unit: reporter eligibility, evidence immutability, and disclosure rules.
- Integration/contract: report, edit/delete-after-report, unauthorized, and status contracts.
- Component: report dialog, validation, acknowledgement, and unavailable state.
- Journey/live: report a Message, remove it, and review preserved evidence as an authorized Moderator.
- Manual/rendered: keyboard dialog, screen-reader labels, sensitive-copy, and narrow-screen review.

## L2-MESS-009: Publish only committed Message events

**Traces to:** L1-MESS-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Realtime Message events are stable, privacy-minimized hints emitted from committed state and never
serve as the durable source of truth.

### Acceptance criteria

1. Given a Message transaction commits, when post-commit processing runs, then eligible current participants may receive an event with stable Message and Conversation identifiers.
2. Given validation, authorization, or persistence fails, when the use case ends, then no Message-created event is publishable.
3. Given event publication fails after commit, when retry processing runs, then the Message remains readable through the API and duplicate events do not duplicate UI state.
4. Given a recipient loses access before delivery, when the event is dispatched, then the recipient is excluded and no private payload is sent.

### States and accessibility

Realtime arrival may update a visible thread or unread indicator without stealing focus; announce
only meaningful aggregate changes and provide a non-realtime refresh path.

### Verification

- Domain/unit: event contract, privacy-minimized payload, and recipient eligibility rules.
- Integration/contract: transaction/outbox, retry, duplicate, and revoked-recipient behavior.
- Component: duplicate-safe insertion, unread update, and non-disruptive announcement.
- Journey/live: commit a Message, interrupt publication, recover, and reconcile without duplication.
- Manual/rendered: focus stability, announcement volume, reduced-motion, and background-tab review.

## L2-MESS-010: Reconnect and reconcile from the API

**Traces to:** L1-MESS-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Clients treat disconnects, missed events, duplicates, and out-of-order events as normal and
reconcile Conversation state from authorized API contracts.

### Acceptance criteria

1. Given realtime disconnects, when the client reconnects, then it resubscribes with current authentication and reloads from its last committed cursor or state boundary.
2. Given events are duplicated or arrive out of order, when processed, then stable identities and versions prevent regression or duplicate Messages.
3. Given the Session expires or Conversation access is revoked while disconnected, when reconnect is attempted, then the subscription is refused and cached private state is no longer presented as current.
4. Given realtime remains unavailable, when the member continues, then API refresh and send behavior remain usable with a clear degraded-state indication.

### States and accessibility

Show connected, reconnecting, offline/degraded, access-lost, and retry states without blocking the
Message history; announce sustained state changes once and respect reduced motion.

### Verification

- Domain/unit: version reconciliation and duplicate/out-of-order reduction rules.
- Integration/contract: reconnect authentication, cursor reload, and revoked-access contracts.
- Component: status, reconciliation, retry, and stale-state handling.
- Journey/live: disconnect, create Messages elsewhere, revoke access, and exercise both reconnect paths.
- Manual/rendered: screen-reader status, focus stability, reduced-motion, and weak-network review.

## L2-MESS-011: Scope realtime subscriptions on the server

**Traces to:** L1-MESS-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every realtime connection and Conversation subscription is authenticated, server-scoped, bounded,
revocable, and observable without trusting a client-supplied channel identifier.

### Acceptance criteria

1. Given a valid Session and accessible Conversation, when subscription is requested, then the server derives the authorized channel scope and records bounded connection state.
2. Given a guessed Conversation identifier, expired Session, Block, lost owning-Community Membership, or lost optional Space access/rules acceptance, when connection or subscription is attempted, then it is denied without revealing resource existence.
3. Given access changes during a connection, when revocation processing runs, then the connection loses that subscription before later private events are delivered.
4. Given connection or subscription limits are exceeded, when additional work is attempted, then it is throttled without affecting durable API correctness.

### States and accessibility

Connection denial and throttling use safe actionable status rather than exposing hidden resources;
durable content remains available through the accessible API fallback when allowed.

### Verification

- Domain/unit: subscription scope, revocation, and bounded-connection rules.
- Integration/contract: handshake, guessed ID, expiry, live revocation, and throttle tests.
- Component: safe access-lost and degraded fallback behavior.
- Journey/live: connect two Communities, revoke one scope, and prove events never cross boundaries.
- Manual/rendered: error clarity, retry controls, and assistive-technology status review.

## L2-MESS-016: End and re-enter direct Conversations deterministically

**Traces to:** L1-MESS-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The one stable direct Conversation for an Account pair and scope may hold multiple explicitly
accepted consent intervals. Either participant, current eligibility policy, or a confirmed Space-
archive dependency plan may end the exact current interval without reading content; neither an ended
interval, leave, Block removal, nor later eligibility silently restores interaction or history.

### Acceptance criteria

1. Given either direct participant ends the current interval, Account/Membership eligibility, Community/Space access, rules, contact policy, Block, or Moderation state invalidates it, or a current Community administrator holding `ArchiveSpace` executes the confirmed Space plan for an exact listed Conversation, when current scope, authority, plan if applicable, and lifecycle versions are rechecked and the transition commits, then send and participant `visibleThrough` boundaries are fixed, the pending request terminalizes, new interaction stops, queued effects reauthorize or suppress, and a lifecycle executor gains no Message-read or participant authority while trigger, identities, Audit Event, and privacy-safe notice are recorded.
2. Given an authorized former participant reads an ended direct interval, when history is requested, then only Messages within that participant's existing visible-from/through intervals and current Block, retention, deletion, evidence, Community/Space, and privacy policy are returned; ending creates no public access, new unread state, receipt, presence, or notification disclosure.
3. Given Message retention, participant deletion, or a Report applies after an interval ends, when lifecycle processing runs, then ordinary data is tombstoned, erased, or retained under the same declared policy, protected evidence remains Case-bound, and ending neither accelerates nor defeats a legal hold or eligible safety path.
4. Given a direct interval ended through leave, withdrawal, decline, Block, eligibility, scope, rules, contact-policy, or Moderation loss, when eligibility later permits renewed interaction, then the pair requires a fresh unique Message request and recipient acceptance to create one new interval in the same unique Conversation with a new `visibleFrom`, without automatically restoring excluded gaps, prior Deliveries, receipts, or participation.
5. Given interval end, Message send, request acceptance, Block, re-entry, retention, or retry races, when processing commits, then one versioned order and stable operation result prevail, a Message cannot commit after its interval boundary, stale consent cannot reopen access, and reconciliation derives client state from the durable Conversation and interval history.

### States and accessibility

Show active, ending, ended interval, history-available, history-limited,
fresh-request-required, new interval, conflict, and unavailable states with clear consequences and
accessible confirmation.

### Verification

- Domain/unit: unique direct interval, automatic scope-loss boundary, history, re-entry, and race-order rules.
- Integration/contract: participant or Space-archive end versus send/request, executor no-read, notice, queues, history, Block/unblock, and fresh consent.
- Component: close/end confirmation, history-only, request-required, conflict, and terminal states.
- Journey/live: end a direct interval, Block/unblock, and require fresh acceptance before re-entry.
- Manual/rendered: consent/closure wording, keyboard, screen reader, focus, reflow, and privacy review.
