# NOTF — Notifications and delivery behavior

## L2-NOTF-001: Create Notifications after commit

**Traces to:** L1-NOTF-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Notification creation consumes stable committed events and records at most one recipient/event/type
combination before any external Delivery is attempted.

### Acceptance criteria

1. Given an eligible committed product event, when Notification processing runs, then one durable Notification per eligible recipient and type is created with a stable target reference.
2. Given the source transaction rolls back or fails, when processing completes, then no Notification or Delivery exists for that event.
3. Given the same event is replayed, when processed, then existing Notifications are reused and duplicate user-visible or external effects are not created.
4. Given event data is malformed or unsupported, when processing fails, then it is quarantined for operators without exposing payload content to users.

### States and accessibility

New Notifications update badges and lists without stealing focus; announce a useful aggregate change
rather than every event and provide a refresh path when processing is delayed.

### Verification

- Domain/unit: recipient/type uniqueness and supported-event mapping.
- Integration/contract: committed outbox, rollback, replay, and malformed-event behavior.
- Component: duplicate-safe badge/list update and delayed state.
- Journey/live: commit an activity and verify exactly one Notification and no pre-commit Delivery.
- Manual/rendered: focus stability, announcement volume, and reduced-motion review.

## L2-NOTF-002: Recheck recipient eligibility

**Traces to:** L1-NOTF-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Current Membership, visibility, Block, Moderation Action, Account, and target lifecycle state are
evaluated before inbox disclosure, external Delivery, and deep-link resolution.

### Acceptance criteria

1. Given a recipient remains eligible, when a Notification is listed or delivered, then only the minimum permitted content and target are presented.
2. Given access, Membership, or target visibility changed after creation, when listing or Delivery runs, then private content is suppressed and no external Delivery leaks it.
3. Given an applicable Block or Moderation Action exists, when Notification creation, listing, or Delivery is evaluated, then prohibited interaction is absent across every channel.
4. Given a target was removed, deleted, or retained only as restricted evidence, when its deep link is followed, then the recipient receives a safe unavailable result rather than hidden details.

### States and accessibility

Show available, access-changed, target-unavailable, and retry states with safe language and a useful
next action; do not reveal which private policy caused suppression.

### Verification

- Domain/unit: eligibility matrix across Membership, visibility, Block, action, and deletion state.
- Integration/contract: list, Delivery, deep-link, and post-creation revocation behavior.
- Component: unavailable target and access-changed presentation.
- Journey/live: create a Notification, revoke target access, and prove inbox, email, and link suppression.
- Manual/rendered: privacy copy, keyboard navigation, and screen-reader context review.

## L2-NOTF-003: Maintain a usable Notification inbox

**Traces to:** L1-NOTF-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The inbox provides stable cursor pagination, authoritative unread state, bounded aggregation, and
safe deep links without using a realtime badge as the source of truth.

### Acceptance criteria

1. Given an authenticated Account, when inbox pages are requested, then accessible Notifications are returned in stable order without duplicates or skipped items.
2. Given an unread Notification or current page is marked read, when the command commits, then the unread boundary advances idempotently and the server returns the authoritative count.
3. Given concurrent activity or duplicate realtime hints, when the inbox refreshes, then counts and items reconcile from the API without regression.
4. Given an invalid cursor or another Account's Notification identifier, when requested, then a typed safe failure reveals no Notification existence.

### States and accessibility

Show loading, empty, unread, load-more, marking, access-changed, and retry states; preserve focus and
reading position, expose semantic list structure, and provide text beyond badge color.

### Verification

- Domain/unit: ordering, pagination, read boundary, aggregation, and count rules.
- Integration/contract: inbox, read, read-all, invalid cursor, and cross-Account isolation.
- Component: list, badge, load-more, empty, and retry behavior.
- Journey/live: receive several Notifications, page them, and reconcile read state across sessions.
- Manual/rendered: keyboard, screen-reader list, zoom, and high-contrast review.

## L2-NOTF-004: Apply Notification preferences

**Traces to:** L1-NOTF-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

An Account can control optional Notifications by event type, Community, and configured channel,
while mandatory security, billing, safety, or service notices remain narrowly classified.

### Acceptance criteria

1. Given an authenticated Account changes an allowed optional preference for an exact event type, Community scope, and configured channel against its current version, when the operation commits, then the value, preference version, and unsubscribe-action epoch advance atomically and future Delivery decisions use only that committed value.
2. Given a notice is classified mandatory, when preferences are viewed or changed, then its purpose and non-optional status are explicit and the user cannot be misled that it is disabled.
3. Given a preference change races an unsubscribe action, queued work, or an operation retry, when either mutation commits or Delivery begins, then one versioned order wins, an action from an earlier unsubscribe-action epoch cannot overwrite the newer choice, replay returns the durable operation result without rolling state back, and Delivery reauthorizes against the current preference before any provider call.
4. Given an unauthorized actor or unknown Community scope, when preferences are changed, then no preference changes and private configuration is not disclosed.

### States and accessibility

Show loading, saved, unsaved, validation, conflict, and unavailable states; group controls with clear
channel/type labels, preserve focus, and announce save results.

### Verification

- Domain/unit: preference precedence, version and unsubscribe-action epoch, mandatory classification, and current-value rules.
- Integration/contract: read/update, token-versus-preference race, queued-work reevaluation, and authorization contracts.
- Component: grouped preference controls, saved/conflict, and mandatory explanations.
- Journey/live: change and re-enable one scoped preference against an old unsubscribe action and queued Delivery.
- Manual/rendered: keyboard groups, screen-reader labels, and cognitive-copy review.

## L2-NOTF-005: Aggregate digests and respect quiet time

**Traces to:** L1-NOTF-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Optional Notifications may be aggregated and delayed using the Account's declared locale and time
zone without delaying urgent mandatory notices or duplicating events across windows.

### Acceptance criteria

1. Given digest delivery is selected, when eligible events occur within a window, then each appears at most once in one bounded digest and unnecessary individual Deliveries are not sent.
2. Given quiet time applies, when optional Delivery becomes due, then it is deferred to the next valid local time with daylight-saving transitions handled deterministically.
3. Given an urgent mandatory notice, when it is committed, then digest and quiet-time settings do not delay it beyond its declared service rule.
4. Given preference, access, or target state changes before dispatch, when the digest is rendered, then ineligible entries are removed and no private summary is leaked.

### States and accessibility

Preference UI explains time zone, next expected window, urgent exceptions, and empty digest behavior
in localized text; time controls and summaries are keyboard and screen-reader usable.

### Verification

- Domain/unit: window membership, deduplication, urgency, time-zone, and DST rules.
- Integration/contract: scheduler, delayed work, changed eligibility, and digest composition.
- Component: schedule controls, next-window explanation, and validation states.
- Journey/live: aggregate events across a quiet-time and DST boundary without duplication.
- Manual/rendered: localized time, keyboard controls, long translation, and screen-reader review.

2. Given user-authored text is included, when rendered for web or an external channel, then it is bounded, safely encoded, and omitted from subjects or lock-screen-like previews when privacy policy requires.
3. Given a template or translation key is missing, when rendering runs, then Delivery fails safely to monitored quarantine rather than sending placeholders or a mixed unsafe template.
4. Given content is viewed with assistive technology, when controls and status are encountered, then names, order, link purpose, and language metadata are meaningful.

### States and accessibility

link text, plain-text alternatives, and non-color-only status across supported channels.

- Component: localized inbox content, RTL, long text, and safe previews.
- Journey/live: change locale and privacy settings, then inspect web and external Delivery.
- Manual/rendered: screen-reader, plain-text, RTL, zoom, and email-client review.

## L2-NOTF-007: Attempt each Delivery idempotently

**Traces to:** L1-NOTF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Each Notification/channel destination has one durable Delivery lifecycle, a stable attempt identity,
and a certified provider duplicate-prevention or reconciliation mechanism, with recipient eligibility
checked at the attempt boundary.

### Acceptance criteria

1. Given an eligible queued Delivery and configured provider, when dispatched, then one bounded attempt is recorded before or atomically with the provider call and reuses one stable provider attempt identity for that Delivery and destination.
2. Given a provider supports idempotency, when an attempt or response is retried, then the same provider idempotency key is used and the provider's previously accepted result is reconciled rather than creating a second user-visible send.
3. Given a provider instead supplies an authoritative status query or signed callback, when a timeout leaves the outcome unknown, then the Delivery enters ambiguous status and no retry occurs until that authority proves not-sent, sent, or terminal failure; an outcome that cannot be proved within the bounded reconciliation window becomes operator-visible terminal work rather than a blind retry.
4. Given current preference, access, suppression, or destination validity forbids sending, when dispatch begins, then the Delivery becomes suppressed and the provider is not called.
5. Given an immediate response, delayed callback, or reconciliation result arrives, when state changes, then its provider identity and attempt version must match, sent is recorded only from authoritative acceptance, and late or duplicate results have no additional effect.

### States and accessibility

User-visible status distinguishes scheduled, sent, suppressed, delayed, and failed only where useful;
operator views expose attempts without showing secrets or full private payloads.

### Verification

- Domain/unit: Delivery lifecycle, eligibility boundary, and idempotency identity.
- Integration/contract: provider call, retry, timeout ambiguity, suppression, and duplicate behavior.
- Component: appropriate member status and redacted operator attempt view.
- Journey/live: interrupt an attempt and prove recovery produces at most one external message.
- Manual/rendered: status terminology, privacy redaction, and screen-reader table review.

## L2-NOTF-008: Retry and quarantine terminal failures

**Traces to:** L1-NOTF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Transient Delivery failures use bounded backoff and terminal failures enter a visible dead-letter
state with safe operator retry or dismissal controls.

### Acceptance criteria

1. Given a classified transient failure, when an attempt fails, then bounded exponential backoff with jitter schedules the next attempt without blocking other recipients.
2. Given a permanent failure or retry budget exhaustion, when classified, then the Delivery becomes terminal, records a sanitized reason, and raises the configured operational signal.
3. Given an authorized operator repairs the cause and requests replay, when reevaluation passes, then the same Delivery resumes idempotently with an Audit Event.
4. Given replay is no longer eligible because preferences, access, or retention changed, when requested, then it is rejected or suppressed and no provider call occurs.

### States and accessibility

Operator views show next attempt, bounded reason, terminal status, replay confirmation, and outcome;
tables, filters, and destructive controls remain keyboard and screen-reader usable.

### Verification

- Domain/unit: error classification, backoff, retry budget, and replay eligibility.
- Integration/contract: transient/permanent provider fakes, terminal queue, alert, and replay.
- Component: attempt history, filters, confirmation, and status states.
- Journey/live: exhaust retries, repair provider state, and replay without duplicate Delivery.
- Manual/rendered: keyboard operations, time presentation, redaction, and announcement review.

## L2-NOTF-009: Process unsubscribe and provider feedback

**Traces to:** L1-NOTF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Verified unsubscribe actions and provider bounce, complaint, or delivery callbacks update channel
eligibility idempotently without allowing forged suppression or Account enumeration.

### Acceptance criteria

1. Given an optional Delivery exposes a no-sign-in unsubscribe action, when the action is issued, then it is bound to the exact Account, stable destination identity and generation/version, event type, Community scope, configured channel, exact preference identity/version and unsubscribe-action epoch, purpose, expiry, and unguessable operation identity without exposing the secret in logs or analytics.
2. Given a presented scoped unsubscribe action, when its unguessable operation identity is reconciled first, then a matching already-committed operation returns the same privacy-safe result, while an uncommitted valid action must atomically recheck destination ownership and generation/version plus the exact preference version and unsubscribe-action epoch before only that bound optional preference is suppressed, its version and epoch advance, and the durable operation result is recorded.
3. Given an invalid, expired, altered, consumed-for-a-different-result, or stale action whose destination was replaced, revoked, reassigned, or rebound after issuance, when used, then no preference or new destination changes and the response reveals no Account or destination details.
4. Given a verified provider bounce or complaint callback, when processed, then current destination ownership/version is rechecked and destination suppression and Delivery status change once even if callbacks are duplicated or out of order; stale feedback cannot suppress a replacement destination.
5. Given a callback signature, timestamp, destination identity/version, ownership, or schema is invalid, when received, then it is rejected, audited safely, and changes no Notification, preference, or suppression state.

### States and accessibility

Unsubscribe pages provide clear scope, confirmed/already-applied, invalid, and recovery states without
requiring sign-in when the secure action suffices; results are focused and announced.

### Verification

- Domain/unit: action binding and epoch, unsubscribe scope, suppression, callback ordering, and idempotency rules.
- Integration/contract: signed callbacks, stale action replay, destination replacement, bounce, and complaint contracts.
- Component: confirmation, applied, invalid, and preference recovery states.
- Journey/live: unsubscribe, replace a destination against an old action, and process duplicate provider callbacks.
- Manual/rendered: keyboard, screen-reader, no-JavaScript, and privacy-copy review.

## L2-NOTF-010: Render safe templates and action links

**Traces to:** L1-NOTF-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Versioned templates accept only declared encoded variables and produce canonical action links that
cannot become open redirects, reusable secrets, or accidental disclosures.

### Acceptance criteria

1. Given a valid template version and declared data, when rendered, then HTML and plain-text forms contain only safely encoded values and configured canonical origins.
2. Given undeclared markup, missing required data, unsafe URL, or unknown template version, when rendering is attempted, then no Delivery is sent and the failure is monitored.
3. Given an action requires a token, when issued, then it is scoped, time-bounded, revocable or single-use as appropriate, and excluded from logs and analytics.
4. Given a link target becomes unauthorized or unavailable, when followed, then authentication and current authorization are enforced before private content is shown; an eligible Appeal may enter only the restricted remediation session, while other actions cannot bypass banned, deactivated, or deletion-pending Account state.

### States and accessibility

Templates use semantic headings, descriptive links, accessible names, declared language, readable
plain text, sufficient contrast, resilient layout, and honest unavailable-link results.

### Verification

- Domain/unit: template variable, link scope, expiry, and canonical-origin rules.
- Integration/contract: renderer, unsafe input, token replay, and target authorization.
- Component: web destination and unavailable-link states.
- Journey/live: send a real test Delivery and exercise valid, expired, and revoked links.
- Manual/rendered: representative email clients, plain text, screen reader, zoom, and dark-mode review.

## L2-NOTF-011: Observe and repair Delivery health

**Traces to:** L1-NOTF-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Operators can detect queue age, latency, failure, suppression, provider degradation, and template
errors through privacy-safe metrics, traces, alerts, and runbooks.

### Acceptance criteria

1. Given Delivery processing runs, when outcomes occur, then metrics and structured events record channel, template version, status, latency, and correlation without destination or private content.
2. Given queue age, failure rate, callback rejection, or provider latency crosses an accepted threshold, when evaluated, then an owned alert links to a current runbook.
3. Given an operator investigates one Delivery, when authorized, then they can correlate Notification, attempts, callback, and outcome using redacted data and Audit Events.
4. Given a provider outage, when degraded operation is invoked, then affected work is paused, deferred, or rerouted only under documented policy and durable Notifications remain intact.

### States and accessibility

Operational dashboards expose freshness, units, thresholds, ownership, and degraded status; charts
have textual equivalents and tables do not reveal destinations or payloads.

### Verification

- Domain/unit: metric classification and privacy-redaction rules.
- Integration/contract: telemetry, alert thresholds, correlation, and provider-outage controls.
- Component: dashboard tables, textual chart alternatives, and investigation view.
- Journey/live: simulate provider degradation and follow alert, pause, recovery, and backlog drain.
- Manual/rendered: runbook accuracy, keyboard dashboard, and sensitive-data inspection.

## L2-NOTF-012: Disable unconfigured channels cleanly

**Traces to:** L1-NOTF-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

An external channel remains inert until its provider, sender identity, callback verification,
templates, privacy purpose, operational ownership, and ambiguous-outcome safety pass startup and
deployment validation.

### Acceptance criteria

1. Given a channel is disabled or incompletely configured, when the service starts, then it does not register workers or callbacks, request credentials, show selectable preferences, or claim Delivery support.
2. Given an event would normally use a disabled channel, when processed, then durable in-app behavior remains correct and no external Delivery or retry backlog is created.
3. Given production contains placeholder sender identity, callback secret, or development provider, when startup validation runs, then deployment fails before traffic is accepted.
4. Given a channel is enabled through an approved change, when promoted, then template, callback, unsubscribe, accessibility, privacy, and live Delivery gates pass first.
5. Given provider capabilities are certified for a channel, when enablement is evaluated, then the provider must either honor a stable idempotency key with a stable accepted result or expose an authoritative status query or signed callback that resolves timeouts before retry; a provider with neither mechanism cannot be enabled.

### States and accessibility

Members see only available channel choices; direct or stale links explain unavailability accessibly,
and operators receive explicit configuration status without exposed secret values.

### Verification

- Domain/unit: channel availability and fallback rules.
- Integration/contract: worker/callback absence, no queued work, and startup validation.
- Component: preference removal and stale-link unavailable state.
- Journey/live: deploy with each channel disabled and enabled, proving the correct boundary.
- Manual/rendered: truthful copy, operator configuration status, and keyboard review.

## L2-NOTF-013: Verify and retire external destinations

**Traces to:** L1-NOTF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Email addresses, telephone numbers, and push registrations become Delivery destinations only through
proof of control and retain an explicit replace, revoke, suppress, and expiry lifecycle.

### Acceptance criteria

1. Given an Account proposes a new email address, telephone number, or push registration, when created, then it remains unverified and receives no private Notification until a purpose-bound, destination-bound, time-bounded challenge proves control under rate and enumeration protections.
2. Given a valid challenge is confirmed, when the destination activates, then stable destination identity, normalized address or provider token, owner Account, channel, verified time, source, and version are recorded without making the raw destination available to unauthorized viewers.
3. Given a destination is replaced, revoked, signed out, expired, reported lost, or invalidated by verified provider feedback, when the transition commits, then its version and the affected preference/destination unsubscribe-action epoch advance atomically before queued Deliveries reauthorize, outstanding actions for the old binding become stale, prohibited sends are suppressed, provider registrations are removed where supported, and repeated transitions have no extra effect.
4. Given the same address or device token is proposed for another Account or provider reports token reassignment, when ownership is evaluated, then the accepted channel-specific uniqueness and recovery policy advances the old binding's unsubscribe-action epoch, prevents one Account from receiving another's private Notification, prevents an old action from suppressing the new owner's preference, and requires fresh proof where ownership is uncertain.
5. Given an Account deactivates, enters deletion, or changes a primary destination, when lifecycle processing runs, then mandatory notices use only currently permitted verified routes, obsolete destinations are minimized under retention policy, and at least one safe recovery path remains when policy requires it.

### States and accessibility

Show unverified, challenge-sent, verified, primary, replacement-pending, suppressed, revoked, expired,
and unavailable states with masked values, resend bounds, alternative routes, and accessible status.

### Verification

- Domain/unit: challenge, ownership, uniqueness, action-epoch advance, primary, replace, revoke, suppression, and expiry rules.
- Integration/contract: email, telephone, push, provider reassignment, queued-work, and lifecycle events.
- Component: add, masked review, verify, resend, replace, revoke, and recovery states.
- Journey/live: verify each channel, replace it during queued work and an old unsubscribe action, then simulate token reassignment.
- Manual/rendered: masked-value clarity, keyboard, screen reader, autofill, and recovery review.
