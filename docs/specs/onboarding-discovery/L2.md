# ONBD — Onboarding and discovery behavior

## L2-ONBD-001: Start and resume onboarding

**Traces to:** L1-ONBD-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The server maintains one versioned onboarding state per Account so progress can resume across
sessions and devices without trusting client-completed flags.

### Acceptance criteria

1. Given an eligible Account has no onboarding state, when onboarding starts, then one state is created with the current required and optional steps.
2. Given saved progress exists, when the Account resumes on another session or device, then the next eligible incomplete step and prior safe selections are returned.
3. Given a client marks an unfulfilled required step complete or submits a stale state version, when progress is saved, then the server rejects the claim, preserves committed progress, and returns current state.
4. Given an optional step is skipped, when progress is saved, then the skip is recorded without fabricating its underlying data and the Account can revisit it where policy permits.

### States and accessibility

Show starting, saved, resume, optional/skipped, stale, offline/retry, and unavailable states; expose
step name and progress in text, retain focus, and avoid time-limited completion.

### Verification

- Domain/unit: one-state, step eligibility, skip, completion, and version rules.
- Integration/contract: start, resume, cross-device save, forged completion, and stale update.
- Component: progress, resume, skipped, saving, and conflict states.
- Journey/live: begin on one session and finish from another without lost selections.
- Manual/rendered: keyboard, screen-reader progress, zoom, and reduced-motion review.

## L2-ONBD-002: Complete required steps

**Traces to:** L1-ONBD-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Onboarding completes only when the server verifies every currently required Account, Profile,
eligibility, and policy condition; optional discovery choices do not become hidden requirements.

### Acceptance criteria

1. Given all required conditions are satisfied, when completion is requested, then the onboarding state becomes complete once and the server returns the permitted destination.
2. Given any required condition is missing, expired, or invalid, when completion is requested, then the server returns the incomplete steps and leaves onboarding incomplete.
3. Given requirements change after the Account begins, when state is refreshed, then newly applicable requirements are shown before completion without discarding valid prior progress.
4. Given completion is retried, when it already committed, then the same complete result is returned without duplicating Profile, Membership, Follow, or preference state.

### States and accessibility

Show requirements, optional choices, validating, incomplete-with-links, complete, and retry states;
use a semantic step list, clear heading focus, and an announced completion result.

### Verification

- Domain/unit: required/optional, changed-requirement, and idempotent completion rules.
- Integration/contract: complete, incomplete, requirement refresh, and replay cases.
- Component: step list, missing-step navigation, validating, and completion states.
- Journey/live: satisfy requirements, fail once with a missing condition, then complete.
- Manual/rendered: keyboard order, screen-reader progress, reflow, and completion announcement.

## L2-ONBD-003: Continue from an invitation

**Traces to:** L1-ONBD-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

A Community invitation survives necessary sign-in, registration, verification, and onboarding steps
without disclosing private Community details before the intended Account is authorized.

### Acceptance criteria

1. Given a valid invitation reaches a signed-out visitor, when sign-in or registration is required, then a protected continuation reference is preserved without placing secret proof in analytics or unsafe redirects.
2. Given the intended Account completes required Account steps, when continuation resumes, then the current invitation is revalidated before any private Community preview or Membership activation.
3. Given the invitation is invalid, expired, revoked, used, or intended for another Account, when continuation resumes, then no Membership changes and a safe unavailable result is shown.
4. Given the invitation is accepted successfully, when onboarding continues, then the new Membership and Community context are reflected without accepting the invitation twice.

### States and accessibility

Show generic invitation received, authentication required, onboarding required, validating, invalid,
accepted, and continuation states; do not expose private names early and keep focus on the next action.

### Verification

- Domain/unit: intended-Account, continuation, expiry, single-use, and disclosure rules.
- Integration/contract: signed-out continuation, registration, revalidation, invalid, and acceptance.
- Component: generic, auth/onboarding handoff, invalid, and accepted states.
- Journey/live: open an invitation signed out, register, verify, and accept as intended.
- Manual/rendered: copied-link, keyboard, screen-reader, and privacy-copy review.

## L2-ONBD-004: Discover eligible Communities

**Traces to:** L1-ONBD-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Community discovery returns a bounded, paged set of active, discoverable Communities the current
Account may know about, with honest context for each suggestion.

### Acceptance criteria

1. Given an eligible Account, when Community suggestions are requested, then only active Communities permitted by visibility, geography/age policy where configured, Account state, and safety rules are returned.
2. Given a Community is private, unlisted, archived, full under a no-waitlist policy, or otherwise ineligible, when suggestions are built, then it is omitted unless a valid invitation or explicit policy permits disclosure.
3. Given suggestions are paged, when a valid cursor is used, then results remain Community-safe, bounded, and free of duplicates despite ordinary concurrent changes.
4. Given a client changes hidden ranking or eligibility fields, when suggestions are requested, then the server ignores unauthorized signals and applies current policy.

### States and accessibility

Show loading skeletons without false content, suggestions with textual reasons, empty, end-of-list,
refresh, and error states; cards use headings/links and work at narrow widths.

### Verification

- Domain/unit: discoverability, eligibility, suggestion-reason, and cursor rules.
- Integration/contract: public/private/unlisted/archived/full filters and paging isolation.
- Component: suggestion cards, loading, empty, paging, and error states.
- Journey/live: compare discovery for Accounts with different invitation and eligibility contexts.
- Manual/rendered: keyboard cards, heading order, reflow, and reason-text review.

## L2-ONBD-005: Discover Profiles and Tags

**Traces to:** L1-ONBD-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Profile and Tag suggestions use authorized public or shared-Community context and never reveal
private Profile fields, hidden Memberships, or safety relationships.

### Acceptance criteria

1. Given Profile suggestions are requested, when results are built, then each Profile is eligible under visibility, shared Community disclosure, Account status, Follow, Block, and recommendation policy.
2. Given Tag suggestions are requested, when results are built, then each Tag comes from content the Account may access and aggregate signals meet the minimum privacy threshold.
3. Given a suggested Profile or Tag becomes hidden or ineligible, when it is selected or refreshed, then the server returns safe unavailable state, changes no Profile, Follow, or Tag state, and grants no access from the stale suggestion.
4. Given there is insufficient eligible data, when suggestions are requested, then an honest empty result is returned rather than invented Profiles, Tags, counts, or activity.

### States and accessibility

Show Profile and Tag sections, reason text, follow state, loading, empty, stale/unavailable, and error;
use semantic lists, labeled actions, accessible fallback imagery, and no auto-moving carousel.

### Verification

- Domain/unit: Profile/Tag eligibility, privacy threshold, and stale-selection rules.
- Integration/contract: shared/no-shared Community, blocked, private, sparse-data, and stale cases.
- Component: suggestion lists, Follow actions, fallback image, empty, and stale states.
- Journey/live: discover a Profile and Tag, then tighten access and verify disappearance.
- Manual/rendered: keyboard lists, screen-reader labels, reflow, and motion review.

## L2-ONBD-006: Control recommendation signals

**Traces to:** L1-ONBD-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

An Account can understand and control declared optional personalization inputs, while essential
eligibility and safety rules remain enforced and sensitive attributes are not inferred for ranking.

### Acceptance criteria

1. Given a suggestion is personalized, when it is displayed, then a concise reason category and a path to relevant controls are available without exposing another Account's private activity.
2. Given the Account disables an optional signal or personalization, when preferences commit, then future suggestions stop using that signal while required safety and access filtering remains active.
3. Given an Account clears declared discovery history, when deletion completes, then retained recommendation inputs follow policy and the action does not delete unrelated Community content or Memberships.
4. Given a client claims interests or eligibility that conflict with authoritative state, when ranking occurs, then the server rejects those values as access proof, treats them only as preference input, and changes no protected state.

### States and accessibility

Show explanation, preference on/off, saving, clear-history impact/confirmation, success, and error;
controls use descriptive labels, not dark patterns, and changed behavior is announced once.

### Verification

- Domain/unit: optional-signal, essential-filter, clearing, and trust-boundary rules.
- Integration/contract: explanation, preference change, history clear, and forged eligibility cases.
- Component: reason disclosure, controls, confirmation, save, and error states.
- Journey/live: disable personalization and compare subsequent eligible suggestions.
- Manual/rendered: keyboard switches, screen-reader state, plain language, and consent review.

## L2-ONBD-007: Preview a Community safely

**Traces to:** L1-ONBD-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

A Community preview shows only the identity, rules, activity summary, and sample content allowed for
the current Account before Membership, with no client-only concealment of member data.

### Acceptance criteria

1. Given a discoverable Community, when an eligible non-member opens its preview, then only fields and Posts explicitly permitted by preview policy are returned.
2. Given a private or invitation-only Community without valid invitation context, when preview is requested, then the response does not confirm private details beyond the declared disclosure policy.
3. Given preview content is removed, archived, or becomes restricted, when a stale preview link is opened, then unavailable state is returned and no cached identifier grants member access.
4. Given the Account is already an active or pending member, when preview loads, then the current Membership state and correct next action are shown rather than a misleading join action.

### States and accessibility

Show loading, public preview, invited preview, joined, pending, restricted, stale, and unavailable;
use semantic content structure, label sample content, and make rules reachable by keyboard.

### Verification

- Domain/unit: preview field/content policy and Membership-state action rules.
- Integration/contract: public, private, invited, member, pending, and stale preview cases.
- Component: preview composition and joined/pending/restricted states.
- Journey/live: preview the same Community as anonymous, invited, pending, and active contexts.
- Manual/rendered: headings, keyboard, screen-reader labels, reflow, and privacy-copy review.

## L2-ONBD-008: Enter the Membership path

**Traces to:** L1-ONBD-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Onboarding routes an Account to direct join, gated request, invitation acceptance, or safe
unavailability from the Community's current server-owned Membership policy.

### Acceptance criteria

1. Given an eligible Account and open Community, when the onboarding join action commits, then one active Membership is created and onboarding reflects it.
2. Given a gated Community, when valid requested information is submitted, then one pending Membership request is created and onboarding shows its status without treating it as active.
3. Given valid invitation context, when acceptance commits, then the intended Account receives one compatible active Membership and the invitation cannot be reused.
4. Given policy, capacity, suspension, Block, or Community lifecycle makes entry ineligible, when entry is attempted, then no Membership changes and the safe next option or unavailability is returned.

### States and accessibility

Show join, request form, invited accept/decline, pending, full, restricted, joined, and error states;
actions have distinct names, form errors associate correctly, and transitions are announced.

### Verification

- Domain/unit: path selection, eligibility, capacity, and single-Membership rules.
- Integration/contract: join, request, invitation, duplicate, suspended, full, and lifecycle cases.
- Component: each entry path, pending, joined, and unavailable state.
- Journey/live: exercise open, gated, and invited Communities with separate Accounts.
- Manual/rendered: keyboard forms, screen-reader action distinction, and narrow-screen review.

## L2-ONBD-009: Establish the first Feed

**Traces to:** L1-ONBD-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

After onboarding, an Account receives an honest first Feed or explicit next-step state composed only
from current Membership, Follow, Tag, Block, Mute, and content-visibility rules.

### Acceptance criteria

1. Given onboarding completes with at least one active Membership or eligible Follow/Tag choice, when the Feed opens, then it contains only authorized Posts and Event occurrences with the canonical mixed-item ordering, stable pagination, type labels, and declared context.
2. Given no eligible Post or Event occurrence exists, when the Feed opens, then a truthful empty state offers relevant Community discovery or creation actions without fabricating activity; an Event-only Feed is populated rather than empty.
3. Given a chosen Community, Follow, Tag, Post, or Event becomes unavailable before Feed load, when results are built, then the item is omitted and no stale onboarding choice grants access.
4. Given onboarding completion is retried, when Feed initialization already exists, then preferences are not duplicated or reset.

### States and accessibility

Show initializing, populated, empty with next actions, access-changed, refresh, and error states;
Feed landmarks and headings are clear, focus is not stolen by new content, and loading is announced.

### Verification

- Domain/unit: mixed-item seed context, eligibility, empty-state, and idempotent initialization rules.
- Integration/contract: Post-only, Event-only, mixed, sparse, stale choice, Block/Mute, and replay cases.
- Component: Feed loading, first items, empty next actions, and error states.
- Journey/live: complete onboarding into Post-only, Event-only, mixed, and genuinely empty first Feeds.
- Manual/rendered: keyboard Feed, screen-reader landmarks, reflow, and live-update review.

## L2-ONBD-010: Record eligibility and consent

**Traces to:** L1-ONBD-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Required eligibility, terms, privacy, and optional consent acknowledgements are versioned,
server-validated, separable, and revisitable without coercing optional personalization.

### Acceptance criteria

1. Given current required policy versions, when an eligible Account explicitly accepts them, then the versions, Account, time, and lawful evidence required by policy are recorded before completion.
2. Given a required acknowledgement is absent or eligibility fails, when completion or protected access is requested, then the server rejects it and preserves prior valid onboarding progress.
3. Given optional personalization or marketing consent is declined, when onboarding continues, then required product access remains available unless a documented lawful dependency applies.
4. Given a material policy version changes, when the Account next enters the applicable surface, then fresh acknowledgement is requested without silently rewriting prior evidence.

### States and accessibility

Show required versus optional choices, policy links, validation, declined optional, changed-policy, and
saved states; choices default safely, have full labels, and never use preselected deceptive consent.

### Verification

- Domain/unit: version, required/optional, eligibility, and re-acknowledgement rules.
- Integration/contract: accept, missing required, decline optional, policy change, and evidence scope.
- Component: consent groups, policy links, validation, and changed-policy states.
- Journey/live: decline optional consent, complete onboarding, then acknowledge a new required version.
- Manual/rendered: keyboard groups, screen-reader labels, zoom, and plain-language review.

## L2-ONBD-011: Exclude unsafe suggestions

**Traces to:** L1-ONBD-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every onboarding suggestion and preview is filtered at response time by Block, Mute where applicable,
Account eligibility, Community visibility, Membership state, moderation state, and privacy policy.

### Acceptance criteria

1. Given a Block exists in either direction, when Profile or authored-content suggestions are built, then the blocked relationship is omitted according to safety policy and no Block owner is disclosed.
2. Given the Account has Muted another Account, when personalized Feed-like suggestions are built, then the muted Account's eligible content is suppressed without changing Community Membership.
3. Given a Community, Profile, Post, or Tag is removed, restricted, unsafe, or inaccessible, when it was previously suggested, then refresh and selection return safe unavailable state and no hidden data.
4. Given recommendation caches or indexes contain an ineligible item, when a response is served, then authoritative filtering removes it even before asynchronous cleanup completes.

### States and accessibility

Do not show gaps that disclose filtered identities; use generic stale/unavailable recovery where a
selected item vanished, preserve focus, and avoid noisy announcements during list filtering.

### Verification

- Domain/unit: complete safety-filter precedence and privacy-safe unavailable rules.
- Integration/contract: Block, Mute, moderation, visibility change, and stale-index cases.
- Component: filtered-list continuity and selected-item unavailable state.
- Journey/live: suggest, then Block/restrict/remove, and verify every onboarding surface.
- Manual/rendered: privacy copy, keyboard focus continuity, and announcement-noise review.

## L2-ONBD-012: Recover from empty and error states

**Traces to:** L1-ONBD-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Onboarding and discovery distinguish honest emptiness, restricted availability, transient failure,
and offline state, preserving committed progress and providing a safe next action.

### Acceptance criteria

1. Given no eligible suggestions exist, when discovery completes successfully, then an empty result identifies useful alternatives without implying an error or inventing content.
2. Given a transient dependency or network failure, when a step or suggestion request fails, then committed progress remains intact and a bounded retry is available without duplicate state.
3. Given an unauthorized or expired session, when a protected onboarding request is made, then the server rejects it, no progress is lost, secrets are not placed in continuation URLs, and reauthentication returns to the safe step.
4. Given repeated retry cannot succeed, when fallback is shown, then support or defer/skip options appear only where policy permits and required steps remain incomplete.

### States and accessibility

Provide distinct loading, empty, restricted, offline, session-expired, transient-error, retrying, and
support states; errors receive heading focus, retries are labeled, and progress is not motion-only.

### Verification

- Domain/unit: progress preservation, retry idempotency, and required-step fallback rules.
- Integration/contract: empty, dependency failure, duplicate retry, expired session, and continuation.
- Component: every loading/empty/error/retry/support state.
- Journey/live: interrupt network and session across save, discovery, and completion flows.
- Manual/rendered: keyboard recovery, screen-reader error focus, reduced-motion, and reflow review.
