# ARCH — Platform architecture detailed requirements

## L2-ARCH-001: Runtime and repository surfaces are explicit

**Traces to:** L1-ARCH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The starter shall contain `backend/`, `frontend/`, `design-system/`, `marketing/`, `e2e/`, `docs/`, `eng/`, and `infra/` boundaries. Its production runtime shall use an ASP.NET Core API, an Angular member application, an independent static marketing site, relational persistence, and a canonical cross-surface design-system source. Supported stable or LTS toolchains and dependency versions shall be selected and pinned when the product is created; .NET package versions shall be centrally managed and package-manager lockfiles shall be committed.

### Acceptance criteria

1. Given a clean checkout, when the documented setup and build commands run, then the backend, Angular application, static marketing surface, and canonical design-system assets build without requiring globally implicit project state.
2. Given a request for `/`, when the initial production artifact serves it, then static marketing owns the response and the Angular application bundle is not required to render useful content.
3. Given a member application route or `/api` request, when it is served, then ownership is unambiguous between Angular routing and ASP.NET Core endpoints.
4. Given toolchain or package restoration, when versions are resolved, then pinned SDK, package-manager, package, and lockfile inputs determine the result.

### States and accessibility

All user-facing surfaces include their applicable loading, empty, failure, unauthorized, narrow-screen, keyboard, and reduced-motion behavior; the static public surface remains useful without client JavaScript.

### Verification

- Domain/unit: Verify repository-boundary policy helpers or architecture tests where implemented.
- Integration/contract: Build each surface from a clean checkout and inspect API/static route ownership.
- Component: Verify the application shell consumes canonical configuration and design assets.
- Journey/live: Smoke-test `/`, an application deep link, and an API endpoint from the published artifact.
- Manual/rendered: Inspect the repository map and rendered public/member surfaces.

## L2-ARCH-002: Backend dependencies point inward

**Traces to:** L1-ARCH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The backend shall contain mandatory `Domain`, `Application`, `Infrastructure`, and `Api` projects. Domain shall own community nouns, value objects, lifecycle, transitions, and pure policy without framework dependencies; Application shall coordinate capability use cases and declare ports; Infrastructure shall implement those ports; Api shall own protocol translation and composition. Application shall not depend on Infrastructure, and Domain shall not depend on any outer project.

### Acceptance criteria

1. Given the solution project references, when dependency direction is inspected, then `Application` references `Domain`, `Infrastructure` references `Application` and `Domain`, and `Api` references `Application` plus Infrastructure only for composition.
2. Given Domain source, when its imports and package references are inspected, then ASP.NET Core, EF Core, SignalR, file-system, clock, mail, and vendor SDK types are absent.
3. Given a controller, endpoint, or hub, when its behavior is inspected, then it translates the protocol, dispatches one use case, and maps the result without implementing community policy or directly calling an Infrastructure implementation.
4. Given persistence, identity, time, event, file, or external-service behavior, when Application consumes it, then the dependency is represented by a focused Application-owned abstraction.

### States and accessibility

Architecture boundaries must preserve typed failure information needed for accessible client error, conflict, and authorization states.

### Verification

- Domain/unit: Run architecture dependency tests and focused policy tests without infrastructure.
- Integration/contract: Resolve the composition root and exercise an endpoint through its real adapters.
- Component: Confirm UI error states can consume the typed contracts emitted by the boundary.
- Journey/live: Complete a community mutation through UI, API, policy, and persistence.
- Manual/rendered: Review project references and representative controller/handler code.

## L2-ARCH-003: Angular libraries have stable ownership

**Traces to:** L1-ARCH-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The Angular workspace shall contain an application shell and `api`, `components`, and `domain` libraries. The app owns pages, routing, and composition; `api` owns typed remote contracts, authentication, interceptors, HTTP, and realtime clients; `components` owns deliberate reusable presentation; `domain` owns justified pure shared logic. Libraries shall receive deployment configuration through injection tokens and shall not hard-code origins or read deployment-specific globals.

### Acceptance criteria

1. Given a route page, when dependencies are inspected, then the page composes libraries and coordinates navigation or feature state while reusable components remain transport-free.
2. Given remote communication, when its source is inspected, then typed models and clients reside in the API boundary and the production origin is relative or injected.
3. Given a feature-specific component without a stable reusable contract, when it is added, then it remains local to the feature rather than being prematurely exported.
4. Given a public component or domain helper, when the corresponding library public API is inspected, then only deliberate supported exports are present.

### States and accessibility

Library contracts must carry enough state and semantics for pages to render accessible loading, empty, error, conflict, authorization, and reconnect experiences.

### Verification

- Domain/unit: Test pure client-domain helpers independently.
- Integration/contract: Verify injected API configuration and typed client/server compatibility.
- Component: Test public components and page composition at their ownership boundaries.
- Journey/live: Run a routed member journey using the production-shaped client configuration.
- Manual/rendered: Inspect workspace boundaries and public API exports.

## L2-ARCH-004: Application code is organized by capability and use case

**Traces to:** L1-ARCH-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Application code shall be grouped by community capability and use case rather than technical grab-bag folders. A command changes state, a query does not, and one handler owns one use case. Handlers shall load only required state, consult domain or application policy, persist changes, and notify only after success; validators shall handle request shape and cheap invariants while state-dependent rules remain authoritative policy.

### Acceptance criteria

1. Given a membership, content, moderation, or notification capability, when Application source is inspected, then its commands, queries, handlers, validators, DTOs, mappers, and typed failures are co-located by capability.
2. Given a command handler, when executed, then it loads only the state needed for its single use case and cannot publish success before persistence succeeds.
3. Given a query, when executed, then it produces a read result without changing persisted domain state.
4. Given a state-dependent community rule, when request validation succeeds but current state violates the rule, then domain or application policy rejects it with a typed failure.

### States and accessibility

Use-case results shall distinguish validation, forbidden, unavailable, conflict, and success states so the UI can explain each outcome accessibly.

### Verification

- Domain/unit: Test policies and command/query semantics directly.
- Integration/contract: Exercise handlers with real persistence and expected typed failures.
- Component: Verify pages map each typed outcome to an appropriate state.
- Journey/live: Run accepted and rejected versions of a core community action.
- Manual/rendered: Review capability folder organization and handler responsibilities.

## L2-ARCH-005: State changes follow one authoritative sequence

**Traces to:** L1-ARCH-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every state-changing request shall authenticate the actor, establish Community scope, validate request shape, load required state, ask authoritative policy whether the transition is legal, persist atomically, publish or broadcast only after persistence, and return a typed result. Authorization, workflow transitions, quotas, moderation rules, and validation with business meaning shall never rely on browser enforcement.

### Acceptance criteria

1. Given an unauthenticated or out-of-scope actor, when a mutation is requested, then the server rejects it before changing state and does not trust a submitted community identifier as access proof.
2. Given an authenticated actor whose transition violates policy, when the mutation is requested, then the server returns a stable failure and persisted state remains unchanged.
3. Given a legal mutation, when persistence fails, then no success event or realtime broadcast is published.
4. Given allowed-action fields shown by the UI, when authorization or state changes between render and submission, then the server re-evaluates policy and the client tolerates the resulting conflict.

### States and accessibility

The client may disable unavailable actions for clarity but shall provide readable explanations and shall handle server-side forbidden and conflict responses without losing focus or user context.

### Verification

- Domain/unit: Test allowed and rejected transitions, quotas, and unchanged state.
- Integration/contract: Verify authentication, Community scope, atomic persistence, and event ordering.
- Component: Test locked-action explanations and conflict reconciliation.
- Journey/live: Exercise a stale-client conflict and an unauthorized cross-Community mutation.
- Manual/rendered: Inspect protocol-to-policy sequencing and rendered explanations.

## L2-ARCH-006: HTTP contracts and failures are stable and explicit

**Traces to:** L1-ARCH-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Product APIs shall use a consistent `/api` prefix, plural resource nouns, stable identifiers, explicit request/response DTOs, and dedicated action or subresource routes where a transition would otherwise be ambiguous. Route identifiers are authoritative. Creation shall return `201` with a location when practical, contentless deletion `204`, and failures sanitized Problem Details using `400`, `401`, `403`, `404`, `409`, and only a deliberately adopted `422` policy according to their documented semantics. OpenAPI shall include representative success and error contracts. API versioning shall be introduced only for a real external compatibility contract, not as speculative duplicate controllers.

### Acceptance criteria

1. Given a request whose body identifier differs from its route identifier, when it is handled, then the route identifier governs and the body cannot redirect the operation to another resource.
2. Given malformed input, missing authentication, forbidden scope, unavailable scoped resource, or an illegal current-state transition, when each request is made, then the response respectively uses the documented `400`, `401`, `403`, `404`, or `409` Problem Details contract.
3. Given a resource is created or deleted successfully, when the response is returned, then it uses `201` with a resolvable location where practical or `204` with no body for deletion.
4. Given generated OpenAPI, when it is inspected, then explicit DTOs and representative error schemas match runtime serialization without exposing persistence navigation graphs.
5. Given no real external compatibility contract exists, when API evolution is planned, then the design does not preemptively multiply versioned controllers.

### States and accessibility

Problem contracts shall provide stable, non-sensitive information that clients can associate with fields, announce when appropriate, and distinguish from authorization or conflict states.

### Verification

- Domain/unit: Test status/error mapping functions.
- Integration/contract: Assert routes, DTO serialization, status codes, locations, and OpenAPI schemas.
- Component: Verify field and summary error presentation from representative Problem Details.
- Journey/live: Complete create, forbidden, conflict, and delete flows against a live API.
- Manual/rendered: Review OpenAPI and rendered error associations.

## L2-ARCH-007: Relational schema evolution is forward and rehearsable

**Traces to:** L1-ARCH-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Community relationships and invariants shall use a relational model with explicit constraints, relationships, and indexes where conventions do not communicate intent. Every schema change shall add a reviewed, checked-in, forward migration; an already-applied migration shall not be edited outside a disposable environment. Changes requiring transformation shall include compatibility, backfill, clean-apply, upgrade-rehearsal, and forward-fix or rollback plans.

### Acceptance criteria

1. Given a schema change, when the change is reviewed, then a new forward migration and any required data backfill or compatibility plan are present.
2. Given an existing database at the prior supported schema, when the migration sequence is applied, then it upgrades without manual editing of migration history or loss of required community data.
3. Given a clean database, when all migrations are applied, then its schema, constraints, and indexes match the expected current model.
4. Given an expected relational conflict, when it occurs, then the application translates it into stable API behavior rather than leaking provider diagnostics.

### States and accessibility

Migration or compatibility failures must fail deployment safely and must not expose partial user-facing states as successful.

### Verification

- Domain/unit: Test invariant representations independent of EF Core.
- Integration/contract: Apply migrations to clean and prior-version real-provider databases and exercise constraints.
- Component: Verify stable conflict presentation for expected persistence conflicts.
- Journey/live: Run a critical journey after a rehearsed upgrade.
- Manual/rendered: Review generated migration SQL, backfill, compatibility, and recovery notes.

## L2-ARCH-008: Community-scoped reads are explicit and efficient

**Traces to:** L1-ARCH-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every Community-owned query shall derive and enforce authorized Community scope on the server. Read paths shall project purpose-built DTOs directly, use no-tracking behavior when entities will not be updated, avoid lazy loading and accidental N+1 access, omit unneeded large columns or graphs, and propagate cancellation through HTTP, EF Core, and external I/O.

### Acceptance criteria

1. Given an authenticated Member submits another Community identifier, when a Community-owned read executes, then the server still derives authorization and returns no out-of-scope data.
2. Given a collection endpoint, when its generated database query is inspected, then it selects only fields needed by its DTO and does not lazy-load related graphs.
3. Given a read-only query, when it executes, then change tracking is not enabled unless a documented use case requires it.
4. Given request cancellation, when a database or external read is in progress, then cancellation propagates through the complete request path.

### States and accessibility

Scoped absence shall use the documented unavailable behavior without revealing whether another Community's resource exists; collection clients shall distinguish loading, empty, and error states.

### Verification

- Domain/unit: Test Community-scope and projection helpers where pure.
- Integration/contract: Run cross-Community access tests and inspect real-provider query behavior.
- Component: Test loading, empty, unavailable, and cancellation-adjacent UI behavior.
- Journey/live: Attempt cross-community reads and browse a large collection.
- Manual/rendered: Review representative SQL/query plans and DTO payloads.

## L2-ARCH-009: Time, demonstration data, and lifecycle state are deliberate

**Traces to:** L1-ARCH-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Persisted and event timestamps shall use UTC `DateTimeOffset`, and policy affected by time shall depend on an injected clock. Demonstration data shall be deterministic and enabled only in development or dedicated demo environments. Reversible resource lifecycle states such as Community archive, Account deactivation, suspension, cancellation, or Case/Conversation closure shall remain distinct from irreversible deletion, with retention and deletion policy made explicit.

### Acceptance criteria

1. Given time-dependent membership, invitation, or moderation policy, when it is tested, then an injected deterministic clock controls the result and persisted timestamps are UTC.
2. Given a production configuration, when the service starts, then demonstration seeding is disabled and cannot introduce example accounts or personal data.
3. Given a Community archive, Account deactivation, suspension, cancellation, or defined Case/Conversation closure action, when it completes, then the resource remains distinguishable from a hard-deleted resource according to policy.
4. Given an irreversible deletion path, when it is requested, then retention, authorization, confirmation, and dependent-data effects are explicit rather than inferred from reversible state.

### States and accessibility

Lifecycle state and destructive consequences shall be communicated by text as well as appearance, with accessible confirmation and recovery guidance where recovery exists.

### Verification

- Domain/unit: Test clock-controlled expiration and lifecycle transitions.
- Integration/contract: Verify UTC persistence, environment-gated seeding, and deletion effects.
- Component: Test lifecycle labels and destructive confirmation states.
- Journey/live: Archive and restore a resource, then separately test an authorized deletion path.
- Manual/rendered: Inspect retention policy, seed configuration, and timestamp serialization.

## L2-ARCH-010: C# code is nullable-safe, asynchronous, and convention-enforced

**Traces to:** L1-ARCH-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

C# shall use one public type per matching file, file-scoped namespaces, nullable reference types, implicit usings, `PascalCase` public names, `camelCase` locals and parameters, `_camelCase` private fields, and `I` only for behavioral interfaces. Names shall prefer descriptive product vocabulary over abbreviations, and comments shall explain rationale, constraints, or surprising provider behavior rather than narrate syntax. Immutable messages and DTOs should be records; lifecycle entities should be classes. Constructors, `required`, and valid defaults shall be preferred over null-forgiving operators. I/O shall be asynchronous, `Async`-suffixed where not framework-fixed, and cancellation-aware without blocking waits.

### Acceptance criteria

1. Given backend source, when analyzers and formatting run, then mismatched public type files, nullable warnings, naming violations, and style violations fail the configured quality gate.
2. Given an immutable request, response, command, query, or event message, when modeled, then its construction establishes valid required state without null-forgiving suppression.
3. Given a controller-to-provider I/O path, when inspected, then `CancellationToken` propagates through every async boundary and no `.Result`, `.Wait()`, or thread sleep blocks it.
4. Given a closed public product vocabulary, when modeled, then an enum or value object is used and string serialization is selected where it improves readable, stable client contracts.
5. Given a type, use case, or non-obvious provider workaround, when source is reviewed, then names use canonical product language and comments explain why or constraints without narrating the code.

### States and accessibility

Nullable-safe contracts must make absence explicit so clients can render deliberate empty or unavailable states rather than fail during presentation.

### Verification

- Domain/unit: Compile with nullable analysis and test value objects/messages.
- Integration/contract: Exercise cancellation and serialized enum/value-object contracts.
- Component: Consume optional fields under strict frontend typing.
- Journey/live: Cancel a long-running request and confirm the app recovers.
- Manual/rendered: Inspect analyzer configuration and representative source files.

## L2-ARCH-011: Configuration, composition, health, and dependencies are governed

**Traces to:** L1-ARCH-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Related configuration shall bind to typed options and validate mandatory production values at startup. Production secrets shall come from environment or secret stores, while development placeholders are unmistakably non-secret. Services shall register through capability or layer extension methods so `Program.cs` remains a readable composition root and middleware pipeline. The API shall expose health and distinguish liveness from dependency readiness where useful. Each new dependency shall have a clear capability, maintenance, license, security, and runtime or bundle-cost justification.

### Acceptance criteria

1. Given a production-required configuration value is missing or still a placeholder, when the service starts, then startup fails with a non-secret diagnostic.
2. Given the composition root, when it is inspected, then capability/layer registration and middleware ordering are readable without business policy embedded in `Program.cs`.
3. Given a process is alive but a required external dependency is unavailable, when health endpoints are queried, then liveness and readiness communicate distinct states where the deployment uses that distinction.
4. Given a proposed package, when it is added, then its capability, platform alternative, maintenance health, license, security posture, and runtime/bundle impact have been assessed and pinned.

### States and accessibility

Configuration and health details exposed publicly must remain sanitized; operational failure pages shall not disclose secrets and shall provide an accessible retry/support path where user-visible.

### Verification

- Domain/unit: Test typed option validators and dependency-policy helpers.
- Integration/contract: Start with valid and invalid configurations and exercise health endpoints.
- Component: Test a sanitized service-unavailable state.
- Journey/live: Remove a staging dependency and confirm readiness plus user-facing degradation.
- Manual/rendered: Review composition, option schemas, lockfiles, and dependency records.

## L2-ARCH-012: Untrusted and external work is bounded behind useful seams

**Traces to:** L1-ARCH-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

All untrusted input and external process, network, file, mail, identity, and vendor work shall have explicit validation, size, concurrency, timeout, cancellation, and retry boundaries appropriate to its risk. Provider SDKs shall remain in Infrastructure behind focused Application-owned seams. The design shall not add generic repositories over EF Core unless a repository expresses a meaningful aggregate boundary.

### Acceptance criteria

1. Given oversized or malformed input, when it reaches a boundary, then it is rejected before unbounded allocation, persistence, or provider work.
2. Given an external provider stalls or fails, when a use case invokes it, then configured timeout/cancellation behavior bounds the request and produces a typed failure.
3. Given provider implementation is replaced by a deterministic fake, when Application tests run, then no provider SDK type is required outside Infrastructure.
4. Given persistence access is introduced, when its abstraction is reviewed, then it uses the Application database seam or a focused aggregate repository rather than a generic repository wrapper.

### States and accessibility

Bounded failures shall map to understandable retry, validation, or unavailable states without trapping keyboard focus or silently discarding user input.

### Verification

- Domain/unit: Test boundary policies, timeout decisions, and provider-independent use cases.
- Integration/contract: Exercise oversized input, timeout, cancellation, and provider-failure mappings.
- Component: Verify retry and preserved-input behavior.
- Journey/live: Simulate a slow external provider during a community action.
- Manual/rendered: Review configured bounds and Application/Infrastructure seams.

## L2-ARCH-013: Realtime delivery follows committed state and reconciles

**Traces to:** L1-ARCH-005

**Status:** proposed

**Source:** product brief + research

**Related:** —

Realtime delivery shall be used only for community experiences where concurrent users need timely updates. SignalR or another transport shall remain behind an Application abstraction, and stable events shall publish only after successful persistence. The client shall treat REST and persisted server state as authoritative, reconnect with bounded behavior, reconcile missed or conflicting state, and retain a usable REST path when the realtime channel is unavailable.

### Acceptance criteria

1. Given a failed transaction, when a community mutation aborts, then no corresponding realtime success event is delivered.
2. Given a committed mutation, when its event is broadcast, then the event contract contains a stable identity and enough version/state information for clients to reconcile.
3. Given a disconnected member client, when it reconnects, then it reloads or reconciles authoritative state rather than assuming every event was received.
4. Given realtime infrastructure is unavailable, when a member uses a supported workflow, then core reads and writes remain usable through REST with an explicit degraded state.

### States and accessibility

Connection, reconnecting, stale, synchronized, and degraded states shall be communicated without relying on color, and updates shall be announced selectively rather than making every event noisy.

### Verification

- Domain/unit: Test event creation only from committed use-case outcomes.
- Integration/contract: Verify transaction/event ordering and stable event serialization.
- Component: Test reconnect, stale, reconciled, and degraded UI states.
- Journey/live: Disconnect, mutate from another client, reconnect, and verify reconciliation.
- Manual/rendered: Inspect event contracts and connection-state communication.

## L2-ARCH-014: Initial deployment is cohesive while source boundaries remain separable

**Traces to:** L1-ARCH-005

**Status:** proposed

**Source:** product brief + research

**Related:** —

The first production topology shall allow one API deployable to host the built Angular application, canonical design assets, and static marketing site when that reduces operational cost. Source and configuration boundaries shall remain independent so marketing can later move to `www` and the app/API to `app` without rewriting community policy, reusable libraries, or hard-coded origins.

### Acceptance criteria

1. Given the initial publish pipeline, when it runs, then it emits one coherent artifact containing hashed Angular output, canonical design assets, static marketing files, and the API.
2. Given the source tree, when deployment packaging is removed from consideration, then marketing, Angular, design-system, and backend source remain independently owned boundaries.
3. Given an origin split is configured, when Angular clients and marketing CTAs are built, then deployment configuration supplies their origins without library source changes.
4. Given no documented operational trigger exists, when topology is reviewed, then speculative service or origin splitting is not required merely for architectural appearance.

### States and accessibility

Both cohesive and split topologies shall preserve keyboard flow, deep-link behavior, error handling, and canonical design-system accessibility.

### Verification

- Domain/unit: Verify route/origin configuration helpers where pure.
- Integration/contract: Publish and inspect artifact contents plus injected configuration.
- Component: Build components without deployment-host assumptions.
- Journey/live: Run public-to-sign-in and refreshed member deep-link journeys in the deployed topology.
- Manual/rendered: Review source ownership and topology-change documentation.

## L2-ARCH-015: Vocabulary, invariants, and vertical slices drive implementation

**Traces to:** L1-ARCH-006

**Status:** proposed

**Source:** product brief + research

**Related:** —

The product thesis, canonical community nouns, primary member journey, product-defining server invariant, measures, risks, and non-goals shall be established before broad scaffolding. One canonical code name shall exist per concept, with unavoidable UI synonyms documented. Delivery shall begin with a differentiating vertical slice through route, UI, typed client, endpoint, use case, domain rule, real persistence, response or event, refreshed UI, failures, tests, and documentation before dashboard or administration breadth.

### Acceptance criteria

1. Given a product concept appears in route, type, UI, test, or documentation, when vocabulary is reviewed, then one canonical noun is used or an explicit UI synonym mapping exists.
2. Given the first implementation slice, when its scope is inspected, then an invited Account accepts a persisted Membership with server-computed Permissions, publishes a scoped Post, a disallowed Account fails to read it by guessed identifier, and a committed Moderation Action removes it from direct, Feed, Search, Notification, and realtime paths rather than stopping at sign-in, settings, or generated CRUD.
3. Given the slice is completed, when its evidence is reviewed, then validation, authorization, loading, empty, error, conflict, responsive, and accessible states are covered where applicable.
4. Given a proposed dashboard, administration area, shared abstraction, or speculative setting, when the core differentiator is incomplete, then the proposal remains outside the current slice unless it is necessary to prove that journey.

### States and accessibility

The first slice shall prove representative success and material failure states across narrow/wide viewports, keyboard use, focus, labels, announcements, and reduced motion.

### Verification

- Domain/unit: Test the product-defining invariant directly.
- Integration/contract: Exercise its API, authorization, and persistence path.
- Component: Test all applicable journey states.
- Journey/live: Complete the differentiating workflow end to end against live contracts.
- Manual/rendered: Compare canonical vocabulary, accepted design record, and rendered outcome.

## L2-ARCH-016: Consequential decisions and documentation remain traceable

**Traces to:** L1-ARCH-006

**Status:** proposed

**Source:** product brief + research

**Related:** —

Choices affecting dependency direction, data ownership or shape, security, external services, public contracts, deployment, design-system semantics, licensing, or operational cost shall be recorded as numbered ADRs. ADR history shall be immutable through Proposed, Accepted, Superseded, or Deprecated states, with replacements linked both ways. Behavior, commands, configuration, architecture, data, deployment, specifications, runbooks, support notes, and changelog shall be updated in the same change that makes them stale.

### Acceptance criteria

1. Given a consequential architecture or product choice, when implementation begins, then an ADR records context, drivers, alternatives, decision, trade-offs, follow-up triggers, and verification.
2. Given an accepted decision changes, when the replacement is approved, then a new ADR is created and the prior record links to it without rewriting historical rationale.
3. Given code changes user behavior or operational truth, when the change is reviewed, then affected specs, diagrams, setup commands, runbooks, security/support notes, and changelog are current.
4. Given a cross-stack requirement is marked implemented, when evidence is inspected, then live integration evidence exists rather than only prose, an illustrative artifact, or an isolated unit test.

### States and accessibility

Decision and product documentation shall record accessibility effects and intentional documented-example/implementation divergences where user-facing behavior changes.

### Verification

- Domain/unit: Validate identifiers, links, and status values with documentation tooling.
- Integration/contract: Confirm documented contracts and commands match runtime behavior.
- Component: Compare accepted component states with implemented behavior.
- Journey/live: Follow requirement-to-test evidence for a completed cross-stack journey.
- Manual/rendered: Review ADR history, documentation freshness, and traceability coverage.
