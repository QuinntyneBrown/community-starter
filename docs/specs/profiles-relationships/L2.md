# PROF — Profiles and relationships behavior

## L2-PROF-001: Provision one Profile

**Traces to:** L1-PROF-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Each eligible Account owns exactly one canonical Profile with a stable identifier and safe defaults;
Community Memberships reference that Profile rather than duplicating personal identity.

### Acceptance criteria

1. Given a newly activated Account without a Profile, when provisioning completes, then one Profile with a stable identifier and privacy-safe defaults is created for that Account.
2. Given provisioning is retried for an Account that already owns a Profile, when the request is processed, then the existing Profile is returned and no duplicate is created.
3. Given an Account or Community identifier is supplied as another Profile's owner, when creation is attempted, then the server ignores or rejects the claim and persists no unauthorized Profile.

### States and accessibility

Show provisioning, ready, retryable failure, and unavailable states; avoid flashing placeholder
identity, give fallback imagery accessible text, and announce when setup can continue.

### Verification

- Domain/unit: one-Profile-per-Account, ownership, identifier, and default-visibility rules.
- Integration/contract: activation provisioning, retry, and forged-owner rejection.
- Component: loading, fallback identity, failure, and ready states.
- Journey/live: activate an Account and reach its single Profile from two Communities.
- Manual/rendered: fallback text, keyboard, zoom, and announcement review.

## L2-PROF-002: Update owned Profile fields

**Traces to:** L1-PROF-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

An Account can update only its owned Profile through bounded, normalized fields and optimistic
concurrency, with server-side validation and sanitization.

### Acceptance criteria

1. Given the owning Account, current Profile version, and valid fields, when an update is submitted, then normalized safe values and a new version are returned.
2. Given a non-owner or a client-supplied owner change, when an update is submitted, then the server rejects it and preserves all Profile fields and ownership.
3. Given fields exceed limits, contain unsupported markup, or violate identifier uniqueness, when an update is submitted, then field-safe validation or conflict details are returned and no values change.
4. Given a stale version, when an update is submitted, then the server returns current safe state and does not overwrite a newer edit.

### States and accessibility

Show loading, editing, unsaved, saving, validation, conflict, forbidden, and saved states; associate
errors, preserve edits on failure, focus the summary, and announce saved status.

### Verification

- Domain/unit: ownership, normalization, field limits, sanitization, and version rules.
- Integration/contract: update, non-owner, validation, uniqueness, and concurrency cases.
- Component: edit form, validation summary, unsaved warning, and conflict reconciliation.
- Journey/live: edit a Profile, provoke a stale update, and reconcile without lost input.
- Manual/rendered: keyboard, screen-reader labels, reflow, and long-content review.

## L2-PROF-003: Control Profile visibility

**Traces to:** L1-PROF-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Profile visibility is server-filtered by field and audience using the owner's current policy,
relationship safety, and Community access; hidden fields are omitted rather than client-masked.

### Acceptance criteria

1. Given an owner selects a supported audience for a Profile field, when the policy is saved, then subsequent reads include that field only for Accounts in the selected audience.
2. Given an unauthorized or anonymous viewer requests a hidden field directly or through a list, Feed, Search result, export, or realtime payload, when the response is built, then the field is omitted.
3. Given privacy is tightened, when cached or indexed Profile projections are refreshed, then new reads obey the tighter policy and stale clients cannot use old data to authorize an action.
4. Given an unsupported visibility transition is submitted, when validation fails, then prior visibility and Profile data remain unchanged.

### States and accessibility

Show audience choices, preview, saving, validation, and saved states; explain audiences in text,
associate controls with fields, and do not represent privacy by icons or color alone.

### Verification

- Domain/unit: field-audience, safety override, and visibility-transition rules.
- Integration/contract: owner/member/anonymous reads across detail, list, Search, and privacy export.
- Component: visibility controls, preview, validation, and save states.
- Journey/live: change a field from public to restricted and verify multiple viewer contexts.
- Manual/rendered: keyboard groups, screen-reader descriptions, and privacy-copy review.

## L2-PROF-004: Manage Profile imagery

**Traces to:** L1-PROF-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

An Account can attach or remove an approved Profile image through the same validated Attachment
lifecycle used by the platform, with a deterministic accessible fallback.

### Acceptance criteria

1. Given the owning Account uploads a supported image within limits, when scanning and processing succeed, then the approved Attachment becomes the Profile image and declared variants are available.
2. Given the upload is unsupported, oversized, unsafe, or processing fails, when the result is known, then the prior Profile image remains active and the failed Attachment is not publicly retrievable.
3. Given a non-owner attempts to select an Attachment or an Attachment belongs to another Account or Community scope, when the change is requested, then the server rejects it and preserves the current image.
4. Given the owner removes the image, when deletion commits, then the Profile uses the accessible fallback and underlying Attachment retention follows policy.

### States and accessibility

Show preview, upload progress, processing, failure, crop/variant preview if offered, removal, and
fallback states; controls are labeled and the image has contextual alternative text or is decorative.

### Verification

- Domain/unit: ownership, approved-state, replacement, and fallback rules.
- Integration/contract: upload association, scan/process failure, cross-scope rejection, and removal.
- Component: preview, progress, processing, error, removal, and fallback states.
- Journey/live: replace and remove an image while a second viewer observes only approved versions.
- Manual/rendered: keyboard upload, zoom, fallback, and assistive-text review.

## L2-PROF-008: Block another Account

**Traces to:** L1-PROF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Block is a private Account safety control that immediately prevents direct relationship and
interaction paths in both directions, independent of shared Community Membership.

### Acceptance criteria

1. Given two distinct Accounts, when one confirms Block, then pending direct-interaction requests are cancelled and prohibited direct interactions become unavailable atomically.
2. Given a Block exists in either direction, when either Account attempts a prohibited direct interaction, then the server rejects it and preserves all protected state.
3. Given the Accounts share a Community, when content, Profiles, directory rows, engagement, Events, notifications, or realtime state are read or changed, then the versioned shared-Community Block matrix is enforced without changing either Membership or granting moderator-like authority.
4. Given the blocker removes Block, when the action commits, then prior interaction authority does not automatically return and new interactions require fresh eligible actions.

### States and accessibility

Show impact summary, confirmation, blocked, unblock, and safe generic unavailable states; do not
notify the blocked Account of Block ownership, trap focus in confirmation, or rely on color.

### Verification

- Domain/unit: bilateral interaction effects, pending-request cancellation, privacy, and unblock rules.
- Integration/contract: block/unblock, prohibited actions, Feed/Search filtering, and shared Community.
- Component: confirmation, blocked-state management, and generic unavailable states.
- Journey/live: begin direct interaction, Block, verify removal, then unblock without restoring authority.
- Manual/rendered: privacy copy, keyboard, screen-reader, and destructive-impact review.

## L2-PROF-009: Mute another Account

**Traces to:** L1-PROF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Mute privately suppresses eligible content and notifications from another Account for the muting
Account without changing Membership, Permission, or the muted Account's access.

### Acceptance criteria

1. Given two distinct eligible Accounts, when one Account mutes the other, then future Feed and notification projections for the muting Account suppress the muted Account according to policy.
2. Given Mute exists, when either Account uses a surface not covered by mute policy, then authorization and interaction eligibility remain unchanged.
3. Given the muted Account queries its relationships or activity, when responses are built, then the existence or owner of Mute is not disclosed.
4. Given Mute is removed, when projections refresh, then newly eligible content may appear without reconstructing Membership or interaction state.
5. Given an Account attempts to create a Mute owned by another Account or targeting itself, when the request is evaluated, then the server rejects it and preserves existing relationship state.

### States and accessibility

Show mute impact, muted, unmute, and refresh states only to the owner; controls have explicit state
names and Feed changes are not announced item by item.

### Verification

- Domain/unit: suppression scope, privacy, and non-authorization effects.
- Integration/contract: mute/unmute, Feed/notification filtering, and relationship preservation.
- Component: mute action, muted management list, and Feed refresh states.
- Journey/live: mute an Account, verify suppression, and unmute without changing authorization.
- Manual/rendered: keyboard, screen-reader state, privacy copy, and live-update noise review.

## L2-PROF-010: Resolve Block and Mute precedence

**Traces to:** L1-PROF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

The server applies one consistent relationship precedence: Block safety restrictions override Mute
presentation, while Community Role and Permission remain separately enforced.

### Acceptance criteria

1. Given Mute and Block states overlap, when visibility or interaction is evaluated, then the shared-Community Block matrix applies first and Mute suppression applies only to the owner's remaining eligible presentation.
2. Given an Account has a privileged Community Role, when Block affects a required moderation or safety workflow, then only the explicitly documented minimum exception applies and all other restricted interaction remains unavailable.
3. Given concurrent Block, Mute, unmute, or unblock actions race, when they commit, then the resulting state satisfies precedence, prohibited interactions are absent, and rejected actions return a conflict without partial state.
4. Given a client submits relationship flags, when a protected action is requested, then the server ignores those claims and derives current relationship state.

### States and accessibility

Render only actions valid under effective precedence, explain unavailable state where safe, reconcile
conflicts without focus loss, and do not disclose another Account's private Block or Mute state.

### Verification

- Domain/unit: complete precedence matrix, safety exception, and concurrency invariants.
- Integration/contract: conflicting states/actions, privileged-role exception, and forged flags.
- Component: effective action states and privacy-safe conflict reconciliation.
- Journey/live: race Mute and Block and verify all Feed, Search result, and interaction surfaces.
- Manual/rendered: privacy language, keyboard recovery, and announcement review.

## L2-PROF-011: Render Community context safely

**Traces to:** L1-PROF-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

A Profile shown in Community context combines only authorized canonical Profile fields with safe
current Membership and Role information from that same Community.

### Acceptance criteria

1. Given a viewer may access a Community and its member directory, when a Profile card or detail is returned, then Profile fields follow their audience policy and Membership/Role fields come only from that Community.
2. Given the Profile owner belongs to another Community with a different Role or Membership state, when current Community context is rendered, then those other-Community details are not disclosed or merged.
3. Given the viewer loses Membership or directory Permission, when the next list, detail, Search result, or realtime update is requested, then Community-only Profile context is withheld.
4. Given a client supplies a Community identifier unrelated to the resource route, when context is built, then the server rejects or uses the authoritative scope and returns no cross-Community data.

### States and accessibility

Show Profile fallback, Community context, restricted fields, access changed, and unavailable states;
label Role and Membership textually and provide safe recovery navigation.

### Verification

- Domain/unit: Profile-field plus Membership-context composition rules.
- Integration/contract: multi-Community cards/details, directory Permission, loss of access, and mismatch.
- Component: Profile card/detail with restricted, fallback, and access-changed states.
- Journey/live: view one Account across two Communities with different Roles and permissions.
- Manual/rendered: semantic headings/lists, keyboard links, privacy copy, and reflow review.

## L2-PROF-012: Retire Profile data safely

**Traces to:** L1-PROF-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

When an Account deactivates, its Profile enters the reversible privacy state; when deletion completes,
Profile and relationship data are removed, de-identified, or narrowly retained according to policy
without deleting Community records outside that policy.

### Acceptance criteria

1. Given Account deactivation commits, when Profile access is requested, then authentication and editing stop immediately and public representation follows the disclosed reversible state without claiming deletion.
2. Given Account deletion reaches its terminal retention action, when de-identification runs, then personal Profile fields and imagery are removed while retained Posts or Comments use the declared neutral attribution.
3. Given Block and Mute records no longer serve a permitted purpose, when deletion completes, then they are deleted or de-identified and cannot reveal the deleted Account through counts or lists.
4. Given a deactivated Account is reactivated within policy, when restoration commits, then the same Profile is restored without duplicating identifiers, relationships, or Memberships beyond policy; deletion-pending or deleted Accounts cannot use this path.

### States and accessibility

Show deactivated/restoration-window, unavailable, neutral attribution, and restored states consistently; avoid broken
image labels, preserve semantic authorship text, and do not expose retention internals.

### Verification

- Domain/unit: grace, de-identification, attribution, relationship, and restoration rules.
- Integration/contract: deactivate, retained-content view, delete/de-identify, list/count, and restore cases.
- Component: deactivated-Account Profile, neutral attribution, unavailable, and restored states.
- Journey/live: deactivate and restore, then separately exercise irreversible deletion in a safe fixture.
- Manual/rendered: assistive attribution, fallback imagery, privacy wording, and reflow review.

## L2-PROF-013: Enforce the shared-Community Block matrix

**Traces to:** L1-PROF-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

The baseline Block policy is bilateral for the blocked pair and suppresses their ordinary discovery,
authored content, attribution, engagement, direct interaction, and personal delivery paths while
leaving Community Membership and third-party access unchanged.

### Acceptance criteria

1. Given a Block exists in either direction, when either Account views Profile detail, member directory, autocomplete, or direct contact surfaces for the other, then the other Account is omitted or privacy-safely unavailable and no Message request, Membership invitation between the pair, Mention, Quote Reference, reply, or targeted interaction can be created.
2. Given either Account authored a Post, Comment, Reaction, Mention, or Quote Reference, when the other Account's direct read, Feed, Search, Tag list, export, cache, Notification, or realtime projection is built, then that attributable item, link, snippet, and identity are suppressed; third-party items remain eligible and per-viewer counts exclude suppressed records without naming Block as the cause.
3. Given both Accounts can otherwise access an Event, when Event discovery or participation is evaluated, then an Event whose responsible organizer is the other is suppressed, a co-organizer assignment cannot override the Block, pairwise Event interaction is denied, and organizer/attendee identity, roster, Message, Mention, Notification, presence, and contact detail are mutually hidden.
4. Given an authorized Moderator or operator performs a required safety workflow involving the other Account, when the action is evaluated, then only the versioned minimum evidence and controls needed for that task bypass ordinary suppression, the exception is audited, and the Block owner or existence is not disclosed through the subject-facing result.
5. Given a Block begins or ends, when requests, queued work, and projections reconcile, then requests re-evaluate before their next effect, derived views converge within the accepted Block-propagation objective, default five minutes, prior interaction or Delivery state is not restored, and anonymously public content remains governed by the separately disclosed Public Projection policy.

### States and accessibility

Use generic unavailable, filtered-gap, recalculated-count, interaction-denied, and unblocked states;
never identify the Block owner, and keep shared Community navigation and safety actions accessible.

### Verification

- Domain/unit: Test the bilateral matrix for every noun, direction, count, precedence, and exception.
- Integration/contract: Exercise direct, list, Feed, Search, content, engagement, Event, export, Delivery, cache, and realtime paths.
- Component: Test safe unavailable items, thread gaps, counts, Event cards/rosters, and unblock reconciliation.
- Journey/live: share content and an Event, Block in each direction, probe every path, then unblock.
- Manual/rendered: privacy language, thread comprehension, keyboard, screen reader, focus, and reflow.

## L2-PROF-014: List a privacy-safe Member Directory

**Traces to:** L1-PROF-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Each Community may expose a Member Directory whose rows combine only current eligible Membership
context and permitted Profile fields under explicit viewer, subject, Block, and lifecycle policy.

### Acceptance criteria

1. Given the Member Directory is enabled and a viewer has current directory Permission, when a bounded page is requested, then only active eligible Memberships visible to that viewer are returned in stable configured order with canonical Profile identity and allow-listed same-Community Role context.
2. Given a Member chooses an allowed directory-discovery preference, when lists, autocomplete, Search, counts, and direct directory detail are built, then that preference is enforced except for a separately disclosed minimum governance or safety listing required by current Role, and no private field is broadened.
3. Given Membership, Profile visibility, Account lifecycle, Block, Moderation Action, Space scope, or directory Permission makes a row ineligible, when any page, filter, total, cursor, cache, export, or realtime result is built, then the row and its contribution to viewer-specific totals are withheld without revealing the cause.
4. Given a query, filter, cursor, target, or Community identifier is malformed, expired, inaccessible, or cross-Community, when requested, then it cannot broaden results or disclose hidden identities, matches, facets, totals, timing, or Membership existence.
5. Given directory eligibility changes, when requests and projections reconcile, then request-time filtering applies immediately and derived lists, totals, Search candidates, and caches converge within the accepted directory objective, default five minutes, without restoring a prior interaction or Notification.

### States and accessibility

Show disabled, restricted, loading, populated, filtered, empty, paging, access-changed, and unavailable
states; lists use semantic structure, searchable controls, stable focus, and privacy-safe empty copy.

### Verification

- Domain/unit: Test viewer/subject matrix, preference, required-role exception, counts, and ordering.
- Integration/contract: Page/filter, cross-Community probes, lifecycle, Block, Action, export, cache, and realtime.
- Component: Directory controls, rows, private fields, empty, paging, and access-changed states.
- Journey/live: list two Communities, opt out, Block, revoke Membership, and verify every projection.
- Manual/rendered: privacy explanation, keyboard search/list, screen reader, focus, and reflow.
