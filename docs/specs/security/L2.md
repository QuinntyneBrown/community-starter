# SECU — Security and privacy detailed requirements

## L2-SECU-001: High-risk community flows have maintained threat models

**Traces to:** L1-SECU-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The starter shall maintain threat models for identity and Session handling, Community isolation, Roles
and moderation boundaries, invitations and share links, uploads and link previews, Messaging, Event
locations and access links, billing and checkout, guardian/age flows, support impersonation,
break-glass access, Appeals, external callbacks, and destructive actions. Each model shall identify
assets, actors, trust boundaries, abuse cases, mitigations, residual risk, verification, owner, and a
trigger for reassessment. Material architecture or behavior changes shall update the affected model
in the same change.

### Acceptance criteria

1. Given a listed high-risk flow, when its documentation is inspected, then assets, actors, trust boundaries, abuse cases, controls, residual risks, owner, and verification are present.
2. Given a new trust boundary, identity provider, upload type, Message/Event/billing/guardian/support flow, callback, sharing mode, or destructive capability, when it is proposed, then threat-model review occurs before production enablement.
3. Given an accepted mitigation changes, when implementation is reviewed, then related requirements, tests, runbooks, and residual-risk notes are updated.
4. Given an unresolved high-risk threat, when release readiness is assessed, then it is explicitly accepted by an owner, mitigated, or blocks release rather than disappearing from the record.

### States and accessibility

Security controls shall preserve accessible authentication, error recovery, confirmation, and support paths and shall not create unlabeled or keyboard-inoperable challenges.

### Verification

- Domain/unit: Test pure policy controls identified by each threat model.
- Integration/contract: Exercise authentication, Community-isolation, callback, upload, and destructive mitigations.
- Component: Test security-state messaging and confirmation behavior.
- Journey/live: Run representative abuse and legitimate-user scenarios in staging.
- Manual/rendered: Review threat models, residual risk, ownership, and accessibility impact.

## L2-SECU-002: Production identity and session behavior cannot inherit development shortcuts

**Traces to:** L1-SECU-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Production shall use an identity and session design selected against the documented threat model. Development-only identity bypasses, seeded credentials, relaxed validation, or placeholder providers shall be visibly labeled, gated to explicit non-production environments, and rejected by production startup validation. Client route guards and hidden controls shall remain navigation affordances, never authentication or authorization controls.

### Acceptance criteria

1. Given production configuration, when the service starts, then development identity bypasses, seeded credentials, placeholder issuers/keys, or relaxed validation cause startup failure.
2. Given an absent, invalid, expired, or otherwise unacceptable session, when a protected endpoint is called, then the server returns the documented authentication failure without changing state.
3. Given the Angular route guard or UI allows navigation, when the server session is invalid or lacks authorization, then the API still rejects access and returns no protected data.
4. Given development authentication is enabled, when the application renders and documentation is reviewed, then its non-production status and limitations are unmistakable.

### States and accessibility

Expired-session, sign-in-required, forbidden, and identity-provider failure states shall be distinguishable, keyboard operable, focus-managed, and free of sensitive diagnostic detail.

### Verification

- Domain/unit: Test session-policy and environment-gate decisions.
- Integration/contract: Start under invalid production configurations and exercise invalid/expired sessions.
- Component: Test sign-in-required, expired, forbidden, and provider-error states.
- Journey/live: Attempt protected navigation with valid, expired, and missing sessions.
- Manual/rendered: Review identity configuration, environment labels, and recovery accessibility.

## L2-SECU-003: Every protected resource is authorized in Community context

**Traces to:** L1-SECU-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every server read and mutation over Community-owned data shall establish the current actor and authorized Community scope before accessing the resource. A route, body, query-string, hub, event, or callback Community identifier shall be treated only as input, never proof of access. Authorization, Roles, moderation powers, quotas, and workflow gates shall be evaluated server-side against current state on every action.

### Acceptance criteria

1. Given a member supplies another community's resource or community identifier, when any protected read or write executes, then no out-of-scope data is returned and no state changes.
2. Given a role, membership, quota, or moderation permission changes after the UI renders, when the old action is submitted, then the server evaluates current policy and rejects it when no longer allowed.
3. Given realtime hub connection or subscription, when scope is established, then the server authorizes the channel/resource and does not trust the requested group name alone.
4. Given scoped absence, when the resource belongs to another Community, then the documented response does not reveal unnecessary existence or ownership information.

### States and accessibility

Locked, forbidden, unavailable, and stale-permission states shall provide non-sensitive explanations and recovery without relying on color or hiding enforcement behind disabled controls.

### Verification

- Domain/unit: Test role, quota, moderation, and workflow authorization policy.
- Integration/contract: Run a cross-Community matrix across reads, writes, hubs, and callbacks.
- Component: Test locked-action explanations and stale-permission reconciliation.
- Journey/live: Attempt cross-community access and permission revocation during a session.
- Manual/rendered: Review Community filters, policy calls, and information-disclosure behavior.

## L2-SECU-004: Secrets stay outside source and placeholders fail safely

**Traces to:** L1-SECU-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Credentials, signing material, tokens, connection secrets, and production-sensitive configuration shall come from environment or managed secret stores and shall never be committed. Typed startup validation shall reject missing production values and known placeholders without echoing secrets. Local examples shall use unmistakably non-secret names and documented provisioning. Generated output, local databases, credentials, and secret-bearing test results shall be excluded from version control.

### Acceptance criteria

1. Given repository history and the current worktree, when secret scanning runs, then no production credential, token, private key, connection secret, or real secret value is detected.
2. Given a required production secret is absent or matches a placeholder, when startup runs, then the service fails before accepting traffic and the diagnostic does not disclose the value.
3. Given local setup documentation, when a developer follows it, then secret values are provisioned outside tracked files and example names cannot be mistaken for production-ready credentials.
4. Given generated or local artifacts, when source-control status is inspected, then local databases, credentials, test results, and build output are excluded.

### States and accessibility

User-visible configuration failures shall present a sanitized unavailable/support state; operational diagnostics remain accessible to authorized operators only.

### Verification

- Domain/unit: Test typed option validators and placeholder rejection.
- Integration/contract: Start with absent, placeholder, and valid secret references.
- Component: Test the sanitized unavailable state.
- Journey/live: Verify a deliberately misconfigured staging release fails closed.
- Manual/rendered: Review secret-store integration, ignore rules, examples, and scanner results.

## L2-SECU-005: Public failures are sanitized and private diagnostics are redacted

**Traces to:** L1-SECU-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Expected application failures shall map centrally to stable sanitized Problem Details, while unexpected diagnostics shall remain server-side. Logs, traces, metrics, support artifacts, and error responses shall omit tokens, passwords, credentials, raw authentication material, and sensitive community content. Structured diagnostics shall use correlation/request identifiers and stable event names so authorized operators can investigate without exposing payloads.

### Acceptance criteria

1. Given an expected validation, forbidden, unavailable, or conflict failure, when returned, then Problem Details contains the stable client contract and no stack trace, provider detail, secret, or sensitive content.
2. Given an unexpected exception, when it is logged and returned, then the client receives a sanitized response and the server log contains a correlation identifier plus actionable redacted context.
3. Given a request includes tokens, passwords, or sensitive content, when logs, traces, metrics, and support exports are inspected, then those values are absent or irreversibly redacted.
4. Given a client reports a correlation identifier, when an authorized operator searches telemetry, then the relevant stable events can be found without needing the member's sensitive payload.

### States and accessibility

Error summaries shall be programmatically associated or announced as appropriate, preserve user input when safe, and distinguish validation from authorization and transient failure.

### Verification

- Domain/unit: Test exception-to-Problem-Details mapping and redaction functions.
- Integration/contract: Trigger expected and unexpected failures and inspect responses/telemetry.
- Component: Test accessible field, summary, forbidden, and retryable-error presentation.
- Journey/live: Correlate a staged failure from UI through server telemetry.
- Manual/rendered: Review representative logs and rendered errors for disclosure.

## L2-SECU-006: Source, fixtures, and demos contain no real personal data

**Traces to:** L1-SECU-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Source, documentation, mocks, screenshots, fixtures, seed data, local databases, and committed test artifacts shall contain no real personal or community data. Demonstration data shall be synthetic, deterministic, and enabled only in development or dedicated demo environments. Screenshots and examples shall avoid production identifiers, tokens, private posts, messages, reports, or member attributes.

### Acceptance criteria

1. Given committed fixtures, mocks, documents, and screenshots, when reviewed, then all people, communities, content, addresses, and identifiers are synthetic and contain no production-derived data.
2. Given production startup, when environment gates are evaluated, then demonstration seeding is disabled and cannot create synthetic default accounts silently.
3. Given a screenshot or support artifact is proposed for commit, when reviewed, then tokens, private content, real identifiers, and personal attributes are absent or replaced with synthetic data.
4. Given test cleanup completes, when local artifacts are inspected, then generated databases, results, and screenshots are not staged for source control by default.

### States and accessibility

Synthetic examples shall still model realistic long content, diverse names, assistive labels, and error states without using real people as test data.

### Verification

- Domain/unit: Validate deterministic seed generation.
- Integration/contract: Verify seeding environment gates and cleanup behavior.
- Component: Render synthetic data across representative states.
- Journey/live: Run demo journeys in the dedicated environment and confirm isolation.
- Manual/rendered: Audit committed examples, screenshots, fixtures, and ignore rules.

## L2-SECU-007: Untrusted input is validated and resource-bounded

**Traces to:** L1-SECU-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every untrusted HTTP, hub, form, callback, import, and external-service boundary shall validate shape, type, identifiers, lengths, counts, and applicable product constraints. Critical rules shall be repeated at the authoritative domain or application layer. Payload size, concurrency, execution time, retries, and external work shall be bounded according to risk, and cancellation shall propagate. The server shall not trust duplicated body identifiers over authoritative route identifiers.

### Acceptance criteria

1. Given malformed, oversized, excessive-count, or otherwise invalid input, when it reaches a boundary, then it is rejected before unbounded allocation, provider work, or persistence.
2. Given request-shape validation succeeds but current-state policy fails, when processed, then the authoritative layer rejects the action and state remains unchanged.
3. Given body and route identifiers disagree, when the request executes, then the route identifier governs and cross-resource redirection cannot occur.
4. Given external work stalls or a request is cancelled, when configured limits are reached, then work terminates or detaches safely according to policy and returns a typed sanitized failure.

### States and accessibility

Validation shall identify affected controls and constraints accessibly, preserve safe user input, and distinguish correction from retry or forbidden outcomes.

### Verification

- Domain/unit: Test boundary limits and authoritative policy independently.
- Integration/contract: Fuzz representative inputs and exercise size, count, timeout, and cancellation limits.
- Component: Test field association, summary, preserved input, and retry states.
- Journey/live: Submit invalid, oversized, stale, and cancelled community actions.
- Manual/rendered: Review limits, error language, and absence of provider diagnostics.

## L2-SECU-008: Uploads validate content and isolate storage paths

**Traces to:** L1-SECU-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every upload shall validate allowed content type, extension, size, filename, and storage path on the server; client-provided MIME and names shall be untrusted. Storage access shall remain behind an Application boundary, derive Community ownership server-side, prevent path traversal and executable placement, and return only stable identifiers or safe delivery URLs according to the threat model.

### Acceptance criteria

1. Given an upload whose declared MIME conflicts with its content or whose type/extension/size is disallowed, when submitted, then the server rejects it without making it publicly available.
2. Given a filename contains traversal, absolute-path, reserved, or unsafe characters, when processed, then storage selects a server-controlled safe path and the input cannot escape its Community storage scope.
3. Given a valid upload, when persisted, then its Community owner and authorization are derived from current server scope rather than client metadata.
4. Given an unauthorized actor requests an upload, when delivery is attempted, then the storage identifier or URL does not bypass resource authorization.

### States and accessibility

Upload controls shall expose allowed formats/limits, progress where useful, validation, cancellation, retry, and success using labels and text rather than color alone.

### Verification

- Domain/unit: Test filename, type, size, path, and ownership policies.
- Integration/contract: Upload spoofed MIME, unsafe names, oversized files, and cross-Community identifiers.
- Component: Test accessible file selection, progress, validation, cancellation, and retry.
- Journey/live: Complete valid and rejected avatar/attachment journeys using real storage integration.
- Manual/rendered: Inspect storage layout, response contracts, and control instructions.

## L2-SECU-009: Security-sensitive workflow transitions fail closed

**Traces to:** L1-SECU-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Invitation and share-link use, Role changes, Moderation Actions, external callbacks, Account deactivation or deletion, Community retirement, and irreversible erasure shall authenticate or validate the caller as applicable, establish current Community scope, enforce authoritative state and anti-abuse controls identified by threat modeling, persist atomically, and emit success only after commit. Unknown, expired, replayed, stale, unauthorized, or malformed attempts shall fail without partial state.

### Acceptance criteria

1. Given an invitation/share credential or callback is invalid, expired, replayed, stale, malformed, or out of scope, when processed, then it fails closed and causes no partial membership or content change.
2. Given a role or moderation transition is requested by an insufficiently authorized actor, when evaluated, then current server policy rejects it even if the client displayed the action.
3. Given an irreversible destructive action, when requested, then authorization, explicit confirmation, retention/dependency effects, and audit/operational consequences defined by policy are enforced before commit.
4. Given persistence fails, when the workflow aborts, then no success response, external notification, or realtime event is emitted.

### States and accessibility

Sensitive confirmations and failures shall clearly state consequences, require keyboard-operable explicit action, manage focus, and avoid disclosing secret-link validity or other Communities' state.

### Verification

- Domain/unit: Test transition, replay/staleness, and destructive policy.
- Integration/contract: Exercise invalid/expired/replayed callbacks and atomic failure behavior.
- Component: Test role/moderation/destructive confirmation and non-disclosing errors.
- Journey/live: Run invitation, permission-change, moderation, and destructive flows including stale clients.
- Manual/rendered: Review controls, audit implications, and failure disclosure.

## L2-SECU-010: Browser and transport headers match the deployment threat model

**Traces to:** L1-SECU-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Production responses shall set Content Security Policy, framing, content-type, referrer, permissions, and transport-security headers appropriate to the deployed public, application, API, and asset surfaces. Cookie-based authentication shall include explicit secure, domain, SameSite, and CSRF analysis and controls. CORS shall be least-privilege and shall not be introduced between marketing and API when marketing has no API requirement.

### Acceptance criteria

1. Given each production surface, when response headers are inspected, then CSP, anti-framing, content-type protection, referrer, permissions, and transport controls match its documented policy.
2. Given an unauthorized framing or script/style/connect origin, when attempted, then browser policy blocks it without breaking approved canonical assets or required connections.
3. Given cookie authentication, when a state-changing request is made, then secure cookie attributes and documented CSRF defenses are present and tested.
4. Given the public marketing origin, when its behavior is inspected, then it does not gain broad API CORS access merely because the origins are separate.

### States and accessibility

Header enforcement shall not block approved accessibility assets, fonts, assistive labels, or authentication recovery; blocked integrations shall degrade to an understandable state.

### Verification

- Domain/unit: Validate header and origin policy generation.
- Integration/contract: Assert headers, CSP enforcement, CORS, cookies, and CSRF behavior over HTTPS-shaped tests.
- Component: Test blocked-embed/script fallback states.
- Journey/live: Run sign-in and core journeys under production browser policies.
- Manual/rendered: Inspect browser console, security headers, and accessible degradation.

## L2-SECU-011: Deployment uses protected environments and short-lived credentials

**Traces to:** L1-SECU-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Production deployment shall occur only from protected branches and environments using OIDC or equivalent short-lived, scoped credentials. Untrusted pull requests shall never receive production secrets or deployment authority. Deployment shall remain conditional when an environment is intentionally absent, and artifact promotion shall use the already-built verified artifact rather than rebuild under more privileged credentials.

### Acceptance criteria

1. Given an untrusted pull request, when CI executes, then no production secret, cloud credential, protected environment token, or privileged deployment step is available.
2. Given an authorized production deployment, when credentials are issued, then they are short-lived, scoped to the required operation/environment, and not stored as long-lived repository secrets when federation is available.
3. Given an environment is intentionally unconfigured, when the pipeline reaches deployment, then it skips or fails with a deliberate status without weakening earlier quality/security gates.
4. Given an artifact passed verification, when promoted, then the identical artifact digest is deployed rather than rebuilding from source under privileged context.

### States and accessibility

Deployment security failures are operational states; any user-visible maintenance or unavailable page shall remain sanitized, keyboard operable, and provide support/retry guidance.

### Verification

- Domain/unit: Validate workflow policy and artifact-digest checks where scripted.
- Integration/contract: Exercise pull-request, protected-branch, absent-environment, and promotion paths.
- Component: Test the public unavailable/maintenance state if provided.
- Journey/live: Smoke-test the promoted artifact and verify its recorded digest.
- Manual/rendered: Review permissions, trust policies, environment protections, and credential lifetimes.

## L2-SECU-012: Supply-chain changes are reviewed and continuously scanned

**Traces to:** L1-SECU-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

SDKs, package managers, dependencies, and lockfiles shall be pinned and committed as applicable. A dependency shall be accepted only for a clear capability after reviewing platform alternatives, maintenance health, license compatibility, security posture, and bundle/runtime cost. CI shall scan dependencies and source for known vulnerabilities and secrets; patches shall be deliberate, and risk or license exceptions shall be documented with owner and expiry/review trigger.

### Acceptance criteria

1. Given a clean restore, when dependency resolution runs, then pinned SDK/package inputs and committed lockfiles determine versions and `npm ci` succeeds without modifying them.
2. Given a new dependency, when reviewed, then capability need, platform alternative, maintenance, license, security, and cost evidence is present before merge.
3. Given CI, when dependency and secret scans detect a policy-blocking issue, then merge or release is blocked unless an authorized time-bounded exception exists.
4. Given a security or license exception, when inspected after its review trigger or expiry, then it is closed, renewed with evidence, or blocks release.

### States and accessibility

Dependency changes that affect UI assets, interaction, or authentication shall preserve accessibility and document any new third-party privacy behavior.

### Verification

- Domain/unit: Validate manifests, lockfiles, and exception schema.
- Integration/contract: Run reproducible restore plus dependency/secret scans.
- Component: Measure affected bundle/runtime behavior and accessibility.
- Journey/live: Smoke-test capabilities introduced or patched by dependencies.
- Manual/rendered: Review dependency evidence, licenses, findings, and exceptions.

## L2-SECU-013: Vulnerabilities have a private reporting and remediation path

**Traces to:** L1-SECU-005

**Status:** proposed

**Source:** product brief + research

**Related:** —

The project shall publish a private vulnerability-reporting path and define ownership for triage, severity, communication, remediation, verification, disclosure, and exception handling. Security findings from reports, scans, tests, incidents, or provider advisories shall be tracked without exposing exploit details prematurely, patched deliberately, and linked to affected threat models, requirements, runbooks, and release notes when appropriate.

### Acceptance criteria

1. Given a researcher or member needs to report a vulnerability, when they consult security documentation, then a private monitored channel and expected response process are clear.
2. Given a new security finding, when triaged, then severity, affected versions/data, owner, containment, remediation target, communication, and verification are recorded with access appropriate to sensitivity.
3. Given a remediation ships, when it is closed, then regression evidence exists and affected threat models, exceptions, runbooks, and public communication are updated as appropriate.
4. Given remediation is deferred, when risk is accepted, then an accountable owner, compensating controls, review date, and observable trigger are documented.

### States and accessibility

Public reporting guidance and any affected-user communication shall be readable, keyboard accessible, and available without sign-in where appropriate.

### Verification

- Domain/unit: Test regression controls associated with resolved findings.
- Integration/contract: Exercise patched boundary behavior without exposing exploit fixtures publicly.
- Component: Test security-notice/support presentation where applicable.
- Journey/live: Validate the private reporting channel and incident intake process.
- Manual/rendered: Review ownership, response targets, exceptions, and communication accessibility.

## L2-SECU-014: Privacy, consent, retention, and analytics purpose are declared

**Status:** proposed

**Related:** —

Collection and use of identity, profile, content, moderation, activity, telemetry, and analytics data shall have a declared product/operational purpose, minimization rule, owner, retention/deletion behavior, access boundary, and applicable consent or notice. Product analytics shall answer declared questions and operate under the privacy/consent policy. 3. Given required consent is absent or withdrawn, when optional analytics/embeds execute, then collection stops while core public and member behavior remains available as policy requires.
4. Given retention or deletion becomes due, when the operational process runs, then affected data is handled according to the declared lifecycle and the result is auditable without retaining prohibited content.

### States and accessibility

Privacy notices, preferences, consent controls, export/deletion requests where provided, and related status messages shall be understandable and keyboard operable without dark patterns.

### Verification

- Domain/unit: Validate analytics schemas and retention-policy calculations.

- Manual/rendered: Review data inventory, privacy language, and control accessibility.

3. Given a production release decision, when security readiness is reviewed, then threat models, scan results, critical integration tests, recovery implications, exceptions, and owners are current.
4. Given an unsupported security or readiness claim is found, when release approval occurs, then the claim is corrected or the missing review is completed before promotion.

### States and accessibility

Security limitations, incidents, and support guidance shall be communicated in plain, accessible language without exposing exploitable detail.

### Verification

- Domain/unit: Validate required readiness fields and evidence links.

- Component: Test any user-facing security/readiness notice.

## L2-SECU-016: Encrypt protected data and govern keys

**Traces to:** L1-SECU-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Production traffic shall use authenticated current TLS according to the accepted threat model.
Relational data, object storage, queues, Search where enabled, backups, and exported sensitive
artifacts shall use provider-supported encryption at rest with named key ownership, least-privilege
use, rotation, revocation, recovery, and audit policy. Product code shall not invent cryptography when
a reviewed platform primitive meets the requirement.

### Acceptance criteria

1. Given a production public, application, API, realtime, provider, or operator connection, when transport is established, then approved TLS and certificate validation protect it and insecure downgrade is rejected or redirected only under the documented browser policy.
2. Given protected data is persisted in a configured relational, object, queue, Search, backup, or export store, when its provider settings are inspected, then encryption at rest and the authorized key boundary match the data classification.
3. Given a workload or operator uses a key, when authorization is evaluated, then only the minimum environment and operation are allowed and key use produces the required privacy-safe audit evidence.
4. Given a key rotates, expires, is suspected compromised, or is revoked, when the runbook executes, then new writes use current material, required existing data remains recoverable or is deliberately re-encrypted, unauthorized use stops, and affected services are verified.
5. Given encryption or certificate configuration is missing, uses a prohibited value, or cannot be verified, when production starts or deployment is promoted, then readiness fails without printing key material.

### States and accessibility

Operator views expose active, rotating, expiring, revoked, unavailable, and unverifiable states in
text, use exact safe timestamps, and never display key values.

### Verification

- Domain/unit: Test classification-to-encryption, key-scope, rotation, and readiness policies.
- Integration/contract: Inspect TLS, store encryption, workload authorization, audit, and rotation behavior.
- Component: Test masked key/certificate status and safe failure states where supplied.
- Journey/live: Rotate a non-production key and certificate while exercising critical reads, writes, Jobs, and recovery.
- Manual/rendered: Independent security review of algorithms, providers, ownership, rotation, and recovery.

## L2-SECU-017: Render untrusted text, links, and exports safely

**Traces to:** L1-SECU-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every Account-authored, imported, provider-supplied, support, administrator, and moderation field is
plain text by default or passes one centralized allow-list sanitizer for its explicitly declared
markup subset. Output uses contextual encoding for HTML text/attributes, URLs, JSON, email, calendar,
CSV/spreadsheet, logs, and other sinks. This applies to Messages, Events and locations, Reports,
internal notes, reasons, Appeal grounds, Profiles, names, labels, and any future rich-text field, not
only Posts and Comments.

### Acceptance criteria

1. Given untrusted text contains markup, script, event handlers, style, template syntax, control characters, or encoded variants, when stored and rendered in any supported UI/email/export context, then only the field's declared safe semantics appear and no executable or structure-breaking content runs.
2. Given an untrusted URL uses an unsupported scheme, credentials, misleading control characters, unsafe redirect, or invalid host form, when submitted or rendered, then it is rejected or presented as inert text and cannot become an executable link or privileged request.
3. Given an export cell begins after whitespace/control normalization with `=`, `+`, `-`, `@`, tab, carriage return, or another configured formula trigger, when CSV or spreadsheet-compatible output is generated, then the value is neutralized according to the documented format and cannot execute as a formula when opened in supported tools.
4. Given rich text is allowed for one field, when another field receives the same input, then that permission does not transfer; each sink re-encodes canonical safe content and legacy stored data is sanitized on output.
5. Given a sanitizer or encoder policy changes, when regression fixtures run, then Message, Event, Report, case note, Action reason, Appeal, email, calendar, and export contexts are all verified before promotion.

### States and accessibility

Validation identifies unsupported formatting or links without echoing executable markup, preserves
safe text, and keeps errors, previews, notes, and exports understandable to assistive technology.

### Verification

- Domain/unit: Test normalization, sanitizer allow-lists, URL policy, contextual encoders, and formula neutralization.
- Integration/contract: Submit attack corpora through every named field, API, import, provider, and export sink.
- Component: Test safe previews, validation, legacy content, link names, and assistive semantics.
- Journey/live: Author, moderate, appeal, notify, export, and reopen representative hostile content.
- Manual/rendered: Independent application-security and accessibility review across browser, email, calendar, and spreadsheet tools.
