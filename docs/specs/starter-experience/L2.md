# STRT — Starter adoption and developer experience

## L2-STRT-001: Pin and validate development toolchains

**Traces to:** L1-STRT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The repository selects supported stable or LTS .NET, Node.js, package-manager, browser-test, and
infrastructure-tool versions at adoption time. SDK selectors, lockfiles, central package versions,
and reproducible restore settings are committed and used identically by local and CI workflows.

### Acceptance criteria

1. Given a supported clean workstation, when the documented bootstrap command runs, then it selects or validates every required tool version before restoring dependencies.
2. Given a required tool is missing or incompatible, when validation runs, then it stops with the expected version and a safe remediation rather than producing a partial environment.
3. Given lockfiles and caches are empty, when restore runs twice against unchanged sources, then the resolved dependency graph is identical.
4. Given a major tool or dependency upgrade is proposed, when reviewed, then compatibility, migration, license, security, runtime or bundle cost, and rollback evidence accompany the change.

### States and accessibility

Bootstrap output uses plain text status and actionable errors, does not depend on color or animation,
and remains usable in narrow terminals and assistive command-line environments.

### Verification

- Domain/unit: Validate version and manifest comparison helpers.
- Integration/contract: Bootstrap on every supported operating-system profile from empty caches.
- Component: Not applicable; verify rendered setup documentation.
- Journey/live: Clone, restore, build, test, and publish on a clean CI runner.
- Manual/rendered: Review prerequisites, supported versions, and upgrade instructions.

## L2-STRT-002: Expose canonical repository commands

**Traces to:** L1-STRT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

The root README and repeatable `eng/` entry points expose setup, start, stop, format, lint, analyze,
build, unit, integration, journey, accessibility, migration, seed, publish, and smoke commands.
Wrappers preserve underlying exit codes and invoke the same behavior in CI.

### Acceptance criteria

1. Given a clean supported checkout, when each documented canonical command runs, then it performs the named scope non-interactively or clearly declares required interaction and returns an accurate exit code.
2. Given a command fails in a child process, when its wrapper exits, then it returns failure and retains enough safe diagnostic context to locate the failing layer.
3. Given CI invokes a quality gate, when its command is compared with local documentation, then developers can run the same gate without reverse-engineering workflow YAML.
4. Given documentation references a removed or renamed command, when docs validation runs, then the change fails before merge.

### States and accessibility

Commands expose progress, success, failure, cancellation, and help as text, avoid noisy decoration, and
do not require a graphical shell.

### Verification

- Domain/unit: Test wrapper argument validation and exit-code propagation.
- Integration/contract: Run each command with successful and representative failing inputs.
- Component: Not applicable; validate generated or rendered command references.
- Journey/live: Execute the full clean-checkout command sequence used by CI.
- Manual/rendered: Review discoverability, names, help, and troubleshooting links.

## L2-STRT-003: Provision repeatable local dependencies

**Traces to:** L1-STRT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

A versioned local environment supplies the relational database, mail catcher, object-storage
substitute, and any enabled Search, cache, queue, or telemetry dependencies. It has deterministic
ports or discoverable configuration, health waits, persisted-volume policy, and clean teardown
without requiring shared developer services.

### Acceptance criteria

1. Given required local tooling is present, when the environment starts, then every enabled dependency reaches a checked ready state and the application receives its endpoints through configuration.
2. Given a dependency cannot start or a port is unavailable, when provisioning runs, then it fails with the exact service and safe remediation rather than starting a partially usable app.
3. Given the environment stops normally, when teardown completes, then declared persistent state remains and ephemeral credentials and processes do not.

### States and accessibility

Provisioning and health output is textual, ordered, and screen-reader friendly; teardown names its
exact local scope and preserves declared persistent state.

### Verification

- Domain/unit: Test environment-name, port, and destructive-scope validation.
- Integration/contract: Start, stop, restart, and fail the complete local dependency set.
- Component: Not applicable; validate any local dashboard independently.
- Journey/live: Run the first vertical slice using only provisioned local services.
- Manual/rendered: Review cross-platform guidance and safety of teardown.

## L2-STRT-004: Validate environment configuration and secrets

**Traces to:** L1-STRT-001

**Status:** proposed

**Source:** product brief + research

**Related:** —

Configuration is grouped in typed options, documented by a safe example containing names and
non-secret placeholders only, and validated at startup. Production rejects development values,
default credentials, missing cryptographic material, localhost provider endpoints, and incompatible
capability settings before readiness.

### Acceptance criteria

1. Given a developer copies the example configuration, when local services start, then required values are explained and no real credential is present in source.
2. Given a production profile contains a missing, placeholder, development-only, or mutually inconsistent value, when startup validates options, then readiness fails with a redacted actionable error.
3. Given a secret appears in a tracked file or generated artifact, when scanning runs, then the quality gate fails and remediation guidance requires rotation when exposure is possible.
4. Given an unknown configuration key or stale documented key exists, when validation runs, then drift is reported rather than silently ignored where the configuration system permits.

### States and accessibility

Diagnostics identify option paths but mask values, use concise text and stable codes, and are readable
in logs and terminals without color.

### Verification

- Domain/unit: Test every production option validator and placeholder pattern.
- Integration/contract: Start all environment profiles with valid and representative invalid matrices.
- Component: Not applicable; validate any configuration UI under its owning capability.
- Journey/live: Deploy a production-like artifact with externally supplied secrets and rotate one.
- Manual/rendered: Security and operations review option documentation and diagnostics.

## L2-STRT-005: Seed deterministic fictional scenarios

**Traces to:** L1-STRT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Development and dedicated demo profiles can create clearly fictional Accounts, Communities,
Membership states, content, Events, Messages, Reports, Notifications, and failure
edges needed by specifications and mocks. Seeds use stable identifiers and clocks where tests or
screenshots require them and never derive from real people or production exports.

### Acceptance criteria

1. Given an empty development database, when a named seed scenario runs, then its documented identities, lifecycle states, relationships, and expected credentials or login path are reproducible.
2. Given the same seed runs again, when records already exist, then it is idempotent or fails with clear cleanup and reseed guidance and never duplicates Membership or participation state.
3. Given production or an unapproved environment is configured, when seed execution is requested, then it is rejected before any record is written.
4. Given source and fixtures are scanned, when fictional data is reviewed, then no real credential, contact, token, or personal production data is present.

### States and accessibility

Seed documentation lists available scenarios and expected states in accessible tables; output names
created, unchanged, and failed aggregates without dumping private-like bodies.

### Verification

- Domain/unit: Test deterministic factories and environment gate.
- Integration/contract: Seed empty and existing real-provider databases and validate constraints.
- Component: Use seeds to render representative empty, success, conflict, and moderation states.
- Journey/live: Run critical journeys against a fresh named scenario.
- Manual/rendered: Review realism, inclusivity, fictionality, and coverage.

## L2-STRT-006: Isolate development-only identity and providers

**Traces to:** L1-STRT-002

**Status:** proposed

**Source:** product brief + research

**Related:** —

Any local sign-in shortcut, fixed code, mail catcher, storage substitute, provider-feedback fixture,
or disabled safety check is explicitly labeled, registered only in an allowed development
profile, and structurally rejected by production validation.

### Acceptance criteria

1. Given an allowed development profile, when a shortcut is used, then its UI and telemetry identify it as non-production and its scope is limited to fictional local data.
2. Given a production or shared unapproved profile, when a development implementation or bypass is registered, then startup fails before serving traffic.
3. Given a development endpoint is guessed in production, when routed, then it is absent rather than merely hidden in navigation.
4. Given a development provider creates feedback or a Delivery, when it executes, then it cannot reach an unapproved public destination or production credential.

### States and accessibility

Development warnings are persistent but not disruptive, include visible text and accessible names, and
do not rely only on environment-colored banners.

### Verification

- Domain/unit: Test environment and provider-registration rules.
- Integration/contract: Enumerate production routes/services and prove shortcuts are absent.
- Component: Test clear development labeling.
- Journey/live: Publish the production profile and attempt every documented development path.
- Manual/rendered: Security review isolation, wording, and provider network boundaries.

## L2-STRT-008: Apply and verify product identity

**Traces to:** L1-STRT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

A documented specialization mechanism updates the product display name, safe code identifier,
namespaces, solution and project names, app selector prefix, package metadata, domains, cookie names,
email sender, public links, support/security contacts, deployment labels, and documentation without
blind text replacement inside generated or historical evidence.

### Acceptance criteria

1. Given valid display and code names, when specialization runs on a clean starter, then every declared active identity location changes and the resulting repository restores, builds, and tests.
2. Given an invalid identifier, reserved name, conflicting path, or unsafe domain is supplied, when validation runs, then no partial rename is applied.
3. Given specialization completes, when placeholder scanning runs, then active source, configuration, routes, metadata, and deployable output contain no prohibited starter identity.
4. Given ADR history or licensed attribution contains a historical name, when verification scans it, then explicit allow-listing preserves the record rather than corrupting history.

### States and accessibility

Interactive prompts, dry-run output, conflict reports, and completion summaries are textual,
keyboard-operable, and provide exact changed categories without exposing secrets.

### Verification

- Domain/unit: Test name normalization, identifier rules, planned-change set, and allow-list.
- Integration/contract: Specialize in paths with spaces and build every project/artifact.
- Component: Render title, navigation, emails, metadata, and shared components with the new identity.
- Journey/live: Run the primary journey and public CTAs after specialization.
- Manual/rendered: Review remaining-name report and visual/product consistency.

## L2-STRT-009: Specialize the shared visual language

**Traces to:** L1-STRT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Adopters customize canonical logo assets, fonts, semantic brand tokens, icons, social imagery, and
email identity from one documented cross-surface source. Specialization preserves component states,
contrast, focus, reduced motion, responsive behavior, licenses, and build-time asset validation.

### Acceptance criteria

1. Given valid branded assets and semantic token values, when the design-system build runs, then marketing, Angular, email previews, and shared components consume the same intended source.
2. Given a referenced asset is missing or an unsupported direct copy drifts, when validation runs, then the build fails with the source and consumer identified.
3. Given custom colors or typography violate an accepted accessibility or layout constraint, when automated and rendered review runs, then readiness fails or the exception is explicitly reviewed.
4. Given starter assets have licensing or placeholder restrictions, when a production artifact is built, then prohibited assets cannot be included.

### States and accessibility

Representative product components render default, hover, focus, active, disabled, loading, invalid,
destructive, empty, long-content, narrow, high-contrast, and reduced-motion states.

### Verification

- Domain/unit: Validate token schema, asset manifest, and prohibited-placeholder rules.
- Integration/contract: Verify deterministic asset copy/build across every surface.
- Component: Run component, visual, and automated accessibility checks.
- Journey/live: Inspect branded marketing, auth, core journey, email, and admin surfaces.
- Manual/rendered: Accessibility and licensing review at representative viewports/themes.

## L2-STRT-010: Configure optional capability profiles

**Traces to:** L1-STRT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Optional Messaging, Events, public indexing, realtime, analytics, and email Delivery profiles have
explicit defaults, dependency validation, route and navigation behavior, processing and
credential behavior, data-migration consequences, and tests. Disabling a capability removes its
claims and attack surface while preserving shared requirements.

### Acceptance criteria

1. Given an optional capability is disabled, when the artifact starts, then its user routes, navigation, Jobs, provider clients, credentials, provider-feedback paths, marketing claims, and public contract behave as documented and do not appear half-configured.
2. Given a capability is enabled without required provider, policy, data, or security decisions, when production validation runs, then readiness fails before serving traffic.
3. Given a capability changes profile after data exists, when transition is requested, then migration, retention, user effect, and rollback or forward-fix rules are explicit and confirmed.
4. Given any profile is selected, when shared gates run, then Community isolation, server authority, privacy, accessibility, audit, and operational requirements remain mandatory.

### States and accessibility

Unavailable capabilities are absent or clearly unavailable rather than disabled without explanation;
configuration UI exposes inherited, invalid, changing, and active states accessibly.

### Verification

- Domain/unit: Test profile dependency and invariant matrices.
- Integration/contract: Build and start every supported profile and inspect routes, services, and Jobs.
- Component: Test navigation and unavailable-state behavior.
- Journey/live: Run applicable primary journeys for each supported profile.
- Manual/rendered: Review product claims, data consequences, and configuration comprehension.

## L2-STRT-011: Record provider and architecture decisions

**Traces to:** L1-STRT-003

**Status:** proposed

**Source:** product brief + research

**Related:** —

Adoption records consequential choices for identity, relational database, media, email, Search,
realtime, Jobs, analytics, hosting, domains, secrets, telemetry, recovery, and licensing in
immutable architecture decision records with drivers, alternatives, trade-offs, verification, and
revisit triggers.

### Acceptance criteria

1. Given a provider or architecture choice changes dependency direction, data ownership, security, public contract, deployment, license, or operational cost, when implementation begins, then a proposed or accepted decision record exists and links affected requirements.
2. Given a decision changes, when the replacement is accepted, then the prior record remains, is marked superseded, and cross-links the replacement.
3. Given a decision asserts a capability, when readiness is evaluated, then its verification evidence proves the selected configuration rather than only documenting intent.
4. Given a trigger for reconsideration occurs, when documentation review runs, then the decision becomes an owned review item.

### States and accessibility

Decision documents use semantic headings, text diagrams or accessible diagram descriptions, stable
links, and explicit proposed, accepted, superseded, or deprecated status.

### Verification

- Domain/unit: Validate ADR metadata, numbering, links, and statuses.
- Integration/contract: Check configured providers against accepted decisions.
- Component: Not applicable; validate rendered documentation.
- Journey/live: Trace one provider-backed journey from requirement through decision to evidence.
- Manual/rendered: Architecture, security, and operations review of open triggers and trade-offs.

## L2-STRT-012: Maintain an outcome-oriented documentation map

**Traces to:** L1-STRT-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

The root README states the product outcome and differentiator, honest readiness, prerequisites, setup,
canonical commands, repository and architecture map, configuration entry points, deployed surfaces,
and documentation map. Detailed documents cover architecture flows, development, testing, data,
security, deployment, recovery, support, and troubleshooting without duplicating unstable facts.

### Acceptance criteria

1. Given a new contributor has only a clean checkout, when following the README, then they can identify product scope, run the supported environment, execute gates, and locate deeper documentation.
2. Given a command, route, configuration key, project, or deployment flow changes, when the change is reviewed, then every authoritative documentation location is updated in the same change.
3. Given documentation links, anchors, commands, or diagrams are invalid, when validation runs, then the quality gate fails.
4. Given a capability is not independently reviewed or implemented, when readiness is described, then documentation names the limitation rather than implying production safety.

### States and accessibility

Documentation uses semantic Markdown, language-tagged code, meaningful links, accessible diagrams or
descriptions, and text that renders at narrow widths and high zoom.

### Verification

- Domain/unit: Validate required README sections and metadata where automated.
- Integration/contract: Run documented commands and link/anchor checks.
- Component: Render docs and verify code, table, and diagram alternatives.
- Journey/live: Onboard a reviewer from clean checkout without undocumented steps.
- Manual/rendered: Editorial, accessibility, and readiness-language review.

## L2-STRT-014: Document contribution, ownership, support, and disclosure

**Traces to:** L1-STRT-004

**Source:** product brief + research

**Related:** —

The repository defines contribution setup and gates, code and operational ownership, review

4. Given a contact or ownership mapping expires or becomes invalid, when scheduled validation runs, then a maintained owner is alerted.

keyboard operation, and accessible error handling.

### Verification

- Component: Test support and disclosure form states where supplied.
- Journey/live: Submit representative support and private disclosure exercises.

## L2-STRT-015: Track dependency and asset provenance

**Traces to:** L1-STRT-004

**Status:** proposed

**Source:** product brief + research

**Related:** —

Every runtime, build, test, font, icon, image, fixture, and distributed document dependency has a
known source, version or revision, license, purpose, distribution status, maintenance and security
posture, and owner. Additions require evidence that platform code or an existing dependency is not a
better fit and record bundle/runtime or operational cost.

### Acceptance criteria

1. Given a new dependency or asset is introduced, when review runs, then purpose, alternatives, source, license, maintenance, security, size or runtime cost, and update owner are recorded.
2. Given a license conflicts with intended use or provenance is unknown, when the artifact builds for distribution, then the gate fails or an approved exclusion prevents distribution.
3. Given a dependency becomes vulnerable, unmaintained, relicensed, or unused, when monitoring detects it, then an owned replace, remove, patch, or documented exception action is created.
4. Given an artifact is published, when its manifest is inspected, then declared dependencies and notices match included contents and no local/generated/test-only material leaks into it.

### States and accessibility

Inventory and notices use accessible tables, durable source labels, and explicit unknown, reviewed,
exception, expiring, and prohibited states.

### Verification

- Domain/unit: Validate provenance manifest and policy rules.
- Integration/contract: Compare dependency graphs, asset manifests, licenses, and published contents.
- Component: Not applicable; render inventory and notices for accessibility.
- Journey/live: Add and then reject a representative incompatible asset in a disposable branch/test.
- Manual/rendered: Legal/licensing, security, and maintenance review before distribution.
